## Contexto del día

El 20 de mayo quedó `DiferencialToSingleEnded` funcionando — la cadena analógica en el PSoC convierte la señal diferencial del geófono y la transmite via UART al PC. Pero un sistema MASW real necesita múltiples geófonos desplegados en línea (24, 48, o hasta 96 canales) a distancias de metros entre sí. Un cable UART de 50m no es práctico — y peor: necesitaría un cable por geófono.

El 24 de mayo es el día en que se diseña la solución inalámbrica: cada geófono tiene un ESP32 que lee el PSoC via SPI y transmite via ESP-NOW al maestro. El maestro agrega los datos de todos los nodos y los envía via TCP a MATLAB. Es un sistema distribuido real.

Son tres commits en el mismo día, en orden temporal de elaboración: código asistido por Claude → revisión manual → iteración con Codex. El mensaje "sin revisar" en el primero es honesto — es código generado que se commitea para tener un punto de partida, sabiendo que hay que revisarlo.

#### `sync_protocol.h` — el protocolo de mensajes ESP-NOW

El protocolo de sincronización está definido con `#pragma pack(push, 1)` para garantizar que no haya padding entre campos en la estructura serializada:

```cpp
#define CMD_ARM       0x10
#define CMD_ARM_ACK   0x11
#define CMD_START     0x12
#define CMD_STOP      0x13
#define CMD_DEBUG     0x14
#define CMD_DATA      0x20
#define CMD_STATUS    0x30

struct MsgArm    { uint8_t cmd; };                              // 1 byte
struct MsgArmAck { uint8_t cmd; uint8_t node_id; uint8_t status; }; // 3 bytes
struct MsgStart  { uint8_t cmd; uint64_t t_start_us; };         // 9 bytes
struct MsgStop   { uint8_t cmd; };                              // 1 byte
struct MsgDebug  { uint8_t cmd; uint8_t enable; };              // 2 bytes
```

El mensaje `MsgStart` lleva el timestamp en microsegundos (`uint64_t`) — esto permite que todos los esclavos estampen sus muestras con una referencia temporal común. La sincronización de relojes entre múltiples ESP32 en ESP-NOW es un problema no trivial: los relojes de cada chip derivan diferente. El `t_start_us` resuelve esto dando a todos el mismo t0.

```cpp
struct SampleBytes {
    uint8_t raw_lo, raw_hi;          // ADC raw (16-bit)
    uint8_t alog0, alog1, alog2;     // post-analog (24-bit)
    uint8_t digi0, digi1, digi2;     // post-digital (24-bit)
    uint8_t gain;                    // PGA gain code
    uint8_t flags;                   // estado del sistema
};                                   // 10 bytes por muestra
```

Cada muestra lleva **tres valores simultáneos** del mismo instante:
- `raw_input`: el ADC sin procesar (antes de filtros)
- `post_analog`: después de la cadena MFB + compensador Pazos
- `post_digital`: después del notch FIR 50Hz del DFB

Este diseño permite comparar las tres señales en MATLAB para evaluar el efecto de cada etapa de procesamiento. Para la tesis: la diferencia entre `raw_input` y `post_analog` es el efecto del compensador analógico; la diferencia entre `post_analog` y `post_digital` es el efecto del notch digital.

El paquete de datos `MsgData` agrupa 15 muestras por paquete ESP-NOW:

```cpp
#define SAMPLES_PER_PART 15
struct MsgData {
    uint8_t     marker;          // 0xBC
    uint8_t     node_id;         // ID del esclavo
    uint16_t    seq;             // número de secuencia
    uint8_t     part;            // parte 0 o 1 (de 30 muestras totales)
    uint8_t     n_samples;       // 15 en este paquete
    uint64_t    timestamp_us;    // timestamp de la primera muestra
    SampleBytes samples[SAMPLES_PER_PART];  // 15 × 10 = 150 bytes
    uint8_t     crc;             // CRC del paquete
};  // total: ~165 bytes
```

El ESP-NOW tiene un límite de 250 bytes por frame. Con 15 muestras × 10 bytes = 150 bytes + header = ~165 bytes total, entra cómodamente en ese límite.

#### `psoc_spi.h` — interfaz SPI ESP32 → PSoC

El ESP32 es master SPI; el PSoC es slave. La conexión VSPI del ESP32:

```cpp
/* Wiring (ESP32 VSPI):
 *   GPIO 18 → SCLK
 *   GPIO 19 → MISO  (PSoC drives)
 *   GPIO 23 → MOSI  (PSoC ignores — future: recibir coeficientes)
 *   GPIO  5 → SS    (activo bajo)
 *   GPIO  4 ← DATA_READY del PSoC (activo alto)
 *
 * Frame por transacción (306 bytes):
 *   Bytes 0-5  : header [0xAB][n_lo][n_hi][seq_lo][seq_hi][flags]
 *   Bytes 6..  : n_samples × 10 bytes
 *
 * SPI: Mode 0, 4 MHz, MSB first.
 */
```

El pin `DATA_READY` (GPIO 4) es una señal del PSoC que indica "tengo un batch de 30 muestras listo para transmitir". El ESP32 espera esa señal antes de iniciar la transacción SPI — sin esto, leerías datos parciales o ceros.

La estructura `PsocSample` expone los tres valores por muestra que el PSoC serializa en SPI:

```cpp
struct PsocSample {
    int16_t  raw_input;    // ADC sin procesar
    int32_t  post_analog;  // después del compensador analógico
    int32_t  post_digital; // después del notch digital
    uint8_t  gain_byte;    // PGA gain code
    uint8_t  sample_flags; // flags de estado
};
```

#### `master/src/main.cpp` — el cerebro de la red

El maestro implementa una máquina de estados que coordina el experimento MASW:

```cpp
enum MasterState { IDLE, ARMING, ARMED, RUNNING, STOPPING };
```

El flujo completo desde MATLAB:

```
MATLAB → cmd 0xA2 (ARM)
    │
    ▼
Maestro → broadcast CMD_ARM (0x10) a todos los esclavos
Esclavos → responden CMD_ARM_ACK (0x11) con node_id
Maestro espera NUM_SLAVES ARM_ACKs (por defecto 2)
    │
    ▼
MATLAB → cmd 0xA3 (START)
    │
    ▼
Maestro:
  1. GPIO 25 HIGH (hardware sync — llega en nanosegundos)
  2. broadcast CMD_START + t_start_us (software sync — llega en ~100µs via ESP-NOW)
    │
    ▼ (simultáneo en todos los nodos)
Slaves: ISR en GPIO 26 (IRAM) → transición ARMED→SAMPLING
PSoCs: ven flanco en SYNC_IN → inician muestreo simultáneo
    │
    ▼
MATLAB → cmd 0xA4 (STOP)
    │
    ▼
Maestro:
  1. GPIO 25 LOW
  2. broadcast CMD_STOP
```

La sincronización tiene dos caminos paralelos: hardware (GPIO) y software (ESP-NOW). El GPIO es el de mayor precisión — el flanco llega en nanosegundos, independiente del throughput de WiFi. El ESP-NOW es backup y lleva el timestamp global para el postprocesado.

La ISR del slave está en IRAM para máxima velocidad:

```cpp
void IRAM_ATTR onSyncEdge()
{
    int level = digitalRead(SYNC_IN_PIN);
    digitalWrite(SYNC_TO_PSOC_PIN, level);  // reenvío inmediato al PSoC
    if (level == HIGH && (g_state == ARMED || g_state == WAIT_ARM)) {
        g_t_start_us = (uint64_t)micros();
        g_state = SAMPLING;
    } else if (level == LOW && g_state == SAMPLING) {
        g_state = STOPPED;
    }
}
```

La función `IRAM_ATTR` pone el código en RAM interna del ESP32 (no en Flash), eliminando los ciclos de latencia por cache miss. Para una ISR de sincronización esto es crítico.

El campo `driftHist` es particularmente interesante — rastrea el drift temporal entre nodos para compensar la diferencia de frecuencias de cristal entre ESP32s. El maestro puede enviar comandos de configuración (PGA gain, VDAC, tx_mode) directamente a un esclavo específico por `node_id`. Esto permite ajustar la ganancia de cada canal individualmente desde MATLAB sin desconectar nada.

**Modo debug mejorado:**
```cpp
static void setDebugMode(bool enable)
{
    g_debug_mode = enable;
    g_debug_count = 0;
    if (enable) {
        g_t_start_us = (uint64_t)micros();
        g_state = SAMPLING;   // debug arranca inmediatamente (sin ARM/START)
```

En modo debug, el slave envía una rampa o datos sintéticos sin necesitar el handshake ARM/START — útil para verificar el canal de comunicación ESP-NOW → Maestro → TCP → MATLAB sin hardware PSoC conectado.
