## Contexto del día

El 17 de junio había cerrado con una cadena DMA de tres canales funcionando: ADC → RAM (crudo), ADC → Filter (FIR hardware) → RAM (filtrado), con un flag de software `g_stream_mode` decidiendo cuál de los dos resultados se usaba para la captura. Es una arquitectura simple y suficiente para elegir entre señal cruda o filtrada, pero tiene una limitación: no deja espacio para promediar múltiples muestras por hardware antes de que la CPU las vea, algo que sería deseable para reducir ruido de cuantización sin gastar ciclos de CPU en sumar y dividir manualmente.

El 18 de junio, Elías generaliza esa arquitectura. En vez de un canal DMA fijo por camino (raw, filtrado), cada camino se desdobla en dos variantes: **Single** (una muestra por disparo, el comportamiento de siempre) y **Lote** (un lote de `DMA_LOTE_SAMPLES` muestras consecutivas por disparo, pensado para alimentar un promediado futuro). Son cuatro canales DMA en total — `DMA_DelSig_RAM_Single`, `DMA_DelSig_RAM_Lote`, `DMA_Filter_RAM_Single`, `DMA_Filter_RAM_Lote` — más el quinto canal fijo que empuja la señal cruda hacia el bloque `Filter` (que ahora se llama `DMA_DelSig_Filter` y corre siempre, sin ISR asociada, alimentando el filtro FIR en segundo plano sin que importe qué ruta esté "viva").

Cuál de los cuatro canales recibe realmente el trigger del hardware no lo decide el software escribiendo o no escribiendo en cada uno — los cuatro DMA están siempre habilitados y reciclando sus Transaction Descriptors. Lo que decide cuál llega efectivamente a downstream es una lógica combinacional en el `TopDesign` gobernada por un registro de 8 bits nuevo, `Reg_Select`, que el firmware escribe con un patrón de bits. Es, en esencia, mover una decisión de enrutamiento desde el dominio del software (qué DMA inicializar y arrancar) hacia el dominio del hardware (qué compuertas dejan pasar la señal de trigger), manteniendo el software liviano: solo escribe 3 bits de configuración.

#### Diff y análisis técnico

Como en los commits anteriores del PSoC, la enorme cantidad de líneas es en su mayoría ruido de regeneración de código: agregar cuatro componentes DMA nuevos (`DMA_DelSig_RAM_Single`, `DMA_DelSig_RAM_Lote`, `DMA_Filter_RAM_Single`, `DMA_Filter_RAM_Lote`) y un componente de registro (`Reg_Select`) al `TopDesign` obliga a PSoC Creator a regenerar el `cyfitter.h`, los `.inc`, el `cymetadata.c` y todos los listados de ensamblador. El cambio de fondo está concentrado en `main.c` y en la librería nueva `Analog_LPF_v1_0.cylib`.

**Los buffers de "lote" (batch) reservados pero no usados todavía.** El primer cambio es la reserva de memoria para los buffers de la ruta Lote:

```c
/* Tamaño del lote HW (canales DMA_*_RAM_Lote) para promediado por DMA.
 * Hoy ningún consumidor lo usa todavía (la ruta queda en Single); el canal
 * ya está armado e inicializado para cuando se conecte el promediado. */
#ifndef DMA_LOTE_SAMPLES
#define DMA_LOTE_SAMPLES   30u
#endif
```

```c
/* Buffers del canal Lote (mux HW) — reservados para promediado por DMA;
 * la ruta hoy siempre selecciona Single, así que quedan sin tocar. */
static volatile uint8  g_dma_raw_lote_buf[DMA_LOTE_SAMPLES * 3u];
static volatile uint8  g_dma_filt_lote_buf[DMA_LOTE_SAMPLES * 3u];
```

Los comentarios son honestos sobre el estado de la funcionalidad: "hoy ningún consumidor lo usa todavía". Treinta muestras de 3 bytes cada una (24 bits por muestra, el ancho nativo del ADC Delta-Sigma) reservan 90 bytes por buffer, 180 bytes en total para ambas rutas (raw y filtrada) — una cantidad modesta de RAM que no compromete al PSoC 5LP, que tiene 64 KB de SRAM.

**El registro de selección de ruta.** La pieza central del día es `dma_route_select()`, una función que traduce tres booleanos de intención (usar filtro, lote crudo, lote filtrado) en un patrón de bits que se escribe a un componente de hardware:

```c
/* Selecciona, vía Reg_Select, qué resultado final usan los DMA *_RAM y si cada
 * camino corre en Single (1 muestra/disparo) o Lote (DMA_LOTE_SAMPLES/disparo).
 * La exclusión Single/Lote y raw/Filter la imponen las compuertas del TopDesign;
 * acá solo se escribe el bit pattern que las gobierna (control_0/1/2). */
static void dma_route_select(uint8 use_filter, uint8 raw_lote, uint8 filt_lote)
{
    uint8 reg = 0u;
    if (use_filter) { reg |= 0x02u; }   /* control_1 */
    if (filt_lote)  { reg |= 0x01u; }   /* control_0 */
    if (raw_lote)   { reg |= 0x04u; }   /* control_2 */
    Reg_Select_Write(reg);
}
```

`Reg_Select` es un componente de PSoC Creator del tipo "Control Register" — un registro mapeado en memoria cuya salida se conecta directamente a pines de lógica del `TopDesign` esquemático (en este caso, presumiblemente a las líneas de habilitación/selección de un multiplexor analógico o digital que decide qué DMA recibe el trigger real del hardware). El comentario deja explícito que la exclusión mutua entre Single y Lote, y entre raw y filtrado, no la impone el software — la imponen "las compuertas del TopDesign". Esto es coherente con el patrón de diseño que se viene consolidando en el proyecto: mover tanta lógica de tiempo real como sea posible del dominio del firmware al dominio del hardware programable del PSoC, dejando al software solo la tarea de parametrizar, no de ejecutar en tiempo real.

**Uso conservador durante la calibración.** El nuevo mecanismo se usa de inmediato para una garantía importante: durante la calibración analógica, el sistema necesita ver la señal cruda del ADC, no la filtrada ni un lote promediado. Dos puntos del código fuerzan explícitamente la ruta "raw + Single" al entrar y salir de calibración:

```c
static uint8 psoc_start_calibration_if_idle(uint8 send_ack)
{
    ...
    psoc_calibration_servo_abort();
    dma_route_select(0u, 0u, 0u);  /* fuerza raw+Single: calibración necesita ADC crudo */
    uart_send_diag(PSOC_EVT_CAL_START, 0u);
    ...
```

```c
static void service_runtime(void)
{
    ...
    uint8 ok = psoc_calibration_async_result_ok();
    g_state = PSOC_IDLE;
    timer_start_runtime();
    dma_route_select(g_stream_mode, 0u, 0u);  /* restaura ruta previa a la calibración */
    uart_send_diag(PSOC_EVT_CAL_DONE, ok);
    ...
```

El primer cambio garantiza que, sin importar en qué modo estuviera el sistema (FIR activado, por ejemplo, si un operador dejó esa opción prendida de una sesión anterior), al arrancar una calibración el hardware se reconfigura para entregar señal cruda. El segundo restaura el modo de streaming (`g_stream_mode`) que estaba activo antes de calibrar, una vez terminada la rutina. Es un detalle de robustez operativa: sin este par de líneas, sería posible calibrar el sistema mientras el DMA está enrutando la señal filtrada por el FIR, lo cual introduciría el retardo de grupo y la atenuación del filtro en las mediciones de calibración, arruinando la precisión de la búsqueda binaria de los DAC.

También se conecta `dma_route_select()` al comando existente `PSOC_CMD_SELECT_STREAM`, de forma que cuando el operador cambia entre crudo y filtrado desde la interfaz web, el registro de hardware se actualiza en el mismo lugar donde se actualiza la variable de software:

```c
case PSOC_CMD_SELECT_STREAM:
    /* rx_p1: 0=crudo, 1=filtrado FIR */
    g_stream_mode = (rx_p1 != 0u) ? 1u : 0u;
    dma_route_select(g_stream_mode, 0u, 0u);
    if (g_stream_mode != 0u) {
        g_fir_discard = 0u;  /* reset — se recarga en próximo SyncIn */
    }
    uart_send_cfg_ack(PSOC_CMD_SELECT_STREAM, g_stream_mode);
    led_toggle();
    break;
```

Nótese que en las tres llamadas a `dma_route_select()` de este commit, los parámetros `raw_lote` y `filt_lote` siempre son `0u` — la ruta Lote nunca se activa todavía. Es exactamente lo que anticipaban los comentarios de los buffers: la infraestructura de hardware está lista y cableada, pero el software todavía no expone ningún comando que active el modo de promediado por lotes. Es trabajo de fontanería (plumbing) puro, preparando el terreno para una funcionalidad que se implementará (o no) en un commit posterior.

**Renombrado de las señales ISR para reflejar la nueva topología.** Como ahora hay dos variantes (Single/Lote) de cada canal, los nombres de los componentes DMA y de las ISR se renombran para ser explícitos sobre cuál variante disparan:

```c
-CY_ISR(isr_DMA_ADC_Handler)
+CY_ISR(isr_DMA_DelSig_RAM_Handler)
```

```c
-CY_ISR(isr_DMA_Filter_Handler)
+CY_ISR(isr_DMA_Filter_RAM_Handler)
```

El nombre `isr_DMA_ADC_Handler` era ambiguo una vez que existen dos canales que llegan a RAM (Single y Lote); `isr_DMA_DelSig_RAM_Handler` dice explícitamente "esta es la interrupción del canal que copia del ADC Delta-Sigma a RAM en modo Single". Es un renombrado disciplinado, del tipo que evita confusión futura cuando el código crezca.

**La función `dma_adc_init()` reescrita para cinco canales.** La función que en el commit del 17 de junio configuraba tres canales DMA ahora configura cinco, con un comentario de cabecera que documenta la topología completa:

```c
/* Inicializa los cinco canales DMA del mux Single/Lote:
 *   DMA_DelSig_RAM_Single/Lote   ADC → RAM
 *   DMA_Filter_RAM_Single/Lote   Filter → RAM
 *   DMA_DelSig_Filter            ADC → Filter (siempre activo, sin ISR)
 * Todos los TD son circulares (nextTd == mismo TD): cada canal recicla solo.
 * Los cinco quedan habilitados aunque solo uno por par esté "vivo" — cuál
 * recibe el trigger real lo decide la lógica combinacional gobernada por
 * Reg_Select, no el software. */
static void dma_adc_init(void)
{
    ...
}
```

El patrón de inicialización de cada canal Lote es idéntico al de Single, salvo por el tamaño del Transaction Descriptor, que ahora transfiere `3u * DMA_LOTE_SAMPLES` bytes de una vez en lugar de 3:

```c
/* ── Raw Lote: ADC DelSig → g_dma_raw_lote_buf (lote de DMA_LOTE_SAMPLES) ── */
DMA_DelSig_RAM_Lote_DmaInitialize(3u, 1u,
    HI16((uint32)ADC_DEC_SAMP_PTR),
    HI16((uint32)(void *)g_dma_raw_lote_buf));
td = CyDmaTdAllocate();
CyDmaTdSetConfiguration(td, (uint16)(3u * DMA_LOTE_SAMPLES), td,
    DMA_DelSig_RAM_Lote__TD_TERMOUT_EN | TD_INC_DST_ADR);
```

Un detalle sutil: el puntero fuente del ADC cambia de `ADC_DEC_OUTSAMP_PTR` (usado el 17 de junio) a `ADC_DEC_SAMP_PTR`. Esto sugiere que, al reorganizar el `TopDesign` para acomodar los nuevos componentes, Elías también revisó (y probablemente corrigió) desde qué registro interno del decimador Delta-Sigma se lee la muestra — `SAMP` y `OUTSAMP` son registros distintos dentro del bloque decimador del ADC_DelSig de Cypress, y elegir el correcto es importante para que el dato leído por DMA sea consistente con el que veía la ISR original basada en `ADC_GetResult32()`. No hay explicación en el commit sobre por qué se hizo el cambio, pero es coherente con un trabajo de ajuste fino mientras se reconstruye la cadena.

**El componente custom `Analog_LPF_v1_0`.** Además del trabajo de DMA, el commit agrega una librería de componente PSoC Creator completamente nueva: `Analog_LPF_v1_0.cylib`, con archivos `lpf.c`/`lpf.h` en su carpeta `API`. Estos archivos son, sin embargo, una plantilla de Cypress sin instanciar — contienen placeholders de generación de código como `` `$INSTANCE_NAME` `` y `` `$CY_MAJOR_VERSION` `` en vez de nombres concretos, lo cual indica que es el esqueleto de un componente custom (probablemente basado en un tutorial o ejemplo oficial de Cypress para crear filtros pasabajos analógicos programables) que Elías incorporó al repositorio pero todavía no colocó en el `TopDesign` de `AcondicionamientoAnalogico`. Su API expone funciones simples — `Start()`, `Stop()`, `SetCutoffFreq()` — que escriben un registro de control (`CR0`) con una máscara de frecuencia de corte. Es, con toda probabilidad, una exploración de una vía alternativa (o complementaria) al filtro FIR digital: un filtro pasabajos analógico programable en frecuencia de corte, que podría usarse para anti-aliasing antes del ADC en vez de (o además de) filtrar digitalmente después. Al no estar instanciado en el esquemático, no tiene impacto funcional inmediato — es una semilla para trabajo futuro, guardada en el repositorio para no perderla.

#### Impacto en el proyecto

Este commit no cambia el comportamiento observable del sistema — con `raw_lote` y `filt_lote` siempre en cero, el firmware se comporta exactamente igual que el 17 de junio desde la perspectiva de un usuario en la interfaz web. Su valor es puramente arquitectónico: generaliza la cadena DMA de un esquema fijo de tres canales a un multiplexor de cinco canales gobernado por un único registro de hardware, dejando la puerta abierta a implementar promediado por lotes (hardware averaging) sin tener que volver a tocar la topología de DMA. Es el tipo de "deuda técnica pagada por adelantado" que hace que features futuras (como el promediado de N muestras por trigger para mejorar la relación señal/ruido) sean un cambio incremental — activar `raw_lote`/`filt_lote` y sumar la lógica de promediado del lado del consumidor — en vez de un rediseño completo del subsistema de adquisición.

Elías toma la cadena DMA de tres canales que había cerrado el 17 de junio y la generaliza a un esquema de cinco canales con selección de ruta por registro de hardware, sin activar ninguna funcionalidad nueva visible — el comportamiento del sistema, medido desde la interfaz web, es idéntico al día anterior.
