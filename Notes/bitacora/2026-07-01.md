#### Diff y análisis técnico

**1. El fin de la captura por temporizador: ahora el maestro sabe con certeza cuándo cada esclavo terminó.**

Este es, sin dudas, el cambio de mayor impacto en la confiabilidad del sistema. Hasta ahora, cuando el maestro ordenaba una captura (`CMD_START`) a los esclavos armados, no tenía forma directa de saber cuándo cada uno había terminado de llenar su buffer de muestras: se limitaba a calcular la duración esperada de la captura y programar el volcado de datos (dump) para ese momento estimado, con un margen de seguridad (`scheduleStoreDump`). El problema con este enfoque es evidente: si un esclavo tarda más de lo esperado (por ejemplo, por jitter de reloj o pérdida de paquetes), el maestro podía empezar a pedir datos antes de que la captura realmente terminara.

El fix introduce un mecanismo de confirmación explícita por nodo:

```c
static uint32_t g_captureDoneMask = 0;
...
static bool allArmedCapturesDone()
{
    if (g_armedCount == 0 || g_armAckMask == 0) return false;
    return (g_captureDoneMask & g_armAckMask) == g_armAckMask;
}
```

Y en el manejo del ACK del comando `START`:

```c
/* START ok=2: captura normal completa en el slave. Este es el camino
 * principal; el timer queda solo como seguridad si algún ACK se pierde. */
if (msg.sub_cmd == CMD_START && !g_viewDump) {
    if (msg.node_id > 0 && msg.node_id <= NUM_SLAVES &&
        msg.ok == 2 && g_state == RUNNING) {
        uint32_t bit = (1UL << msg.node_id);
        if (g_armAckMask & bit) {
            g_captureDoneMask |= bit;
            ...
            if (allArmedCapturesDone()) {
                syncOutWrite(LOW);
                beginDump("start_done");
            }
        }
        ...
    }
}
```

Cada esclavo, al terminar de llenar su buffer de captura, envía ahora un `ACK` de `CMD_START` con `ok=2` (un valor nuevo, distinto de `ok=1` que ya se usaba para confirmar la recepción del comando). El maestro acumula estas confirmaciones en una máscara de bits (`g_captureDoneMask`) y dispara el volcado de datos apenas *todos* los nodos armados confirmaron el fin real de su captura — sin esperar innecesariamente si terminan antes de lo estimado, y sin arriesgarse a empezar tarde si alguno tarda más. El temporizador viejo no desaparece, pero cambia de rol: pasa de ser el mecanismo principal a ser una red de seguridad ante la pérdida de algún ACK, y su ventana se duplica y se vuelve más conservadora:

```c
-    uint32_t waitMs = captureDurationMs(nBatches) + STORE_CAPTURE_MARGIN_MS;
-    if (waitMs < STORE_AUTO_DUMP_MIN_MS) waitMs = STORE_AUTO_DUMP_MIN_MS;
+    uint32_t durationMs = captureDurationMs(nBatches);
+    uint32_t waitMs = durationMs * 2u + STORE_CAPTURE_MARGIN_MS;
+    if (waitMs < STORE_CAPTURE_TIMEOUT_MIN_MS) waitMs = STORE_CAPTURE_TIMEOUT_MIN_MS;
```

El renombrado del log de `"auto DUMP in %u ms"` a `"DUMP safety timeout in %u ms"` es elocuente: ya no es el mecanismo esperado de disparo, sino la excepción para cuando algo salió mal.

```cpp

+        /* No START_ACK here: once SYNC rises the PSoC is measuring, so RF stays
```

```js
  const hw = String(hwType || '').trim().toUpperCase();
```

#### Diff y análisis técnico

**1. El rediseño de la EEPROM: de un slot único a un slot por código de ganancia PGA.**

Este es el cambio más profundo del commit. El layout viejo de EEPROM guardaba un único conjunto de valores DAC de calibración, sin importar con qué ganancia de PGA se hubiera calibrado — lo cual significaba que, si el usuario cambiaba la ganancia del PGA (por ejemplo, para adaptarse a una señal más fuerte o más débil), la calibración guardada dejaba de ser válida para la nueva ganancia, pero el firmware no tenía forma de saberlo. El nuevo diseño reserva un slot de EEPROM por cada uno de los 9 códigos de ganancia posibles (`PSOC_NV_GAIN_SLOTS = 9u`):

```c
/*
 * Layout EEPROM v2 (1 fila de 16 bytes por ganancia PGA 0..8):
 *   [0] magic, [1] version, [2] hw_class, [3] pga_code, [4] stage_count,
 *   [5] valid_mask, [6..9] cal_dac[4], [10..13] reserved, [14..15] crc_le.
 */
```

Cada slot lleva ahora su propio `magic`, número de `version` (para poder invalidar formatos viejos si el layout cambia de nuevo en el futuro), el `hw_class` con el que se guardó (para no confundir una calibración de GEO con una de HAMMER si alguna vez comparten EEPROM), el código de ganancia PGA exacto, la cantidad de etapas guardadas, y una máscara de bits (`valid_mask`) que indica exactamente cuáles de esas etapas realmente se calibraron con éxito:

```c
uint8 psoc_nv_load_for_gain(uint8 pga_code, uint8 *cal_dac, uint8 cal_count)
{
    ...
    if (row0[0] != PSOC_NV_MAGIC) { return 0u; }
    if (row0[1] != PSOC_NV_VERSION) { return 0u; }
    if (row0[2] != PSOC_HW_CLASS) { return 0u; }
    if (row0[3] != pga_code) { return 0u; }
    if (row0[4] != cal_count) { return 0u; }
    ...
    valid_mask = row0[5];
    if ((valid_mask & (uint8)((1u << cal_count) - 1u)) !=
        (uint8)((1u << cal_count) - 1u)) {
        return 0u;
    }
    ...
}
```

Cinco verificaciones distintas antes de aceptar un slot como válido — un nivel de paranoia defensiva que tiene sentido para datos que sobreviven un power-cycle y que, si están corruptos o son de un contexto distinto (otra ganancia, otro hardware), podrían sembrar un punto de partida de calibración completamente erróneo sin que nada lo detecte.

```matlab
%   5) Aplica low-pass FIR final con filtfilt
%
% Importante:
%   - El high-pass NO se invierte.
%   - El low-pass NO se invierte.
%   - Ambos son filtros finales de limpieza.
```

Esto es exactamente el procedimiento de deconvolución inversa clásico que se necesita para un geófono como el SM-24 usado en este proyecto: los geófonos son, físicamente, sistemas resonantes de segundo orden con una frecuencia natural propia (típicamente unos pocos Hz), y su respuesta en frecuencia no es plana — atenúan las frecuencias por debajo de su resonancia natural. Para poder interpretar correctamente las velocidades de propagación de ondas superficiales (el objetivo de un ensayo MASW), hace falta "deshacer" ese filtrado natural del sensor, aplicando el inverso de su modelo de transferencia (`model`, discretizado con el método de Tustin, el mismo que ya se usaba en las simulaciones de Simulink documentadas en días anteriores de esta bitácora). El filtro pasa-alto y pasa-bajo finales, explícitamente marcados como "no invertidos", son simplemente limpieza de ruido de muy baja y muy alta frecuencia, sin relación con el modelo físico del sensor.

Y, notablemente, el archivo que lee (`C:\Users\elias\Downloads\captures\002_start_2\geo1\filt.csv`) es exactamente el tipo de estructura de carpetas por nodo (`geo1/filt.csv`) que el commit anterior del mismo día (`b2218dd`) había preparado en `export.js` con su nuevo sistema de roles y nombres de directorio. Es decir: en la misma madrugada, se construyó el camino de exportación con la estructura correcta, y se usó esa misma estructura para procesar por primera vez una captura real — un ciclo completo de ida y vuelta entre firmware/web y análisis de datos, cerrado en unas pocas horas.
