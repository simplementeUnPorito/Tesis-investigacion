#### SPI → UART: el cambio de paradigma

El enlace anterior (SPI desde el 20 de mayo) requería:
- 4 cables SPI (SCLK, MISO, MOSI, SS) + 1 cable DATA_READY
- PSoC como SPI slave (configuración específica en PSoC Creator)
- Timing crítico: el ESP32 debía esperar el flanco de DATA_READY antes de iniciar la transacción

El nuevo enlace UART requiere:
- 2 cables (TX → RX, RX ← TX)
- Baud rate: 460800 bps
- PSoC envía frames cuando tiene datos listos; ESP lee continuamente

```cpp
/* psoc_uart.h — cableado */
#ifndef PSOC_UART_BAUD
#define PSOC_UART_BAUD 460800UL
#endif
#ifndef PSOC_UART_RX
#define PSOC_UART_RX 16    /* GPIO16 ESP32 ← TX PSoC */
#endif
#ifndef PSOC_UART_TX
#define PSOC_UART_TX 17    /* GPIO17 ESP32 → RX PSoC */
#endif
```

El frame UART es más simple que el frame SPI (95 bytes vs 306 bytes):

```cpp
/* Frame PSoC → ESP (95 bytes):
 * [0xAB][n=30][seq_lo][seq_hi] + 30×3 bytes (raw LE: b0,b1,b2) + [crc XOR]
 */
#define PSOC_FRAME_BYTES  (4 + SPI_BATCH_SAMPLES * 3 + 1)   /* 95 */
```

Cada muestra es solo 3 bytes (24-bit raw) en lugar de 10 bytes (raw + analog + digital + gain + flags). La razón: el PSoC ya no tiene el `Filter` (DFB notch 50Hz) en esta arquitectura — el filtrado se hace en el ESP32 o en MATLAB. Esto simplifica el PSoC y acelera el throughput.

A 460800 baud, 95 bytes tarda: 95 × 10 bits / 460800 ≈ 2.06ms. Con 30 muestras a 1020 SPS, un batch dura 30/1020 ≈ 29.4ms — hay margen amplio para transmitir el batch antes de que llegue el siguiente.

#### Máquina de estados PSoC

```c
/* Estados del PSoC */
#define PSOC_IDLE     0u
#define PSOC_ARMED    1u   /* armado: espera flanco SYNC para muestrear */
#define PSOC_SAMPLING 2u
```

El PSoC ahora tiene una máquina de estados que mapea directamente al protocolo del sistema:

```
PSOC_IDLE → (0xB1 pre-start via UART) → PSOC_ARMED
PSOC_ARMED → (flanco SYNC_IN) → PSOC_SAMPLING
PSOC_SAMPLING → (g_batches_sent >= g_n_batches) → PSOC_IDLE
```

Esta es la implementación del store-and-forward en el PSoC: el ESP esclavo envía `0xB1 pre-start` para armar el PSoC, luego levanta el pin SYNC_IN para disparar el muestreo. El PSoC toma exactamente `N` batches (configurable hasta 16 bits = 65535 batches) y luego para.

El ISR del ADC:

```c
CY_ISR(isr_DelSigReady)
{
    g_adc_raw = ADC_GetResult32();

    if (g_state == PSOC_SAMPLING && g_batch_ready == 0u &&
        g_batch_fill < BATCH_SAMPLES)
    {
        g_batch_raw[g_batch_fill] = g_adc_raw;
        g_batch_fill++;
        if (g_batch_fill >= BATCH_SAMPLES)
        {
            g_batch_ready = 1u;
        }
    }
```

La condición `g_batch_ready == 0u` previene que el ISR sobreescriba un batch que el main todavía no transmitió por UART — es la forma más simple de double-buffering sin usar DMA.

#### `N` de 16 bits — experimentos más largos

```c
/* RX UART — set N 16 bits */
/* [0xAB][0xA3][n_lo][n_hi][0xA3^n_lo^n_hi] */
```

Antes, N era de 8 bits (máximo 255 batches = 255 × 30 / 1020 ≈ 7.5 segundos). Ahora N es de 16 bits: hasta 65535 batches = 65535 × 30 / 1020 ≈ 32 minutos de grabación continua. Para un experimento MASW real (que puede durar varios minutos con múltiples impactos), esto es crucial.

#### `debug_log.h` — logging unificado

```c
/* Flujos: humano (Serial) + máquina (prefijo #M,), timestamp micros() */
/* Gateado por DBG_ENABLE: compila a cero en release */
```

Un sistema de logging unificado para master y slave. Los logs de máquina tienen el formato `#M,timestamp,tag,value` — parseable por MATLAB sin necesidad de expresiones regulares complejas. El gate de compilación `DBG_ENABLE` garantiza que en el build de producción no hay overhead de printf.

#### `psoc_uart.h` — la nueva API

```cpp
class PsocUART {
public:
    void begin(BatchCallback cb = nullptr, HardwareSerial *serial = nullptr);
    void poll();           // drena UART, ensambla frames
    
    void setN(uint16_t n);        // CMD_SETN (0xA3)
    void preStart();              // CMD_PRESTART (0xB1)
    void captureNow();            // CMD_VIEW (0xB2)
    void setVdac(uint8_t v);      // CMD_VDAC (0xAA)
    void setPga(uint8_t code);    // CMD_PGA (0xA6)
    void setPgavdac(uint8_t code);// CMD_PGAVDAC (0xA9)
    void debugRamp(bool en);      // CMD_DEBUG (0xB3)
```

La API es limpia y completa. El método `poll()` se llama en el loop del ESP32 para drenar el buffer UART — no necesita ISR porque 460800 baud en un ESP32 es manejable por polling.

### `9193ed52` — "PSoC arranque solo por flanco; quita gráfico/acelerómetro del maestro en MATLAB" — 17:49 UTC (Author: Claude)

Este commit refina el commit anterior con dos cambios:

#### Arranque solo por flanco SYNC

El commit anterior tenía el comando `0xB2 captureNow` ("Ver") que disparaba el muestreo inmediatamente sin esperar el flanco SYNC. Esto creó una race condition: si el ESP esclavo enviaba `captureNow` mientras el PSoC todavía procesaba el pre-start, el estado era inconsistente.

La solución: **el muestreo SIEMPRE arranca por flanco SYNC**. El comando "Ver" ahora arma el PSoC por UART (pre-start) y luego el ESP esclavo levanta el pin SYNC_IN con un pequeño delay para evitar la carrera:

```
"Ver" solicitado desde MATLAB
    │
    ▼
Esclavo: envía 0xB1 pre-start al PSoC (→ PSOC_IDLE → PSOC_ARMED)
Esclavo: espera ~50µs
Esclavo: levanta SYNC_IN (GPIO_OUT HIGH)
PSoC: ISR flanco → PSOC_ARMED → PSOC_SAMPLING
```

El delay de 50µs es suficiente para que el pre-start UART llegue al PSoC y la máquina de estados procese la transición, eliminando la race condition.

#### `InterfaceESP.m` simplificada

El tab Maestro en MATLAB pierde el control del acelerómetro del martillo (que ahora es un "esclavo normal" como se indica en el commit anterior). Esto simplifica la GUI y refleja la nueva arquitectura: el maestro ya no lee el martillo directamente — si hay un martillo instrumentado, sería otro nodo esclavo con su propio PSoC.
