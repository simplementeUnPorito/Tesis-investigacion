### Desarrollo detallado: El nuevo protocolo de 4 bytes

El cambio más importante del commit es el rediseño del protocolo PSoC→MATLAB. La versión anterior usaba 5 bytes fijos:

```
[0x56][0x00][b2][b1][b0]   — 5 bytes, datos
[0x56][0x01][0x00][0x00][0x00]   — 5 bytes, heartbeat
```

La nueva versión comprime a **4 bytes** para datos y HB, y mantiene 5 bytes para paquetes de configuración/estado:

```
% Protocolo RX — 4 bytes datos/HB, 5 bytes cfg/estado:
%   Data:     [0x56][TYPE:4|CH:2|D17:1|D16:1][D15:D8][D7:D0]  18-bit signed
%   HB:       [0x56][0x10][0x00][0x00]
%   CFG-ADC:  [0x56][0x20][res][fsH][fsL]
%   CFG-PGA:  [0x56][0x30|code][vrefH][vrefL][0]
%   VREF_ST:  [0x56][0x40][vdac_p][vdac_n][0x01]
%   VREF_CFG: [0x56][0x50][vdac_p][vdac_n][0x00]
%   AMUX_ST:  [0x56][0x60|ch][0][0][0]
```

El diseño es ingenioso: el segundo byte es multifunción. Para paquetes de **datos**:
```
byte[1] = [TYPE:4 bits | CH:2 bits | D17:1 bit | D16:1 bit]
```

Los bits 7-4 indican el tipo (0x0 = datos), bits 3-2 indican el canal (0-3 para AMUX), y bits 1-0 son los 2 bits más significativos de los 18 bits del ADC. Los bytes 2 y 3 tienen los 16 bits restantes:

```
valor_18bit = (byte[1] & 0x03) << 16 | byte[2] << 8 | byte[3]
```

La decodificación en sign-extension a 18 bits:
```
si bit 17 = 1:  valor_signed = valor_18bit - 2^18 = valor_18bit - 262144
sino:           valor_signed = valor_18bit
```

Esto reduce el overhead del protocolo de 5 a 4 bytes por muestra, un ahorro del 20% en bandwidth UART.

Para los paquetes de configuración y estado, el nibble alto del byte 1 distingue el tipo:
- `0x2_` → CFG-ADC: respuesta con resolución, fs
- `0x3_` → CFG-PGA: código PGA + referencia de voltaje
- `0x4_` → VREF_ST: estado actual de VDAC_p y VDAC_n
- `0x5_` → VREF_CFG: confirmación de configuración VDAC
- `0x6_` → AMUX_ST: canal actual del MUX analógico

Este diseño permite que el mismo stream UART lleve datos y metadatos entremezclados sin necesidad de un canal separado.

#### El AMUX: medir modo común

```matlab
S.amux_ch    = 0;   % 0=diferencial, 1=CM positiva, 2=CM negativa
```

El **AMUX** (Analog Multiplexer) permite conmutar la entrada del ADC entre tres señales:
1. **Canal 0 (diferencial)**: V+ − V− del geófono → la señal útil
2. **Canal 1 (CM positiva)**: La línea positiva del geófono respecto a masa → modo común positivo
3. **Canal 2 (CM negativa)**: La línea negativa del geófono → modo común negativo

¿Para qué medir el modo común? Para diagnosticar:
- Si hay interferencia de 50Hz en modo diferencial, ¿viene de las dos líneas igual (modo común) o es asimétrico?
- El rechazo de modo común (CMRR) del amplificador diferencial se puede medir comparando CM y diferencial
- Si CM ≫ diferencial, hay un problema de alineación de las referencias

Los comandos para controlar el AMUX desde MATLAB:
```matlab
write(S.sp, uint8([0xA7, uint8(ch_nuevo)]), 'uint8');  % cambiar canal AMUX
write(S.sp, uint8([0xAA, uint8(pV), uint8(nV)]), 'uint8');  % cambiar VDAC
```

El comando `0xA7` setea el canal del AMUX; `0xAA` ajusta las referencias VDAC. Estos son comandos del PC → PSoC, en contraste con los paquetes 0x56 que van del PSoC → PC.

#### `BandLimitedOpAmp.slx` — El modelo de op-amp real

La aparición de un nuevo modelo Simulink `BandLimitedOpAmp.slx` indica que se está modelando el comportamiento de un op-amp real en la simulación del circuito analógico compensador.

Un op-amp ideal tiene:
- Ganancia infinita (A0 → ∞)
- Ancho de banda infinito (fp → ∞)
- Impedancia de entrada infinita (Rin → ∞)

El op-amp real en `BandLimitedOpAmp`:
- Ganancia finita: A0 = 90dB = 31,623 V/V
- Polo dominante a fp = 8MHz
- Función de transferencia: `A(s) = A0 / (1 + s/ωp)`

Con estos parámetros, el GBP (Gain-Bandwidth Product) es:
```
GBP = A0 × fp = 31,623 × 8MHz ≈ 253 MHz
```

Esto es consistente con op-amps de propósito general rápidos (como el TL071, TLV2371, o similares). A las frecuencias de trabajo del compensador (1-100 Hz), el op-amp todavía tiene ganancia muy alta (~30,000×), pero el modelo captura los efectos de fase y ganancia que se desvían del ideal a frecuencias más altas.

### Desarrollo detallado: `graficar_compensador_csv.py` — Validación con op-amp real

Este script Python de 157 líneas es la herramienta de validación del compensador analógico que considera las no-idealidades del op-amp.

#### El modelo matemático del op-amp real

```python
A0_dB   = 90        # ganancia DC: 10^(90/20) = 31,623 V/V
f_p     = 8e6       # polo dominante: 8 MHz
omega_p = 2*np.pi * f_p
Rin     = 35e6      # impedancia de entrada: 35 MΩ
```

La función de transferencia ideal del MFB compensador:
```python
def H_ideal(R1, R2, C1, C2, freqs_hz):
    s = 1j * 2*np.pi * freqs_hz
    beta = (s*R2*C1) / ((s*R1*C1 + 1) * (s*R2*C2 + 1))
    return -beta
```

La **beta** es la función de realimentación del MFB:
```
β(s) = s·R2·C1 / [(1 + s·R1·C1)(1 + s·R2·C2)]
```

Con el op-amp ideal (A0 → ∞), la ganancia de lazo cerrado es `-β`, que es la función de transferencia del compensador MFB.

La transferencia **real** con op-amp finito:
```python
def H_real(R1, R2, C1, C2, freqs_hz):
    s      = 1j * 2*np.pi * freqs_hz
    beta   = (s*R2*C1) / ((s*R1*C1 + 1) * (s*R2*C2 + 1))
    z_load = (R2 / Rin) / (s*R2*C2 + 1)  # efecto de carga por Rin
    nif    = (1 + s/omega_p) / A0         # factor no-ideal (sin A real)
    return -beta / (1 + nif * (1 + beta + z_load))
```

El denominador `1 + nif × (1 + β + z_load)` modifica la ganancia de lazo cuando `nif ≠ 0` (es decir, cuando el op-amp tiene ganancia finita o limitación de ancho de banda).

Para frecuencias donde `|A(jω)| ≫ 1`:
- `nif → 0`, `H_real → -β = H_ideal` ✓
- La aproximación ideal es válida mientras estemos mucho por debajo del polo

Para f = 100 Hz (frecuencia máxima de interés):
```
A(j·2π·100) = A0 / (1 + j·100/8MHz) ≈ A0 × (1 - j·100/8MHz)
               ≈ 31,623 × (1 - j·1.25e-5)  [error < 0.001%]
```

En la banda de trabajo del geófono (0.1-100 Hz), el op-amp es prácticamente ideal. Pero el script permite verificarlo cuantitativamente para cada candidato de componentes.
