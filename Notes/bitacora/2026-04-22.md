## `geophone_scope.m` — El osciloscopio MATLAB (657 líneas)

El `geophone_scope.m` es la alternativa MATLAB al `geophone_scope.py` del 19 de abril. ¿Por qué hacer una segunda versión en MATLAB si ya existe en Python?

**Razones técnicas:**
1. **MATLAB tiene mejor soporte para análisis de señales** en tiempo real — sus funciones de filtrado (`filter`, `butter`, `fir1`) están disponibles interactivamente
2. **El entorno de trabajo principal es MATLAB** — los scripts de simulación del SM-24 son todos `.m`, tiene más sentido analizar los datos en el mismo entorno
3. **PyQtGraph vs. uifigure**: la GUI de Python requería PyQt instalado; la de MATLAB usa `uifigure` que es nativa de MATLAB R2025b

**El protocolo del `geophone_scope.m`:**

```matlab
% Protocolo RX (PSoC -> PC, 10 bytes, 921600 baud):
%   [0xAA][0x00][sar_hi][sar_lo][ds2][ds1][ds0][flt2][flt1][flt0]
%   sar = int16  big-endian
%   ds  = int24  big-endian signed
%   flt = int24  big-endian signed
%
% Protocolo TX (PC -> PSoC):
%   [0xBB][0x03][fhi][flo][cksum]   freq_dh = Hz*10, uint16 big-endian
```

Este protocolo es diferente al del Python (`geophone_scope.py`). El Python tenía paquetes de 12 bytes con FIR output; el MATLAB tiene 10 bytes con los 3 canales (SAR + DelSig + DFB filter). Esto implica que hay **dos firmwares PSoC diferentes corriendo en paralelo** — uno para el testbench Python y otro para el MATLAB scope.

El `geophone_scope.m` implementa 4 ejes simultáneos:
- **CH1 (amarillo):** SAR — entrada 12-bit, ±1.024 V
- **CH2 (cyan):** ADC DelSig crudo — 18-bit, ±6.144 V
- **CH3 (magenta):** Filtro DFB de salida
- **CH4 (verde):** Filtro digital aplicado en MATLAB (FIR/IIR seleccionable por el usuario)

```matlab
S.fs        = 3000;         % Frecuencia de muestreo del scope
S.sar_mV    = 0.5;          % Conversión SAR: 0.5 mV/count
S.ds_mV     = 1000.0 / 21333.0;  % Conversión DelSig: ~0.047 mV/count
```

El scope usa 3000 Hz como frecuencia de muestreo, aunque el ADC_DelSig corre a ~1500 SPS. Los 3000 Hz pueden ser el double-rate incluido el log de debug, o es una constante legacy.

## `compensadorKai.m` y `compensadorPazos.m` — Extensión de banda del geófono

Los dos scripts de compensador abordan un problema fundamental para el MASW de baja frecuencia: **el geófono SM-24 no mide bien por debajo de su frecuencia natural**.

### El problema de la frecuencia natural

La función de transferencia del SM-24 como sensor de velocidad es:

$$H_v(s) = \frac{G \cdot s^2}{s^2 + 2h_0\omega_n s + \omega_n^2}$$

donde $\omega_n = 2\pi \cdot f_n$, $h_0$ es el amortiguamiento, $G$ es la sensibilidad.

Esta función tiene un cero doble en $s=0$ (DC) y dos polos en $s = -h_0\omega_n \pm j\omega_n\sqrt{1-h_0^2}$. Por debajo de $f_n$, la ganancia cae a 40 dB/dec — la señal sísmica queda sepultada en ruido.

Para MASW de baja frecuencia (Rayleigh waves en 1-10 Hz, que corresponden a profundidades de 30-100 m), un geófono de 4.5 Hz tiene un problema serio: las ondas de 1-4 Hz son prácticamente invisibles.

### `compensadorKai.m` — El compensador analítico

```matlab
fn = 10;        % Frecuencia natural (Hz) — nota: SM-24 es 4.5 Hz
h0 = 0.25;      % Amortiguamiento original
G  = 28.8;      % Sensibilidad
fLow = 0.01;    % Objetivo: extender banda hasta 0.01 Hz

h1 = wn / (2 * wLow);      % h1 = 10π / 0.02π = 500
K  = h1 / h0;              % K = 500 / 0.25 = 2000

Hcomp = 1 - (2*(h1-h0)*wn*s) / (s^2 + 2*h1*wn*s + wn^2);
Hv_comp = K * Hv * Hcomp;
```

La técnica de `compensadorKai.m` es una **compensación de segundo orden**: el compensador $H_{comp}$ modifica los ceros del sistema para "cancelar" los efectos de la resonancia y extender la banda hacia abajo. 

El amortiguamiento $h_1 = 500$ es físicamente irrealizable en hardware (requeriría una resistencia de carga de varios megohmios conectada al geófono), pero en el dominio digital es perfectamente implementable como un filtro IIR. Este es exactamente el tipo de procesamiento que el FIR Q1.15 del PSoC puede implementar.

El `fn=10` en el script (en lugar de 4.5) puede indicar que el usuario estaba trabajando con un **geófono diferente** (tal vez el SM-24 tiene fn=10 Hz en la configuración del laboratorio, o es el mismo geófono pero con amortiguamiento diferente), o que usó 10 Hz como caso de prueba más sencillo de visualizar.

### `compensadorPazos.m` — La búsqueda exhaustiva

El script de Pazos usa un enfoque diferente: búsqueda por grilla sobre el espacio de parámetros (R_L, C):

```matlab
R_L_vec = logspace(log10(100), log10(100000), 150);  % 100 Ω a 100 kΩ
C_vec   = logspace(log10(1e-6), log10(1000e-6), 150); % 1 µF a 1000 µF
target_damping = 1 / sqrt(2);  % Butterworth (0.707)
```

La búsqueda evalúa 150 × 150 = 22,500 combinaciones de R_L y C, buscando cuáles dan un amortiguamiento exactamente Butterworth ($h = 1/\sqrt{2}$). El criterio Butterworth es el óptimo para la respuesta de paso: maximally flat passband sin ripple.

El enfoque de Pazos es más riguroso que el de Kai: en lugar de derivar analíticamente el compensador, explora el espacio de parámetros del circuito RC + geófono para encontrar los valores de componentes que dan la respuesta deseada. Es el mismo principio que `FiltroPasivoEntrada.m` del 17 de abril, pero aplicado a la extensión de banda en lugar de al filtro de entrada.

**¿Por qué dos enfoques?**

El "compensadorKai" probablemente es la referencia teórica (el método estándar de la literatura, posiblemente del libro de Scherbaum "Of Poles and Zeros" o similar). El "compensadorPazos" es el enfoque empírico del usuario — buscar los componentes reales que implementan la compensación. Tener ambos permite validar que los componentes encontrados por búsqueda dan la misma respuesta que la teoría predice.

## `ComparacionDiferencial.cydsn` — El PSoC del MATLAB scope

El proyecto PSoC `ComparacionDiferencial.cydsn` es específico para el MATLAB scope. Sus archivos C (`AnalogToDigital.c`, `Communication.c`) son versiones del firmware del 19 de abril pero adaptadas al protocolo de 10 bytes del `geophone_scope.m`.

El nombre "ComparacionDiferencial" es más descriptivo que "Comparation": el circuito compara la señal diferencial del SM-24 (entrada diferencial al INA) con la salida single-ended del ADC. La "comparación" aquí es entre el modo diferencial (que rechaza ruido de modo común) y el modo single-ended (que lo acepta).

El `.hex` compilado (4626 líneas) confirma que el firmware compila y está listo para grabar en el PSoC 5LP.

## El "LocuraCCp" descifrado

El mensaje de commit "LocuraCCp" puede leerse como:

- **Locura**: el estado emocional — este commit concentra tres días de trabajo caótico
- **CC**: "Circuit Comparation" o "Canal Compensado" — referencia al circuito diferencial y el compensador  
- **p**: "prueba" — los scope logs son datos de prueba, no mediciones finales

O más simplemente: es la "locura" del CC (PSoC Cypress CY8CKIT — la plataforma) en modo prueba. Como "XDXDXD" o "AYUDAAAAAA", es un marcador emocional más que un descriptor técnico.
