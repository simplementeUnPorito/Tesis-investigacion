## Commits

### `a207a4da` — "Add HELLO beacon, sample pulse, IMU support" — 04:48 hs

**Un commit a las 4:48 AM después del 24 de mayo.** La noche entera trabajando en hacer que los ESP32 se hablen entre sí.

#### `CMD_HELLO` — el diagnóstico más básico

El primer problema que aparece con ESP-NOW: ¿cómo saber si el esclavo está ahí? El maestro no tiene visibilidad de la presencia de los esclavos hasta que estos intentan enviar datos. La solución: un beacon periódico.

```cpp
#define CMD_HELLO      0x40   /* beacon diagnóstico: esclavo → maestro */

struct MsgHello {
    uint8_t cmd;      /* CMD_HELLO */
    uint8_t node_id;
};
```

El esclavo envía `MsgHello` cada N segundos en modo IDLE. El maestro lo recibe, actualiza la tabla de presencia, y notifica a MATLAB con un paquete especial. En la GUI de MATLAB aparece el indicador "Maestro" o "Esclavo X" en verde cuando el HELLO llega.

Sin este beacon, el operador no podría saber si el esclavo está encendido, si el ESP-NOW funciona, o si la dirección MAC está bien configurada. Es lo primero que debe implementarse en cualquier sistema distribuido — el equivalente a un "ping".

#### `SAMPLE_PULSE` — el osciloscopio no miente

Otro diagnóstico nuevo: un pulso GPIO por cada muestra procesada.

```c
/* Configurable en platformio.ini:
 *   -DSAMPLE_PULSE_PIN=23
 *   -DSAMPLE_PULSE_IDLE=0
 *   -DSAMPLE_PULSE_WIDTH_US=20
 */
```

Con el osciloscopio conectado al pin `SAMPLE_PULSE_PIN`, se puede ver en tiempo real cuándo el sistema está generando muestras. Si la frecuencia del pulso coincide con fs (ej. 1020 Hz → un pulso cada ~980µs), el ADC está funcionando. Si los pulsos son irregulares, hay algo mal en el timing del DFB o del SPI.

Este tipo de diagnóstico de hardware — un pulso GPIO que es visible en el osciloscopio — es la forma más directa de verificar la temporización real del sistema embedded. Los debuggers de software no pueden ver timing de µs; el osciloscopio sí.

#### Soporte IMU — el martillo instrumentado

El commit reemplaza el stub del martillo ADC por una abstracción de IMU:

```c
/* Flags en platformio.ini:
 *   -DHAMMER_IMU_ENABLED=1    usar IMU en lugar de ADC para el martillo
 *   -DHAMMER_BTN_ENABLED=0    deshabilitar botón hardware
 */
```

El martillo de impacto PCB Piezotronics 086D20 tiene un acelerómetro/IMU integrado que mide la fuerza del impacto en tiempo real. Con `HAMMER_IMU_ENABLED=1`, el maestro ESP32 lee el IMU y lo incluye en los datos enviados a MATLAB, permitiendo:
1. Detectar automáticamente el instante del impacto (sin marcar manualmente)
2. Medir la fuerza del golpe para normalizar los datos
3. Detectar doble-golpe (rebote del martillo) que contaminaría el apilado coherente

El sampling del IMU del martillo se mueve a ~1020 Hz (mismo que el ADC del geófono) para que las muestras del martillo y del geófono estén temporalmente alineadas.

#### `MatlabTransport` — cola de comandos

Un bug encontrado durante el debug: si MATLAB envía dos comandos TCP back-to-back sin delay entre ellos, el maestro puede perder el segundo. La solución: una cola circular en `MatlabTransport`.

```cpp
/* Queue incoming MATLAB commands to avoid dropped back-to-back commands */
```

En protocolos serial simples sin cola, la lectura del buffer en el loop principal puede perder bytes si el procesamiento de un comando tarda más que el tiempo entre comandos. Una cola de comandos completos (no bytes individuales) resuelve esto.

#### WiFi mode — AP_STA para mejor compatibilidad ESP-NOW

```cpp
/* master AP_STA; ESP8266 slave uses hidden AP on channel 1 */
```

ESP-NOW requiere que los dispositivos estén en el mismo canal WiFi. Si el maestro está en modo AP (canal configurable) y los esclavos en modo STA (se conectan al AP), el canal se negocia automáticamente. El modo AP_STA del maestro (Access Point + Station simultáneo) permite que MATLAB se conecte via TCP mientras el maestro también opera en el mismo canal que los esclavos para ESP-NOW.

El ESP8266 (versión más barata del esclavo) tiene una limitación: como STA, no puede cambiar el canal WiFi libremente. La solución es crear un AP oculto en el canal 1 en el ESP8266 para fijar el canal, y configurar el ESP32 maestro para trabajar también en canal 1.

Los drivers CP210x son necesarios para que Windows reconozca el ESP32 como puerto COM. Sin ellos, `COMxx` no aparece en MATLAB y no hay forma de conectarse al maestro. Este es el tipo de problema que aparece cuando el sistema se prueba por primera vez en una máquina diferente — alguien (o el mismo Elías en otro PC) intentó conectarse y el dispositivo no fue reconocido.

El directorio `CP210x_Windows_Drivers/` contiene:
- Instaladores x64 y x86
- drivers `.sys` para ambas arquitecturas
- `.inf` de configuración
- `.cat` de firma digital
- Release notes de la versión v6.7.6

La historia detrás del "ayuda": el sistema ESP32 compila, flashea, y el firmware funciona. Pero cuando se conecta al PC de prueba (que puede ser un laptop de laboratorio o de un compañero), Windows no detecta el ESP32 porque falta el driver. La sensación es de frustración total — todo funciona, pero el sistema no se puede usar.

El commit también incluye un nuevo proyecto PSoC `DiferencialToSingleEnded__ESP.cydsn` (con dos guiones bajos). Esto sugiere que el proyecto ESP original tenía algún conflicto de nombre con el proyecto existente, y se creó una copia con nomenclatura diferente. El `.hex` actualizado indica que el PSoC fue recompilado en este punto, probablemente con ajustes para el SPI slave interface.

### `22fe14be` — "Fix batch/dump handling, debug timing, and UI" — 21:53 hs

**30 minutos después del "ayuda".** El debugging activo continúa.

#### El bug del batch detection

La descripción del commit es técnica y precisa:

> `master/src/espnow_rx.h: add ReassembledBatch::active and improve batch reset logic to detect new batches even when seq==0.`

El bug: los batches de datos del esclavo se envían en dos partes (`part=0` y `part=1`, 15 muestras cada una). El maestro los ensambla en `ReassembledBatch`. El problema: si el número de secuencia `seq==0` (primer batch después de un START), la lógica de detección de "nuevo batch" no lo reconocía como nuevo — pensaba que era continuación del batch anterior (que también tenía `seq==0` de la sesión anterior).

La solución: agregar el campo `active` a `ReassembledBatch`:

```cpp
struct ReassembledBatch {
    bool     active;       // NUEVO: este batch está siendo ensamblado
    uint8_t  node_id;
    uint16_t seq;
    // ...
};
```

Cuando el maestro recibe `part=0` de un batch y `active=false`, resetea el batch y lo marca `active=true`. Cuando recibe `part=1` del mismo `node_id+seq`, lo agrega. Cuando están las dos partes: callback, `active=false`. Simple pero necesario.

#### DUMP mode — store-and-forward

> `master/src/main.cpp: enforce DUMP mode filtering in onBatchReady, only accept expected node/seq during dump, and preserve ARMED state for store-and-forward start behavior.`

El "DUMP mode" es una modalidad donde los esclavos guardan datos localmente durante la medición (en RAM del ESP32) y los envían al maestro después de recibir CMD_STOP — en lugar de enviar en tiempo real durante la medición. Esto es útil para mediciones cortas donde la confiabilidad es más importante que el streaming en tiempo real.

El bug: en modo DUMP, el maestro recibía batches fuera de orden (el esclavo los envía post-STOP sin esperar ACKs) y los procesaba como si fueran en tiempo real, causando confusión en el ensamblado.

La solución: cuando el maestro está en modo DUMP, `onBatchReady` filtra por `node_id` y `seq` esperado — solo acepta el batch que corresponde al slot actual de recolección.

#### Debug timing determinístico

> `slave/src/main.cpp: add g_debug_last_us, use g_debug_last_us for periodic fake-batch generation.`

El modo debug del esclavo genera "fake batches" (datos sintéticos de rampa) para verificar el canal de comunicación sin conectar el PSoC. El problema: el timing de los fake batches era no determinístico (dependía del loop de Arduino, que tiene latencias variables). La solución: `g_debug_last_us` registra el timestamp del último batch enviado y el siguiente se genera exactamente `BATCH_PERIOD_US` después.

### `9b89dddd` — "Add START probe/ACK and debug hardware pulse" — 22:28 hs

**35 minutos más tarde.** El commit más sofisticado del día.

#### El problema de latencia de START

El comando CMD_START tiene que llegar a todos los esclavos "al mismo tiempo" para que el muestreo sea simultáneo. En la práctica, ESP-NOW tiene una latencia variable de ~1ms a ~5ms dependiendo de la carga del canal WiFi. Para MASW, la sincronización debe ser mejor que 0.5ms (la mitad del período de muestreo a 1000 Hz).

La solución ya existía en el diseño: el GPIO hardware sync (maestro→esclavos via cable). Pero ¿cómo saber cuánta latencia hay en el camino del mensaje ESP-NOW? El "START probe" mide esto.

#### `CMD_START_ACK` y `MsgStartAck` — medir la latencia

```cpp
#define CMD_START_ACK  0x15   /* esclavo → maestro: recibió CMD_START */

struct MsgStart {
    uint8_t  cmd;
    uint64_t t_start_us;
    uint8_t  target_node;  // NUEVO: 0xFF=todos, o node_id específico
};

struct MsgStartAck {
    uint8_t  cmd;
    uint8_t  node_id;
    uint8_t  status;       // 0=ignorado, 1=start real, 2=probe
    uint32_t start_token;  // low32(t_start_us), para asociar ACK con envío
    uint32_t rx_us;        // timestamp de recepción en el esclavo
};
```

El protocolo de probe:

```
Maestro:
1. Genera start_token = (uint32_t)micros()
2. Envía MsgStart con status=probe, target_node=1
3. Registra t_tx = micros()

Esclavo 1:
4. Recibe MsgStart (probe)
5. Registra rx_us = micros()
6. Envía MsgStartAck con rx_us y start_token

Maestro:
7. Recibe MsgStartAck
8. t_rx = micros()
9. RTT = t_rx - t_tx
10. Latencia estimada = RTT/2  (asumiendo canal simétrico)
```

El resultado se envía a MATLAB como paquete 0xFC con la latencia medida. La GUI muestra el RTT de cada nodo para que el operador pueda decidir si la sincronización es adecuada antes de iniciar una medición real.

#### Debug GPIO pulse en START

Además del START probe, se agrega un pulso GPIO en el PIN de debug cuando el maestro envía CMD_START real:

```c
/* Master: emite pulso hardware en START real */
/* platformio.ini: -DDEBUG_START_PULSE_PIN=2 */
```

Con un osciloscopio, se puede medir el tiempo entre el flanco del pulso del maestro y el flanco del GPIO `SYNC_OUT_PIN 25`. Si el delay es menor a 1µs, la secuencia `GPIO → ESP-NOW` es correcta. Si hay jitter, puede indicar que el GPIO está siendo retrasado por la radio WiFi del ESP32 (conocido problema: la radio WiFi puede interrumpir el código de usuario hasta por 200ms en modo AP).

#### `CMD_SET_RECLEN` y `CMD_REQ_BATCH` — store-and-forward robusto

```cpp
#define CMD_SET_RECLEN 0x50   /* master → todos: cuántos batches grabar */
#define CMD_REQ_BATCH  0x51   /* master → esclavo: pedir batch específico */

struct MsgSetRecLen {
    uint8_t  cmd;
    uint16_t n_batches;
};

struct MsgReqBatch {
    uint8_t  cmd;
    uint8_t  node_id;
    uint16_t batch_seq;
};
```

En modo store-and-forward, el esclavo graba `n_batches` batches en RAM local. Después de STOP, el maestro puede pedir cualquier batch específico por `batch_seq` en orden aleatorio — útil si un batch fue perdido en la transmisión inicial (el maestro puede re-pedirlo).

Este es el primer esquema de retransmisión del sistema: hasta ahora, si un paquete ESP-NOW se perdía, se perdía para siempre. Con `CMD_REQ_BATCH`, hay recuperación de paquetes perdidos.
