## Contexto del día

El problema del offset DC en el front-end analógico del PSoC lleva semanas en la lista de pendientes. El geófono SM-24 es un dispositivo de señales AC (respuesta plana aproximadamente entre 10 Hz y varios cientos de Hz), pero la cadena analógica del PSoC introduce inevitablemente un offset DC en los puntos de suma: los amplificadores operacionales (OPAref, PGAp, PGAn, OPAbp, OPAadder, OPAlp) tienen corrientes de polarización, offsets de entrada, y la tensión de referencia del VDAC nunca es exactamente 2.5 V en todos los chips por igual.

Ese offset DC tiene dos efectos adversos. El primero es trivial: llena parte del rango dinámico del ADC con una componente constante. El segundo es más sutil: en el ADC_DelSig de modo diferencial, un offset asimétrico entre los dos caminos del diferencial puede aparecer como ruido correlacionado con la señal de alimentación. El DFB notch de 50 Hz que se exploró meses atrás era una forma de mitigarlo en frecuencia; la auto-calibración que se está implementando ahora es la forma de mitigarlo en DC.

La idea es simple en papel: al encender el PSoC, antes de armar el primer disparo de captura, ejecutar un algoritmo bloqueante que ajuste cada VDAC de referencia del front-end para que la lectura diferencial del ADC sea lo más cercana posible a cero. Si la referencia de tierra virtual del sistema es 2.5 V (que es lo que el VDAC_ref ajusta), y se aplica esa misma referencia como entrada del AMux_IN canal 1, entonces "calibrar" cada etapa significa buscar el código DAC [0–255] tal que el ADC diferencial lea ~0 counts.

La complejidad técnica no está en la idea sino en la ejecución dentro del entorno del PSoC Creator / PSoC 5LP:
- El ADC_DelSig tarda en asentarse después de cambiar configuración o canales del multiplexor analógico.
- El VDAC tiene resolución de 16 mV/LSB (rango 0–4.08 V en modo 1×), lo cual limita la precisión alcanzable.
- Hay cuatro etapas en la cadena GEO (PGA, BP, ADDER, LP) que se calibran secuencialmente, y cada una puede interactuar con las demás.
- El proceso debe ser lo suficientemente rápido como para que el ESP32 no time-out esperando que el PSoC responda.

Para abordar todo esto, este domingo Elías hace un refactor sustancial del firmware del PSoC: extrae la lógica de hardware en módulos separados, implementa el lazo de calibración por búsqueda binaria, y agrega un protocolo de telemetría de diagnóstico para que el ESP pueda monitorear el proceso en tiempo real.

##### 2. El algoritmo de calibración: búsqueda binaria sobre el espacio DAC [0,255]

El archivo `calibration.c` (315 líneas) implementa el lazo de auto-calibración. La estructura central es `calibrate_stage()`, que recibe una descripción de etapa (`PsocCalStage`) y escribe el resultado en un `PsocCalResult`.

```c
typedef struct {
    const char *name;
    uint8 adc_channel;
    int32 target_counts;
    uint8 avg_n;
    uint16 max_iter;
    int32 tolerance_counts;
    int8 direction;
    PsocCalVdacWrite write;
} PsocCalStage;
```

El algoritmo tiene tres fases:

**Fase 1 — Inicialización**: Selecciona el canal AMux_ADC correspondiente a la etapa, espera que el multiplexor se asiente (`CyDelay(CAL_AMUX_SETTLE_MS)`), y arranca el ADC. Luego escribe `CAL_DAC_INIT = 0x9C` (156 en decimal) en el VDAC. Este código corresponde a `156 × 16 mV = 2.496 V ≈ 2.5 V`, el punto central del rango del VDAC8 en modo 1×.

**Fase 2 — Detección empírica de pendiente**: Para saber si el ADC sube o baja cuando el código DAC sube, se aplica un paso `CAL_PROBE_STEP = 32` en la dirección sugerida por `stage->direction`. Si la lectura mejora (error menor), el algoritmo sabe que esa dirección es la correcta; si empeora, invierte. Esto resuelve la ambigüedad de que en diferentes configuraciones del front-end analógico, un VDAC puede tener polaridad positiva o negativa respecto a la lectura del ADC.

**Fase 3 — Búsqueda binaria**: Con la pendiente conocida, se bisecta el espacio [0,255]:

```c
for (iter = 0u; iter < stage->max_iter; iter++) {
    uint8 go_up = increasing
        ? ((measured < stage->target_counts) ? 1u : 0u)
        : ((measured > stage->target_counts) ? 1u : 0u);
    uint8 next_dac;

    if (go_up) {
        if (dac == 255u) { break; }
        lo = (uint8)(dac + 1u);
    } else {
        if (dac == 0u) { break; }
        hi = (uint8)(dac - 1u);
    }
    if (lo > hi) { break; }

    next_dac = (uint8)(lo + ((hi - lo) / 2u));
    if (next_dac == dac) { break; }
    dac = next_dac;

    stage->write(dac);
    CyDelay(CAL_DAC_SETTLE_MS);
    if (!stable_avg(stage->avg_n, &measured)) { ... }
```

La convergencia es logarítmica: con `CAL_MAX_ITER = 12`, el algoritmo puede, en el peor caso, explorar 2^12 = 4096 combinaciones, mucho más que las 256 posibles. En la práctica, converge en 7–9 pasos (log₂(256) ≈ 8) si la planta es monótona.

La función `stable_avg()` es una pieza clave para la robustez:

```c
static uint8 stable_avg(uint8 n, int32 *avg_out)
{
    int32 prev_avg;
    int32 cur_avg;
    uint8 settle_iter;

    if (!avg_counts(n, &prev_avg)) { return 0u; }
    cur_avg = prev_avg;
    for (settle_iter = 0u; settle_iter < CAL_SETTLE_MAX_WINDOWS; settle_iter++) {
        if (!avg_counts(n, &cur_avg)) { return 0u; }
        if (abs_counts(cur_avg - prev_avg) <= CAL_SETTLE_TOL_COUNTS) {
            break;
        }
        prev_avg = cur_avg;
    }
    *avg_out = cur_avg;
    return 1u;
}
```

En lugar de tomar una muestra inmediata después de escribir el DAC, `stable_avg()` espera que dos ventanas consecutivas de `n=32` muestras promediadas difieran en ≤ `CAL_SETTLE_TOL_COUNTS = 100` counts. Esto asegura que el amplificador de la etapa anterior se asentó antes de tomar la medida. Sin esta lógica, el ADC podría leer el transitorio del escalón del DAC y tomar una decisión de búsqueda incorrecta.

Los targets para GEO son todos `0L` (cero counts diferenciales): el objetivo es que cada etapa tenga offset cero cuando la entrada es la referencia de tierra virtual (AMux_IN canal 1 = 2.5 V). Para HAMMER son `52429L` (~1 V) y `78644L` (~1.5 V) porque el front-end del martillo usa una topología single-ended diferente.

El parámetro `direction` = `1` en todas las etapas indica que el autor asume que las lecturas crecen con el código DAC. Esto puede ser incorrecto para alguna etapa específica del GEO (dependiendo de la polaridad del OPA realimentador), y el algoritmo lo corrige empíricamente en la Fase 2. El comentario en `HANDOFF_CALIBRATION.md` marca esto como una pregunta abierta (#4).

##### 4. Protocolo de telemetría PSOC_CMD_DIAG_EVT (0xC4)

Para que el ESP pueda monitorear el proceso de calibración, se agrega un nuevo frame de diagnóstico de 6 bytes:

```
[0xAB][0xC4][event][value][psoc_state][CRC]
```

donde `CRC = 0xC4 ^ event ^ value ^ psoc_state`. En `psoc_hw.h` se definen 27 eventos:

```c
#define PSOC_EVT_BOOT             0x01u
#define PSOC_EVT_ANALOG_READY     0x02u
#define PSOC_EVT_CAL_START        0x10u
#define PSOC_EVT_CAL_DONE         0x11u
#define PSOC_EVT_CAL_BUSY         0x12u
#define PSOC_EVT_CAL_STAGE_DAC    0x13u
#define PSOC_EVT_CAL_STAGE_MEAS   0x14u
#define PSOC_EVT_WAIT_ESP         0x20u
...
#define PSOC_EVT_CAPTURE_DONE     0x36u
#define PSOC_EVT_DUMP_START       0x37u
#define PSOC_EVT_DUMP_DONE        0x38u
```

El PSoC llama a `uart_send_diag(event, value)` en puntos clave del flujo. Por ejemplo, al iniciar el muestreo:

```c
static void psoc_enter_sampling(uint8 debugMode)
{
    uint8 saved;
    uint16 target = capture_target_batches();
    ADC_StopConvert();
    timer_stop_quiet();
    saved = CyEnterCriticalSection();
    ...
    g_state = PSOC_SAMPLING;
    CyExitCriticalSection(saved);
    uart_send_diag(PSOC_EVT_SAMPLING_START, diag_u16_sat(target));  // <- nuevo
    ADC_StartConvert();
}
```

Para los resultados de calibración, se usa un esquema de dos bytes para transmitir el valor `int32` medido (saturado a `int16` antes de enviar, con HI byte primero):

```c
static void uart_send_cal_stage_meas(int32 value)
{
    uint16 u;
    if (value > 32767L)  { value =  32767L; }
    if (value < -32768L) { value = -32768L; }
    u = (uint16)(int16)value;
    uart_send_diag(PSOC_EVT_CAL_STAGE_MEAS, (uint8)(u >> 8));
    uart_send_diag(PSOC_EVT_CAL_STAGE_MEAS, (uint8)(u & 0xFFu));
}
```

Del lado del ESP esclavo, `psoc_uart.h` agrega el struct `PsocDiagEvent` y el typedef `DiagCallback`:

```cpp
struct PsocDiagEvent {
    uint8_t  event;
    uint8_t  value;
    uint8_t  psoc_state;
    uint32_t timestamp_ms;
};

typedef void (*DiagCallback)(const PsocDiagEvent &event);
```

Y en `psoc_uart.cpp` se implementa `_parseDiagEvent()`:

```cpp
void PsocUART::_parseDiagEvent()
{
    const uint8_t event    = _buf[2];
    const uint8_t value    = _buf[3];
    const uint8_t psocState = _buf[4];
    const uint8_t crc      = _buf[5];
    if (crc != (uint8_t)(PSOC_CTRL_DIAG_EVT ^ event ^ value ^ psocState)) {
        _lastOK = false;
        _badLen++;
        return;
    }
    _lastOK = true;
    _diagEventsRx++;
    if (_diagCb) {
        PsocDiagEvent ev = { event, value, psocState, millis() };
        _diagCb(ev);
    }
}
```

El buffer RX del UART del ESP también se amplía a 32768 bytes (`PSOC_UART_RX_BUFFER_SIZE`) para no perder frames de diagnóstico durante una calibración larga.

##### 5. Nuevo comando `0xB5` (PSOC_CMD_CALIBRATE)

El ESP puede ahora pedirle al PSoC que ejecute la calibración de arranque en cualquier momento enviando:

```
[0xAB][0xB5][0x01][0xB5 ^ 0x01]
```

En `psoc_uart.cpp`:
```cpp
void PsocUART::calibrate() { _sendCmd1(PSOC_CMD_CALIBRATE, 1); }
```

El PSoC, al recibir `0xB5`, llama a `psoc_calibration_run_blocking()`, que ejecuta todas las etapas secuencialmente (con el `isr_SyncIn` deshabilitado para no interrumpir el proceso), y luego restaura el camino de captura normal.

La nota en el header lo confirma:


La discrepancia entre el PID de `cal_sim.py` y la búsqueda binaria de `calibration.c` es una de las pistas de que el commit dice "medio hecho": el simulador exploró el espacio de diseño, y el código C implementó la opción más robusta, pero todavía queda trabajo de validar que los resultados de `cal_sim.py` mapeen correctamente a la búsqueda binaria real.

##### 8. Nuevos componentes PSoC en el TopDesign

El `TopDesign.cysch` modificado (binario) y los archivos generados (AMux_0/1/2/3, AMux_ADC, AMux_IN, VDAC_ref_PGA, VDAC_ref_BP, VDAC_Ref_Adder, VDAC_ref_LP, ADC_SAR, Opa_SAR) indican que el esquemático del PSoC fue modificado sustancialmente. Se agregaron:

- **AMux_IN**: multiplexor analógico para conmutar la entrada del ADC entre la señal del geófono (canal 0) y la referencia de tierra virtual de 2.5 V (canal 1) durante la calibración.
- **AMux_ADC**: multiplexor para conectar la salida de cada etapa de la cadena al ADC durante la calibración.
- **VDAC_ref_PGA, VDAC_ref_BP, VDAC_Ref_Adder, VDAC_ref_LP**: cuatro VDACs dedicados, uno por etapa de la cadena analógica GEO, que generan la tensión de referencia ajustable para nullear el offset de cada etapa.
- **ADC_SAR**: un ADC SAR (Successive Approximation Register) adicional, posiblemente para lecturas de diagnóstico más rápidas durante la calibración.

#### Impacto en el proyecto

Este commit es uno de los más densos del proyecto. No en términos de líneas de código funcional (la mayoría de las +131k líneas son artefactos de compilación PSoC regenerados), sino en términos de arquitectura. Se están sentando las bases para que el PSoC sea un nodo de medición que:

1. **Se auto-calibra al arranque**, sin intervención del usuario, ajustando el offset DC de cada etapa del front-end analógico.
2. **Reporta el estado de calibración** al ESP en tiempo real a través del protocolo de diagnóstico `0xC4`.
3. **Permite calibración bajo demanda** a través del comando `0xB5`, útil para recalibrar en campo si la temperatura cambia.
4. **Abstrae el hardware** de manera que el mismo `main.c` funciona para el nodo de geófono (GEO) y el nodo del martillo sismológico (HAMMER) sin modificaciones manuales.
