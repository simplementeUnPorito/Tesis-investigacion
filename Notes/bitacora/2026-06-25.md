## Contexto del día

El 24 de junio había cerrado con `HAMMER_LP` convergiendo a error cero y una reorganización completa de la parametrización de calibración en archivos por etapa. El 25 de junio continúa ese ajuste fino, pero encuentra un problema real al intentar correr la calibración con un objetivo de tensión más exigente para `HAMMER_LP`: al pedir 3750 mV (dentro de la banda operativa deseada de 3.5-4V), la etapa se detenía antes de llegar, quedando en 3.48V con `CAL_DONE ok=0`. El `TEST_LOG_2026-06-25.md` documenta la conclusión: el objetivo de 3750 mV era más estricto que lo que el filtro de corte muy bajo combinado con una ventana de lock de 512 muestras podía resolver de forma confiable, así que el objetivo se ajustó a 3500 mV con un feedforward ("adelanto") de 1280 mV.

Pero el trabajo de fondo del día no es ese ajuste puntual de constantes — es una revisión más profunda de cómo el controlador PI maneja el **signo** de la ganancia física entre el DAC de calibración y la señal medida. Hasta ahora, la ganancia se trataba como un número siempre positivo (o, en el caso especial de HAMMER_PGA con ganancia unitaria, cero). El 25 de junio reconoce que eso no es cierto en general: la etapa HAMMER_PGA tiene una ganancia que depende de la configuración del PGA y que puede ser tanto positiva como negativa según el punto de operación, mientras que HAMMER_LP tiene una ganancia fija que siempre es positiva porque la referencia VDAC entra por la entrada no inversora del amplificador MFB (Multiple Feedback, la topología clásica de filtro activo pasabajos).

#### Diff y análisis técnico

**Ganancia con signo explícito.** El cambio central es que `cal_pi_stage_gain_x1000()` deja de asumir que la ganancia de HAMMER_PGA es siempre positiva o cero:

```c
static int32 cal_pi_stage_gain_x1000(uint8 stage_index)
{
    if (stage_index == 0u) {
        int32 pga_gain = (int32)psoc_hw_pga_gain_x1000();
        return 1000L - pga_gain;
    }
    return g_cal_pi_cfg[stage_index].gain_x1000;
}
```

Antes la fórmula era `pga_gain - 1000L` con un clamp a cero si el resultado era negativo; ahora es `1000L - pga_gain`, sin clamp, permitiendo que el resultado sea negativo cuando la ganancia directa del PGA supera la unidad (`1000` en la escala ×1000). El comentario en el código lo explica: "PGA con ganancia directa 1 implica ganancia efectiva VDAC→medida 0 (1 - GainDirecta)" — la ganancia efectiva que el VDAC de calibración ve **desde el punto de vista del lazo de control** no es la ganancia del PGA en sí, sino la diferencia entre 1 y esa ganancia, porque el VDAC de calibración entra en un punto del circuito donde su efecto se resta (o se suma, según el signo) de la señal directa según la topología del PGA.

Como la función `cal_pi_gain_scaled_term()` ya dividía por la ganancia para escalar los términos P e I, ahora tiene que manejar explícitamente el caso de ganancia negativa:

```c
static int32 cal_pi_gain_scaled_term(int32 value, int32 num, int32 div, int32 gain_x1000)
{
    int32 sign = 1L;

    if (div == 0L) { div = 1L; }
    if (gain_x1000 < 0L) {
        sign = -1L;
        gain_x1000 = -gain_x1000;
    }
    if (gain_x1000 == 0L) { gain_x1000 = 1000L; }
    return sign * cal_round_div_i64((int64)value * (int64)num * 1000LL,
                                    (int64)div * (int64)gain_x1000);
}
```

Se separa el signo de la magnitud antes de dividir (para evitar división entera con truncamiento hacia el lado equivocado en negativos), y se reaplica el signo al resultado final.

**Anti-windup direccional.** El cambio más sofisticado del día es reemplazar un anti-windup simple ("si el esfuerzo satura contra cualquier riel, no integres") por uno que solo bloquea la integración cuando **acumular empeoraría** la saturación existente — exactamente el mecanismo "clamping" del modelo de Simulink que se había evaluado y descartado como "de mayor riesgo" el 23 de junio, ahora implementado con más cuidado:

```c
static int8 cal_pi_effort_delta_sign(int32 control_error, int8 direction, int32 gain_x1000)
{
    int8 sign;
    if (control_error == 0L || direction == 0 || gain_x1000 == 0L) {
        return 0;
    }
    sign = (control_error > 0L) ? 1 : -1;
    if (direction < 0) { sign = (int8)-sign; }
    if (gain_x1000 < 0L) { sign = (int8)-sign; }
    return sign;
}
```

y su uso al decidir si el DAC saturó contra el riel inferior o superior:

```c
if (effort < (int32)dac_lo) {
    dac_target = dac_lo;
    can_integrate = (effort_delta_sign > 0) ? 1u : 0u;
} else if (effort > (int32)dac_hi) {
    dac_target = dac_hi;
    can_integrate = (effort_delta_sign < 0) ? 1u : 0u;
} else {
    dac_target = (uint8)effort;
    can_integrate = 1u;
}
```

La lógica: si el esfuerzo calculado excede el riel inferior del DAC, pero la próxima corrección que el integrador *querría* aplicar apunta hacia **arriba** (`effort_delta_sign > 0`, alejándose del riel), sí se permite integrar — porque seguir acumulando en esa dirección eventualmente va a sacar al controlador de la saturación, no a empeorarla. Si en cambio la corrección deseada empujara *más* hacia el riel ya saturado, se bloquea la integración. Es la diferencia entre un anti-windup ingenuo (que a veces se queda "pegado" más tiempo del necesario, porque bloquea la integración incluso cuando esa integración ayudaría a salir del riel) y uno direccional, que solo interviene cuando efectivamente haría falta.

**Capacitor del multiplexor: verificación de existencia del canal.** Otro ajuste, motivado por la observación práctica documentada en el `TEST_LOG`: "un canal de capacitor solo se conecta si el hardware expone un canal extra más allá de los canales de señal; en el HAMMER actual hay dos canales de señal, así que LP no queda accidentalmente cortocircuitado como si fuera un capacitor". Esto se traduce en una nueva función `psoc_amux_disconnect_capacitor()` envuelta en una directiva condicional (`CAL_AMUX_HAS_CAP_CHANNEL`) que solo emite la llamada a `AMux_ADC_Disconnect()` si el `TopDesign` efectivamente tiene un canal adicional dedicado al capacitor — evitando que, en una variante de hardware donde todos los canales del multiplexor están ocupados por señales reales, el código intente desconectar por error un canal que en realidad es la etapa LP.

**Nuevos eventos de diagnóstico para depurar la ganancia y el deadband en vivo.** Se agregan varios códigos de evento UART nuevos (`PSOC_EVT_CAL_PI_GAIN32`, `PSOC_EVT_CAL_PI_DEADBAND`, `PSOC_EVT_CAL_PI_ERROR32`, `PSOC_EVT_CAL_PI_BUCKET32`, `PSOC_EVT_CAL_PI_STABLE`, `PSOC_EVT_CAL_AMUX_IN`, `PSOC_EVT_CAL_AMUX_CAP`) que exponen, por la telemetría periódica ya existente, la ganancia calculada de la etapa, el tamaño del deadband, el error en escala DAC, el bucket cuantizado, y el conteo de estabilidad — información que antes solo existía dentro de variables internas del firmware y que ahora se puede observar en tiempo real desde los logs del ESP mientras la calibración corre. Es una ampliación de observabilidad que tiene sentido inmediatamente después de haber introducido lógica de signo más compleja: cuando algo no converge como se espera, poder ver la ganancia con signo y el bucket de error en vivo ahorra tener que instrumentar el código de nuevo cada vez.

**Resultado en hardware.** El `TEST_LOG` documenta la corrida final exitosa con los objetivos ajustados:

```text
CAL_DONE,ok=1
HAMMER_PGA: ok=1, dac=65, raw_mV=996, target_mV=1024, err_mV=28
HAMMER_LP:  ok=1, dac=81, raw_mV=3589, target_mV=3500, err_mV=-89
```

Con `HAMMER_LP` ahora operando en la banda de 3.5-4V requerida por el diseño (antes se calibraba cerca de 1V, un punto de operación distinto), y `HAMMER_PGA` estable cerca de 1V con un margen de 28 mV.

#### Impacto en el proyecto

El manejo correcto de ganancias con signo y el anti-windup direccional no son features visibles desde la interfaz de usuario, pero son exactamente el tipo de generalización necesaria para que el mismo controlador PI, sin cambios de estructura, pueda calibrar tanto HAMMER como GEO — cuyas etapas (PGA, banda pasante, sumador, pasabajos) probablemente tengan combinaciones de ganancia y signo distintas entre sí. El título "Pre Geo autocal" es literal: este es el trabajo de robustecer el controlador antes de intentarlo por primera vez sobre la placa de geófonos.
