**Bug 1 — la envolvente reemplazaba la señal cruda en vez de superponerse.** El diagnóstico es exacto:

**Bug 2 — el bug más caro del día: `LOCK_N=4096` volvía frágil la calibración de GEO_BP, GEO_LP y HAMMER_LP.** El handoff calcula el deadband físico por etapa con la fórmula `ceil(gain_x1000 × 6 / 5000)`:

| Etapa | Deadband (códigos DAC) |
|-------|------------------------|
| GEO_BP | `ceil(1000×6/5000) = 2` ← muy ajustado |
| GEO_ADDER | `ceil(3000×6/5000) = 4` |
| GEO_LP | `ceil(6000×6/5000) = 8` |

Con un deadband de apenas 2 códigos DAC para GEO_BP, el PI necesitaba **4096 muestras consecutivas** (1.37 segundos ininterrumpidos a 3 kHz) dentro de esa ventana angosta para declarar "lock". Cualquier pico de ruido — algo inevitable en una medición analógica real durante más de un segundo — reiniciaba el contador de estabilidad a cero, y el lazo terminaba agotando el timeout sin cerrar nunca la etapa como exitosa. El fix es directo:

| Parámetro | Antes | Ahora |
|-----------|-------|-------|
| `CAL_PI_LOCK_SAMPLES_GEO_BP` / `_GEO_LP` / `_HAMMER_LP` | `4096` | `1024` |
| `CAL_PI_TIMEOUT_SAMPLES_GEO_BP` / `_GEO_LP` / `_HAMMER_LP` | `30000` (default) | `45000` |

```c

-#define CAL_PI_LOCK_SAMPLES_GEO_BP 4096u
+#define CAL_PI_LOCK_SAMPLES_GEO_BP 1024u
 #endif
 ...
 #ifndef CAL_PI_TIMEOUT_SAMPLES_GEO_BP
-#define CAL_PI_TIMEOUT_SAMPLES_GEO_BP CAL_PI_DEFAULT_TIMEOUT_SAMPLES
+#define CAL_PI_TIMEOUT_SAMPLES_GEO_BP 45000u

```

Reducir el requisito a 1024 muestras (~341 ms de estabilidad continua) es un compromiso razonable: suficiente para descartar que el "lock" sea casualidad, pero no tan exigente como para que el ruido normal de la señal lo haga prácticamente imposible de alcanzar. El timeout global sube en compensación (a 45000 muestras, ~15 s por etapa) para dar más margen antes de rendirse, ya que el lazo ahora puede necesitar más reintentos de estabilización con un lock más corto. El propio handoff aclara el motivo por el que el mismo bug pasó desapercibido en HAMMER hasta esta sesión: "botón funciona, web no" — cuando el usuario dispara la calibración con el botón físico, generalmente lo hace en condiciones ya estables (después de que la auto-calibración terminó); cuando lo hace desde la web, puede coincidir con la ventana de auto-calibración post-boot (`PSOC_AUTO_CAL_ON_READY`, 500 ms) todavía en curso, lo cual agrava el problema.

  if (!src || src.length <= trim) return null;
  const start = Math.max(0, trim | 0);
  const n = src.length - start;
  if (n < 2) return null;
  let mean = 0;
  for (let i = start; i < src.length; i++) mean += src[i];
  mean /= n;
  const centered = new Float64Array(n);
  for (let i = 0; i < n; i++) centered[i] = src[start + i] - mean;
  const values = hilbertEnvelope(centered);

  for (let i = 0; i < values.length; i++) if (values[i] > peak) peak = values[i];

}

Este es un bug de diseño más que de implementación: la interfaz había sido escrita asumiendo implícitamente que los nodos esclavos se numeran de forma contigua desde 1 (`S1`, `S2`, `S3`...), una asunción razonable mientras solo se probaba con un único esclavo numerado `1`. **4. Ocultamiento robusto del control "Offset m" para HAMMER — cuando el CSS gana la pelea contra el JS.**

La solución es defensiva: aplicar explícitamente `display: none` además de `hidden=true`, sin confiar en que el atributo semántico sea suficiente frente al CSS existente. **6. Validación en hardware físico real — y un hallazgo importante sobre la calibración de S2.**

La sección de "Live verification" del `WEB_FIELD_TESTS.md` documenta una prueba completa contra el hardware físico: comandos WebSocket directos (`STATUS`, `ARM`, `START`) contra el geófono S2 real, confirmando `8790` muestras recibidas a `fsExact=2929` Hz con una latencia de `4653 μs`. Esta conclusión es importante porque descarta una hipótesis más cómoda (que el punto rojo de calibración fallida en la web fuera simplemente un timeout de interfaz de 8 segundos, sin relación con el estado real del hardware) a favor de una más incómoda pero más honesta: la calibración de S2 falla de verdad, en una etapa real del lazo PI. La recomendación que se deja para el próximo turno de trabajo es concreta y bien dirigida: agregar telemetría de resultado por etapa (DAC medido, tensión medida, target, error) para poder ver exactamente cuál de las cuatro etapas de GEO está fallando y por qué, en lugar de tener que inferirlo indirectamente de los ACKs de progreso.

**7. El LED que "titila" según el protocolo pero no según los ojos del usuario — un bug que sigue abierto.**

> "La web/log mostraba `S2 LED titilando`, pero eso solo significaba ACK. El usuario no ve el LED físico parpadear. ... Cambio aplicado: el log web dirá `LED blink ACK`; el slave usa patrón más largo (`BLINK_TIMES=20`, `BLINK_INTERVAL_MS=250`) y explicita `BLINK_LED_PIN=2`, `BLINK_LED_ACTIVE_LOW=0`."

Este es un capítulo adicional del mismo bug de LED que el primer commit del día ya había "arreglado" (la polaridad activo-alto vs activo-bajo). Pero incluso después de ese fix, el usuario seguía reportando que no veía el LED titilar en la placa S2 concreta — lo cual sugiere que, o bien esa placa tiene el LED de identificación en un GPIO físico distinto al asumido (`BLINK_LED_PIN=2`), o bien el patrón de parpadeo era demasiado breve para notarlo a simple vista. La respuesta es doble: primero, dejar de sobre-prometer en el log ("LED titilando" pasa a ser honestamente "LED blink ACK" — se confirma que el comando llegó y fue aceptado, no que el LED efectivamente se vio parpadear); segundo, alargar el patrón de parpadeo (`BLINK_TIMES` de 10 a 20 destellos, intervalo de 150 a 250 ms) para hacerlo más fácil de percibir mientras se sigue investigando si el pin físico es el correcto. Es honesto reconocer, en la documentación, que este bug queda parcialmente sin resolver al final del día — el propio handoff lo deja como ítem pendiente explícito: "Si sigue sin verse tras flashear S2, hay que identificar el GPIO real del LED de esa placa y cambiar `BLINK_LED_PIN`."

**8. Diez pedidos nuevos del usuario, documentados para la siguiente sesión.**

El `CODEX_HANDOFF_MASW_WEB.md` cierra con una lista de diez puntos pedidos por el usuario que quedan explícitamente fuera del alcance de esta sesión, entre ellos: soportar esclavos no contiguos de cualquier ID (`S5` sin que existan `S1`-`S4`), mover el control "Offset Y (mV)" del panel de esclavo al panel de preservación (para desplazar visualmente capturas guardadas, no la captura en vivo), aplicar la política de "sin offset DC" de forma consistente también a las capturas preservadas, calcular la envolvente siempre sobre la señal sin DC (independiente de los offsets visuales aplicados), y hacer que el toggle de "Espectro" reutilice el gráfico existente en vez de generar uno nuevo cada vez que se acciona. Es una lista de trabajo pendiente clara y priorizada, exactamente el tipo de documento que permite retomar un proyecto complejo de front-end sin tener que re-descubrir el estado desde cero.

#### Impacto en el proyecto

Este commit da el salto de "el sistema calibra y transmite datos" a "el sistema empieza a entender qué tipo de hardware tiene conectado y a adaptarse solo" — la detección automática de GEO vs HAMMER es un paso de madurez operativa importante para un sistema pensado para desplegarse en terreno con múltiples nodos de tipos distintos. El simulador de martillo por HTTP es una herramienta de testing inteligente que desacopla el desarrollo de la interfaz de la disponibilidad física de todas las variantes de hardware. Y la validación honesta contra hardware real —que confirma que la calibración de S2 falla de verdad y que el LED sigue sin verse titilar pese al primer fix— evita que el proyecto avance sobre supuestos de "ya está arreglado" que no se sostienen frente a la prueba física.
