## El firmware unificado — la decisión de diseño central

El archivo `src/psoc/Martillo_ESP.cydsn/main.c` existe en el repositorio y contiene una de las decisiones de diseño más interesantes del proyecto: **un único firmware para dos nodos de hardware distinto**.

### Detección automática de hardware en compilación

El encabezado del `main.c` lo declara explícitamente:

```c
/*******************************************************************************
* main.c — Firmware PSoC unificado para geófono (GEO) y martillo (HAMMER).
*
* El hardware del TopDesign determina automáticamente el modo de compilación:
*   Si el proyecto tiene PGAgain  → CLASS = GEO    (geófono diferencial)
*   Si el proyecto tiene PGA      → CLASS = HAMMER  (martillo piezoeléctrico)
* No hay que cambiar nada en este archivo al cambiar de proyecto.
*******************************************************************************/
```

El mecanismo es limpio: PSoC Creator genera un `project.h` que define macros según los componentes instanciados en el TopDesign. Si el TopDesign tiene un componente llamado `PGAgain`, el compilador ve `PGAgain_DEFAULT_GAIN`. Si tiene `PGA`, ve `PGA_DEFAULT_GAIN`. El `main.c` usa ese hecho para fijar el `CLASS` en tiempo de compilación:

```c
#define GEO    0
#define HAMMER 1

#if defined(PGAgain_DEFAULT_GAIN)
  #define CLASS GEO
#elif defined(PGA_DEFAULT_GAIN)
  #define CLASS HAMMER
#else
  #error "Hardware no reconocido: el TopDesign debe tener PGAgain (GEO) o PGA (HAMMER)."
#endif
```

Si el TopDesign no contiene ninguno de los dos componentes esperados, el código emite un `#error` en compilación — el programador no puede flashear un firmware inválido sin recibir el aviso.

Esta técnica es conocida en sistemas embebidos como **hardware abstraction via preprocessor** — en lugar de un HAL con funciones virtuales, se usa el preprocesador para seleccionar en tiempo de compilación qué rutas de código se incluyen. El resultado es cero overhead en runtime y cero ambigüedad de configuración: el firmware sabe qué es porque el hardware le dice qué es.

### La cadena analógica del geófono vs la del martillo

El `main.c` unificado inicializa las cadenas analógicas completamente distintas en el `main()`, usando `#if CLASS`:

#### Cadena GEO (geófono diferencial SM-24):

```c
#if CLASS == GEO
    /* ── Cadena analógica del geófono diferencial ── */
    OPAref_Start();
    PGAp_Start();  PGAn_Start();
    PGAp_SetGain(PGAp_GAIN_02);
    PGAn_SetGain(PGAp_GAIN_02);
    LPF_1_Start(); LPF_2_Start();
    PGAgain_Start();
    PGAgain_SetGain(g_pga_code);
    OPAbp_Start();
    OPAadder_Start();
    PGAvdac_Start();
    PGAvdac_SetGain(g_pgavdac_code);
    OPAlp_Start();
```

El SM-24 genera una señal diferencial (+/−) con amplitud de milivoltios. La cadena del geófono:
1. `OPAref` — genera el nivel de referencia virtual (Vdd/2)
2. `PGAp` / `PGAn` con ganancia 2 — primera etapa de amplificación de la señal diferencial
3. `LPF_1` / `LPF_2` — filtros pasa-bajos para eliminar alias y ruido de alta frecuencia
4. `PGAgain` — segunda etapa de ganancia configurable por UART (0–8)
5. `OPAbp` / `OPAadder` — conversión diferencial a single-ended (resta positivo menos negativo)
6. `PGAvdac` — ajuste fino de offset vía VDAC + PGA
7. `OPAlp` — filtro de salida antes del ADC_DelSig

Es la cadena que tardó meses en converger (mayo completo, con pivotes Sallen-Key → MFB, compensadores analógicos, etc.).

#### Cadena HAMMER (martillo piezoeléctrico 086D20):

```c
#else  /* HAMMER */
    /* ── Cadena analógica del martillo piezoeléctrico ── */
    LPF_ref_Start();
    LPF_ADC_Start();
    Opa_ref_Start();
    PGA_ref_Start();
    PGA_ref_SetGain(g_pgavdac_code);
    PGA_Start();
    PGA_SetGain(g_pga_code);
#endif
```

El PCB Piezotronics 086D20 genera una señal de alta impedancia de salida de baja amplitud (orden de mV a cientos de mV), single-ended. La cadena del martillo es más simple:
1. `LPF_ref` / `LPF_ADC` — filtros anti-aliasing para la referencia y para la señal del ADC
2. `Opa_ref` — buffer de referencia
3. `PGA_ref` — ganancia para la referencia (configurable via `0xA9`)
4. `PGA` — ganancia de la señal del sensor (configurable via `0xA6`)

La diferencia clave: el geófono usa un ADC_DelSig (Delta-Sigma, alta resolución, hasta 20 bits, velocidad baja ~1020 SPS). El martillo usa un ADC SAR (Successive Approximation Register, menor resolución pero mucho más rápido — cientos de kSPS). En los `codegentemp` del proyecto se ve:

```
codegentemp/ADC_INT.c       → ISR del ADC Delta-Sigma (GEO)
codegentemp/ADC_AMux.c      → Multiplexor analógico del ADC (HAMMER SAR)
codegentemp/LPF_ADC.h       → Filtro antes del SAR
codegentemp/isr_SyncIn.h    → Mismo para ambos (SYNC_IN es común)
```

### El protocolo — idéntico para ambos nodos

El protocolo UART PSoC↔ESP es el mismo para GEO y HAMMER. Los frames, comandos, y la máquina de estados son compartidos:

```c
/* TX UART — frame de datos RAW (95 bytes):
 *   [0xAB][n=30][seq_lo][seq_hi] + 30×3 bytes (raw LE: b0,b1,b2) + [crc XOR]
 *
 * RX UART — comandos con checksum XOR:
 *   1 param : [0xAB][cmd][param][cmd^param]
 *       0xA5 status/probe   0xA6 set PGAgain (0-8)   0xA9 set PGAvdac (0-8)
 *       0xAA set VDAC       0xB1 arm (espera SYNC)   0xB3 debug rampa (0/1)
 *       0xB4 start-now      0xC1 pong
 *   2 param : [0xAB][0xA3][n_lo][n_hi][0xA3^n_lo^n_hi]   (set N lotes)
 */
```

La máquina de estados también es compartida:

```c
#define PSOC_IDLE          0u
#define PSOC_ARMED         1u
#define PSOC_SAMPLING      2u
```

Cuando el ESP envía `0xB1` (ARM), el PSoC entra en `PSOC_ARMED` y espera el flanco en `SYNC_IN`. La ISR `isr_SyncIn` es idéntica para GEO y HAMMER:

```c
CY_ISR(isr_SyncIn)
{
    uint8 saved;
    if (SYNC_IN_Read())
    {
        if (g_state == PSOC_ARMED)
        {
            psoc_enter_sampling(g_debug_psoc);
        }
    }
    else
    {
        ADC_StopConvert();
        timer_start_runtime();
        saved = CyEnterCriticalSection();
        /* ... reset de buffers ... */
        g_state = PSOC_IDLE;
        CyExitCriticalSection(saved);
    }
}
```

Esto significa que el maestro ESP32 trata al nodo del martillo exactamente igual que a un nodo de geófono: le envía ARM, sube el flanco SYNC, y recibe frames de datos. El ESP32 del martillo ni siquiera necesita saber que su PSoC mide fuerza en vez de velocidad de partícula — el protocolo es ciego al tipo de sensor.

### Store-and-forward compartido

El mecanismo de captura en RAM sin RF durante el muestreo también está en el firmware unificado:

```c
static volatile uint8  g_capture_raw[PSOC_CAPTURE_MAX_BATCHES][CAPTURE_BATCH_BYTES];
static volatile uint16 g_batch_fill       = 0u;
static volatile uint8  g_batch_ready      = 0u;
static          uint16 g_seq              = 0u;
static volatile uint16 g_batches_captured = 0u;
static volatile uint8  g_capture_done     = 0u;
```

512 batches × 30 muestras × 3 bytes = 46 080 bytes en el PSoC. El PSoC 5LP tiene 64 KB de SRAM — esto usa el 72% en el peor caso. Durante el muestreo (`PSOC_SAMPLING`), el main loop hace silencio total:

```c
static void service_runtime(void)
{
    if (g_state == PSOC_SAMPLING) {
        if (g_capture_done == 0u) {
            return;   /* Silencio total: sin UART RX/TX, sin LED, sin pings. */
        }
        /* ... continúa solo cuando g_capture_done = 1 ... */
    }
```

El comentario "Silencio total" es el corazón del fix del beacon que tomó meses entender. El ADC muestrea; el firmware no hace nada más hasta terminar. Ningún ruido generado por el UART.

### El nombre `Martillo_ESP` — convención de nomenclatura del proyecto

Los nombres de proyectos PSoC en este repo siguen una convención implícita:
- `DiferencialToSingleEnded_ESP` → circuito que convierte diferencial a SE para conectar a ESP (primer nombre del geófono)
- `AcondicionamientoAnalogico` → nombre semántico, propósito del circuito (nombre actual del geófono)
- `AcondicionamientoMartillo` → mismo patrón, primer intento del martillo (abandonado)
- `Martillo_ESP` → `Martillo` + `_ESP` — el sufijo `_ESP` es el patrón del primer nombre (interfaz directa con ESP)

El nombre `Martillo_ESP` dice que este PSoC es el que va conectado directamente al ESP32 del nodo del martillo. No es solo acondicionamiento: es el nodo completo. La arquitectura de hardware es:

```
PCB Piezotronics 086D20 (sensor de fuerza)
         ↓ cable BNC/LEMO
Martillo_ESP.cydsn PSoC 5LP
         ↓ UART 115200 + pin SYNC_IN
ESP32 esclavo "nodo martillo"
         ↓ ESP-NOW
ESP32 maestro
         ↓ USB Serial
MATLAB
```

Es la misma arquitectura en capas que los nodos geófonos — el ESP32 maestro ni siquiera necesita saber qué tipo de sensor tiene conectado cada esclavo.

### Cualitativo

El 4 de junio es, como commit, trivial. Como dato arquitectónico, es una confirmación de que el proyecto tomó la decisión correcta de no bifurcar el firmware.

La decisión de tener un `main.c` unificado para GEO y HAMMER es non-trivial. La alternativa obvia es tener dos archivos `main.c` separados — uno para el geófono, uno para el martillo — y mantenerlos en sincronía manual. Pero esa alternativa tiene un costo invisible: cada vez que se arregla un bug en la máquina de estados del GEO (ej: el watchdog RX, la lógica del LED en PSOC_SAMPLING), hay que recordar aplicar el fix también al HAMMER. El firmware unificado hace que esa sincronía sea automática: no existe "el GEO se desincronizó del HAMMER" porque son el mismo archivo.

El mecanismo de detección por `#if defined(PGAgain_DEFAULT_GAIN)` es elegante porque usa algo que ya existe — las macros generadas por PSoC Creator — sin introducir ninguna dependencia nueva. No hay un archivo de configuración separado, no hay un flag en platformio.ini, no hay una constante que alguien puede olvidar cambiar. El TopDesign es la fuente de verdad: si el hardware tiene un `PGAgain`, es un geófono. Si tiene un `PGA`, es un martillo.

El mensaje "Ayuda" a las 00:10 después del "Locura" a las 23:52 describe con precisión el ritmo del trabajo en hardware embebido: nunca hay tiempo para celebrar porque siempre hay una siguiente cosa que no funciona todavía. El "Locura" del geófono duró diez minutos como logro exclusivo. Luego, martillo.
