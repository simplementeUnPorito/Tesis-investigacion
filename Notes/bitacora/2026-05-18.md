#### Cuatro estrategias de búsqueda — el arsenal

##### 1. `calculoCompensadorSimple.py` (829 líneas) — Grilla con modelo real

La versión original hacía una búsqueda exhaustiva de grilla pero con modelo simplificado del op-amp. La reescritura incorpora el modelo real completo: ganancia de lazo abierto A0, polo dominante fp, y resistencia de entrada Rin. El op-amp deja de ser ideal y se convierte en un modelo de primer orden con ganancia finita:

```python
# ── Op-amp del BP ────────────────────────────────────────────────────────────
A0_dB_bp    = 90                       # dB — ganancia DC en lazo abierto
A0_bp       = 10**(A0_dB_bp / 20.0)   # lineal ≈ 31623
f_p_bp      = 8e6                      # Hz — polo dominante
omega_p_bp  = 2*np.pi * f_p_bp
Rin_bp_amp  = 35e6                     # Ω — resistencia de entrada al nodo inversor
```

Estos valores (A0=90dB, fp=8MHz, Rin=35MΩ) corresponden a un op-amp de precisión real — probablemente el que está en el diseño actual del filtro MFB. La función de transferencia del bandpass con op-amp real es:

```python
def H_bp_real(R1, R2, C1, C2, freqs):
    s      = 1j*2*np.pi*freqs
    beta   = (s*R2*C1)/((s*R1*C1+1)*(s*R2*C2+1))
    z_load = (R2/Rin_bp_amp)/(s*R2*C2+1)
    nif    = (1+s/omega_p_bp)/A0_bp
    return -beta/(1+nif*(1+beta+z_load))
```

El término `nif = (1+s/omega_p_bp)/A0_bp` es el factor de no-idealidad del op-amp: cuando A0→∞ y fp→∞, NIF→0 y recuperas la respuesta ideal. Con los valores reales, NIF produce desviaciones en frecuencia que el script cuantifica y usa como criterio de filtrado.

La búsqueda cover the space:
- R1 ∈ [1kΩ, 1MΩ] (comerciales E96/E24)
- C1, C2 ∈ valores comerciales (cerámicos + electrolíticos)
- R2 determinado analíticamente de la condición de f0 exacta
- Adder (sumador) resuelto por mínimos cuadrados (lstsq)

TOP_N = 15 candidatos finales.

##### 2. `calculoCompensadorFull.py` (606 líneas) — GPU + multicore CPU exhaustivo

La búsqueda "Full" no usa el subset TOP_N: guarda *todos* los candidatos que pasan el filtro. El criterio de filtrado es más estricto:

```python
# ── Filtro de banda de paso ───────────────────────────────────────────────────
```

Para acelerar la evaluación (el espacio es mucho mayor que en Simple), el script detecta automáticamente hardware GPU:

El fallback a CPU usa `ProcessPoolExecutor` con todos los núcleos disponibles. Los candidatos se procesan en chunks:

```python

```

La diferencia fundamental con Simple: Full usa una grilla de 1200 frecuencias (vs 800 en Simple) y evalúa la respuesta en banda de paso `[ω₀(ζ−√(ζ²−1)), ω₀(ζ+√(ζ²−1))]` — la banda real del bandpass compensado, no un rango fijo.

##### 4. `calculoCompensadorOptimo.py` (666 líneas) — Optimización global continua

En lugar de grilla exhaustiva (que es O(n³) en R1, C1, C2), este script usa `scipy.optimize.differential_evolution` — un algoritmo evolutivo que explora el espacio continuo:

```python
from scipy.optimize import differential_evolution, minimize

# Espacio continuo 3-D: [log10(R1), log10(C1), log10(C2)]
#   → R2 determinado por condición ω0 exacta
#   → Ratios adder optimizados analíticamente (lstsq)
#   → Snap a comerciales + T-redes al final
```

El proceso en tres fases:
1. **Fase 1 (DE global):** `differential_evolution` en el espacio log-continuo, todos los núcleos en paralelo
2. **Fase 2 (bandas de ζ):** Para cada rango de ζ objetivo, un proceso independiente via `ProcessPoolExecutor`
3. **Fase 3 (snap + adder):** Mapeo a componentes comerciales más cercanos + `ThreadPoolExecutor` para el lstsq del sumador

Esta estrategia es fundamentalmente distinta: no intenta evaluar todos los puntos posibles, sino encontrar el mínimo global de una función de error continua y *luego* cuantizar al espacio comercial. Es más elegante pero puede perder candidatos que están en regiones no convexas del espacio.

#### El "diablo digital" — qué se estaba resistiendo

El PSoC 5LP tiene un DFB (Digital Filter Block) — hardware dedicado para filtros digitales, implementando el notch FIR de 464 taps en hardware sin usar CPU. Si se quisiera, se podría implementar toda la compensación del geófono **digitalmente** en el DFB:

1. Medir con ADC_DelSig a 1500 SPS
2. Aplicar filtro FIR de compensación en DFB (que ya tiene los 464 taps del notch; agregar otra sección es posible)
3. Transmitir via UART

El problema: la tesis requiere un sistema de **acondicionamiento analógico**. El geófono SM-24 tiene que ser llevado a condiciones operativas (frecuencia natural de 4.5Hz → extender a 0.5-1Hz con compensación Pazos) mediante electrónica analógica, no software. El circuito MFB + compensador en el PCB es la contribución de hardware del proyecto.

Pero después de días buscando el conjunto de componentes que satisfaga todas las restricciones simultáneamente — y con cuatro scripts distintos que dan resultados contradictorios — la tentación de simplemente programar el DFB es real.

#### Por qué cuatro scripts y no uno

Cada script tiene un nicho:

| Script | Estrategia | Ventaja | Límite |
|--------|-----------|---------|--------|
| Simple | Grilla exhaustiva | Reproducible, comprensible | Lento en espacios grandes |
| Full | Grilla + GPU paralela | Exhaustivo, guarda todo | Requiere VRAM |
| GPU | PyTorch vectorizado | Muy rápido con CUDA | Necesita GPU |
| Optimo | Differential Evolution | Escala bien a más variables | Puede quedar en mínimos locales |

La estrategia de tener cuatro en paralelo es defensiva: si uno no converge, otro puede dar el candidato correcto. El CSV resultante mezcla los mejores de cada uno.
