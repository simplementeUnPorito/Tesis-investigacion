## Contexto del día

El 18 de mayo terminó con cuatro scripts de Python buscando componentes para el compensador analógico. Los mejores candidatos del CSV son matemáticamente viables, pero siguen siendo simulación. El 20 de mayo es el día en que la simulación se convierte en firmware que corre en el PSoC real.

La decisión arquitectónica es radical: en lugar de intentar implementar el compensador con resistencias y capacitores externos, se usa el **PSoC 5LP como el soporte de la cadena analógica**. El chip tiene cuatro op-amps (OA0–OA3), dos PGAs diferenciales, un VDAC, y un SAR ADC — todos estos bloques analógicos pueden conectarse internamente via el Universal Analog Interconnect (UAB) sin salir al PCB. El nuevo proyecto se llama `DiferencialToSingleEnded` y eso describe exactamente lo que hace: convierte la señal diferencial del geófono SM-24 a single-ended mediante una cadena de procesamiento analógico interno.

Son tres commits en el mismo día, separados por horas: el primero a la 1 AM (trabajando hasta tarde del 19), el segundo 1 hora después, y el tercero a las 9 PM del mismo 20.

#### El proyecto `DiferencialToSingleEnded` — arquitectura de la cadena

El firmware nuevo en `main.c` revela la cadena analógica completa. El orden de inicialización en `main()` no es arbitrario — refleja la topología del circuito:

```c
UART_PC_Start();
OPAref_Start();           // 1. Referencia de tensión (Vref para VDAC y PGAs)

PGAp_Start();             // 2a. PGA diferencial positivo
PGAn_Start();             // 2b. PGA diferencial negativo
PGAp_SetGain(PGAp_GAIN_02);   // ganancia 2×
PGAn_SetGain(PGAp_GAIN_02);   // ganancia 2× (balanceado)

LPF_1_Start();            // 3a. Filtro paso bajo analógico #1
LPF_2_Start();            // 3b. Filtro paso bajo analógico #2

PGAgain_Start();          // 4. PGA de ganancia ajustable (1×–32×)
PGAgain_SetGain(g_pga_code);

OPAbp_Start();            // 5. Op-amp bandpass (compensador)
OPAadder_Start();         // 6. Op-amp sumador (BP + señal directa)
PGAvdac_Start();          // 7. PGA del VDAC de offset
PGAvdac_SetGain(g_pgavdac_code);
OPAlp_Start();            // 8. Op-amp lowpass (filtro anti-alias final)

VDAC_Start();             // 9. VDAC offset calibration
VDAC_SetValue(VDAC_INIT_VALUE); // 0x94 = 148 decimal

Filter_Start();           // 10. DFB — notch FIR 50Hz
Filter_SetCoherency(Filter_CHANNEL_A, Filter_KEY_LOW);
Filter_SetDalign(Filter_STAGEA_DALIGN, 0u);
Filter_SetDalign(Filter_HOLDA_DALIGN,  0u);

ADC_Start();              // 11. SAR ADC (ya no es ADC_DelSig)
ADC_StartConvert();
isr_DelSig_StartEx(isr_DelSigReady);
isr_Filter_StartEx(isr_FilterReady);
```

Esta cadena implementa en hardware analógico lo que los scripts Python estaban buscando:

```
Geófono SM-24 (diferencial)
    │
    ▼
[PGAp / PGAn] — 2× (convertir diferencial a single-ended interno)
    │
    ▼
[LPF_1 + LPF_2] — Filtros analógicos anti-alias de primera etapa
    │
    ▼
[PGAgain] — Ganancia ajustable (comando 0xA6)
    │
    ├──────────────────────────────┐
    │                              │ (señal directa)
    ▼                              │
[OPAbp] — Bandpass compensador    │
    │                              │
    └──────────────┐               │
                   ▼               │
              [OPAadder] ←─────────┘
                   │ (BP path + direct path, compensación Pazos)
                   │
                   ├── [PGAvdac + VDAC] → offset calibration
                   │
                   ▼
              [OPAlp] — Lowpass final (anti-alias para ADC)
                   │
                   ▼
              [ADC SAR] ← ya no Delta-Sigma, es un SAR ADC
                   │
                   ▼
              [DFB Filter] — Notch FIR 50Hz (digital)
                   │
                   ▼
              UART → PC (protocolo 5-byte [0x56])
```

Esta es la implementación completa del sistema de acondicionamiento descrito en el marco teórico de la tesis. El compensador Pazos está físicamente implementado con `OPAbp` y `OPAadder` usando los bloques analógicos internos del PSoC.

#### El cambio de ADC: de `ADC_DelSig` a `ADC`

Un cambio crítico que no se ve en el nombre del commit: el proyecto usa `ADC` en lugar de `ADC_DelSig`. El ADC Delta-Sigma tiene latencia de pipeline y ruido de cuantización muy bajo (18-bit effective), pero tiene restricciones en la velocidad de muestreo. El SAR ADC es más rápido pero con menor resolución.

En el contexto de esta arquitectura, el SAR ADC tiene sentido: la cadena analógica ya filtró la señal (OPAlp), y el ADC solo necesita convertir una señal ya acondicionada. La resolución efectiva del sistema no depende solo del ADC sino de toda la cadena.

#### `Analog_LPF_v1_0.cylib` — componente personalizado de terceros

Las librerías `Analog_LPF` y `CONSULTRONLib` son componentes PSoC Creator de terceros que implementan el `LPF` analógico como un bloque PSoC reutilizable. El header template muestra la API:

```c
void `$INSTANCE_NAME`_Start(void);
void `$INSTANCE_NAME`_Stop(void);
void `$INSTANCE_NAME`_SetCutoffFreq(uint8 freq);

#define `$INSTANCE_NAME`_DEFAULT_CUTOFF  `$Cutoff_Frequency` << 4
```

El `$INSTANCE_NAME` es la macro de PSoC Creator para parametrizar el nombre del bloque en el esquemático. Cuando se agregan `LPF_1` y `LPF_2` en TopDesign.cysch, PSoC Creator genera automáticamente `LPF_1_Start()`, `LPF_1_SetCutoffFreq()`, etc.

La CONSULTRONLib tiene una versión más avanzada con PDF de documentación (717KB) y un proyecto demo `LPF_Demo.cydsn` que muestra cómo usar el componente. El `LPF_Demo.cydsn/main.c` tiene 73 líneas — un punto de partida para entender cómo se configura.

#### Protocolo actualizado — checksum XOR en recepción

El nuevo `main.c` introduce **checksum XOR en el protocolo de recepción**:

```c
/*
 * Protocolo RX — 4 bytes con checksum XOR: [0xAB][cmd][param][cmd^param]
 *   0xAB 0xA5 0x00 0xA5  -> send_config()
 *   0xAB 0xA1 <en>   <cs>-> stream_enabled
 *   0xAB 0xA6 <code> <cs>-> PGAgain_SetGain(code)
 *   0xAB 0xA8 <mode> <cs>-> tx_mode (0=crudo, 1=flt)
 *   0xAB 0xA9 <code> <cs>-> PGAvdac_SetGain(code)
 *   0xAB 0xAA <v>    <cs>-> VDAC_SetValue(v)
 */
```

El checksum `cmd ^ param` es simple pero efectivo para descartar bytes corruptos. El guard byte cambió de `0x56` a `0xAB` para el canal de recepción (el `0x56` sigue siendo el guard del TX), lo que permite distinguir dirección del tráfico solo por el primer byte.

El watchdog de RX es una adición de robustez:

```c
static void rx_watchdog_start(void)
{
    Timer_Stop();
    Timer_ReadStatusRegister();
    isr_Timer_ClearPending();  // evita falsa disparo post-reset
    Timer_WritePeriod(TIMEOUT_COUNTS);  // 10 ms @ 24 MHz
    watchdog_rx = 0u;
    Timer_Start();
}
```

Si un comando queda a medias (por ruido en UART, cable desconectado, etc.), el watchdog expira en 10ms y resetea `rx_state` a 0. Sin esto, el firmware podría quedarse esperando el tercer byte de un comando indefinidamente.

#### Modo DEBUG para validación sin hardware

```c
#define DEBUG 0   // 0=real, 1=rampa creciente
```

Con `DEBUG 1`, el firmware genera una rampa en lugar de muestras del ADC:

```c
#if DEBUG
    pkt[0]=0x56u; pkt[1]=0x00u;
    pkt[2]=(uint8)((dbg_cnt>>16u)&0xFFu);
    pkt[3]=(uint8)((dbg_cnt>>8u )&0xFFu);
```

La rampa incrementa en +1 por muestra y rollover a 0 al llegar a 2^24. Esto permite verificar que MATLAB recibe los datos correctamente antes de conectar el hardware real.

#### `analisis_modelo.m` — el modelo físico completo del experimento

Este es uno de los scripts más importantes del proyecto — implementa el modelo forward completo para el experimento de MASW con martillo de impacto:

```matlab
% MODELO FORWARD:
%   Y(f) = H_sys(f) × H_tip(f) × H_suelo(f,r) × F_0
%
%   Donde:
%     H_sys(f)     = H_circuito(f) × H_geofono(f)  ← lo que queremos estimar
%     H_tip(f)     = TF de la punta (ventana Hann de duración T_c = 0.9/fc)
%     H_suelo(f,r) = dinámica suelo+interfaz: propagación geométrica (1/√r)
%                    + atenuación anelástica. DESCONOCIDA pero relativamente
%                    constante si la posición es fija.
%     F_0          = amplitud de fuerza del golpe (desconocida, constante)
%
%   H_sys_est(f) = Y_stack(f) / H_tip(f) / G_dist(r)
```

El script implementa el flujo completo de procesamiento de señal:

1. **Detección de impactos** (`THRESH_SIGMA = 2.5`): umbral automático de 2.5σ sobre el ruido de fondo para detectar cada golpe del martillo
2. **Alineación por correlación cruzada**: los hits se alinean usando `xcorr` de la envolvente Hilbert — si el offset supera el 20% de la ventana, el hit es descartado
3. **Stack coherente**: promedio de N impactos alineados → SNR mejora ≈ √N
4. **Deconvolución de H_tip**: la TF de la punta del martillo se modela como una ventana Hann de duración T_c = 0.9/fc, donde fc depende del color de punta:

```matlab
PUNTA_FC = containers.Map( ...
    {'gris','marron','roja','negra'}, ...
    {170,    250,    650,   1600});
PUNTA_DESC = containers.Map( ...
    {'gris','marron','roja','negra'}, ...
    {'Very Soft','Soft (084B61)','Medium (084B62)','Med/Hard (084B63)'});
```

El PCB Piezotronics 086D20 (martillo de impacto) tiene puntas intercambiables de diferente dureza, cada una con frecuencia de corte distinta. La punta "gris" (Very Soft, fc=170Hz) transmite más energía en bajas frecuencias — ideal para MASW con geófonos de 4.5Hz. La punta "negra" (fc=1600Hz) es para análisis de estructuras de alta frecuencia.

5. **Corrección geométrica**: 1/√r para ondas Rayleigh superficiales (decaimiento cilíndrico vs 1/r³ para ondas de cuerpo)
6. **Estimación de H_sys**: si dos circuitos diferentes se miden al mismo r, H_suelo cancela y la comparación es directa

El comentario al final lo explica mejor que cualquier descripción:
> "IMPORTANTE: H_sys_est absorbe H_suelo. Para comparar dos circuitos al mismo r, H_suelo cancela y la comparación es válida. Para valores absolutos se necesita calibrar H_suelo."

Esto es MASW real. No simulación, no modelo — mediciones reales con martillo real en el suelo real.

#### `marcar_secuencia.m` y `visor_rapido.m` — herramientas de etiquetado

Scripts auxiliares para el flujo de trabajo experimental:
- `marcar_secuencia.m`: permite al usuario marcar manualmente en una gráfica el inicio de cada impacto con el martillo (clicando en el eje de tiempo)
- `visor_rapido.m`: visualizador rápido de sesiones de captura sin procesar, para revisar si la medición es válida antes de guardarla

Estos scripts existen porque el flujo de trabajo en el campo (o en el estacionamiento, como fue en mayo de 2022... digo 2026) requiere revisar rápidamente si un hit fue limpio o si el operador se movió, si hay un click de interferencia, si el geófono se desacoupled del suelo.

#### `migrar_campos.m` — compatibilidad hacia atrás

Un script de migración de datos que actualiza la estructura de los `.mat` guardados por versiones anteriores del scope. Si `geophone_scope_simple.m` cambió los campos guardados (por ejemplo, renombrar `fs` a `Fs`, o agregar el campo `entidad` con metadatos de la sesión), los `.mat` viejos no tienen esa estructura y los scripts nuevos fallan. `migrar_campos.m` agrega los campos faltantes con valores por defecto.

El hecho de que necesite esto indica que hubo capturas reales *antes* del 20 de mayo — probablemente durante el 18 o 19, cuando el firmware `DiferencialToSingleEnded` estaba siendo probado.
