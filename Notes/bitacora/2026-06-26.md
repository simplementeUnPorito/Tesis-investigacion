## Contexto del día

El 25 de junio había cerrado con el título "Pre Geo autocal": el signo de la ganancia física del PGA se había corregido de forma genérica (ya no había bifurcación `#if PSOC_HW_CLASS == PSOC_HW_HAMMER`), y HAMMER calibraba de punta a punta con `ok=1` en ambas etapas. El terreno estaba listo, en teoría, para extender el mismo controlador PI a GEO. El 26 de junio es el día en que eso se intenta en la práctica, y como suele pasar cuando una abstracción se prueba por primera vez contra un caso real nuevo, aparecen fricciones que no se veían venir: constantes de ganancia mal calibradas, un algoritmo legacy que estorbaba en el archivo activo, un bug de escala en la ruta de streaming por hardware, y —al final— un problema de hardware disfrazado de bug de software.

GEO tiene cuatro etapas encadenadas (PGA → BP → ADDER → LP), cada una con su propio VDAC de referencia, su propia ganancia física, y su propio archivo de parámetros (`calibration_tables_geo_pga.h`, `_bp.h`, `_adder.h`, `_lp.h`). A diferencia de HAMMER (solo dos etapas, PGA y LP), GEO ejercita mucho más la generalización del lazo PI: si algo estaba hardcodeado pensando solo en HAMMER, GEO lo iba a exponer.

**4. La novedad algorítmica más interesante: un paso de "refine" de un LSB al cerrar cada etapa.**


y su servicio, un nuevo estado `CAL_PI_REFINE_SETTLE`:

```c

{
    ...
    trial_abs_error = cal_pi_abs_error_counts(stage, sample);
    if (trial_abs_error < g_cal_pi.refine_base_abs_error &&
        cal_pi_measurement_valid(sample)) {

        return cal_pi_finalize_stage(g_cal_pi.refine_ok,
                                     g_cal_pi.refine_trial_dac,
                                     sample);
    }

    stage->write(g_cal_pi.refine_base_dac);
    g_cal_pi.dac_current = g_cal_pi.refine_base_dac;
    g_cal_pi.last_fir_output = g_cal_pi.refine_base_measured;

                                 g_cal_pi.refine_base_dac,
                                 g_cal_pi.refine_base_measured);
}
```

La idea es sutil pero elegante: cuando el PI lockea (converge y se estabiliza), el DAC de 8 bits solo puede tomar valores discretos separados por un LSB (16 mV en GEO). Es perfectamente posible que el lock haya ocurrido en un valor de DAC que no es el más cercano posible al target — que el LSB de al lado, en la dirección correcta, en realidad reduciría más el error. El paso de "refine" prueba exactamente eso: calcula hacia qué lado debería moverse un LSB según el signo del error y la ganancia (reutilizando `cal_pi_effort_delta_sign`, la misma función de anti-windup del día anterior), prueba ese candidato, espera a que el FIR se asiente (`refine_settle_samples`), y si el error absoluto mejora, se queda con el nuevo valor; si no, vuelve exactamente al DAC anterior. Es un mini paso de bisección de un solo candidato, acotado y determinístico, que rescata parte del espíritu de precisión de la bisección legacy recién eliminada, pero sin su complejidad de máquina de estados de 11 nodos — una prueba de que no toda la bisección era mala idea, solo que su implementación completa era demasiado pesada para lo que realmente hacía falta.

**5. El bug de escala en el streaming filtrado por hardware — el origen de los "-20V imposibles".**

En `main.c`, el ISR que atiende las muestras filtradas por DMA:

```c

-    int32 filt = dma_buf_to_i24(g_dma_filt_buf);
+    int32 filt = psoc_adc_counts_right_aligned(dma_buf_to_i24(g_dma_filt_buf));
```

El `TEST_LOG` documenta el síntoma exacto: "This addresses the web showing around `-20 V` with FIR hardware enabled; the ESP/web scale expects right-aligned +/-2.5 V ADC counts, not left-aligned Delta-Sigma words." Este es un bug de escala clásico en sistemas con ADC Delta-Sigma: el filtro de hardware (`Filter`, el DFB) entrega su salida en un formato "left-aligned" (con el bit de signo y los bits significativos corridos hacia la izquierda dentro de la palabra de 32 bits), mientras que el resto del sistema —incluida toda la cadena de conversión a milivolts que ve la web— espera counts "right-aligned" en la escala física real. Sin la corrección, cada muestra filtrada aparecía multiplicada por un factor de escala equivocado, produciendo lecturas de -20V en una señal que en realidad estaba en el rango de ±2.5V. Es exactamente el mismo tipo de bug de alineación de bits que ya se había encontrado "una vez" según el comentario que se borra en `cal_pi_run_service` un poco más abajo en el mismo diff — lo cual sugiere que esta clase de error (izquierda vs. derecha en el shift de escala del ADC) es una fuente recurrente de bugs en este proyecto, y quizás merecería una función de conversión única y bien probada en lugar de aplicarse ad-hoc en cada punto de consumo.

**7. Los coeficientes FIR se mueven de los `.c` a los `.h` como macros editables.**

El `TEST_LOG` lo resume: "`FIR_adquisition.h`: acquisition FIR coefficients are editable in the header... `FIR_adquisition.c` and `FIR_calibration.c`: now only instantiate the arrays from the header macros." **9. La envolvente de Hilbert en la web — post-procesamiento visual sin tocar los datos crudos.**

El archivo nuevo `signal_proc.js` implementa una FFT radix-2 completa (`fftRadix2`, con bit-reversal y mariposa iterativa) y la usa para calcular la señal analítica y su envolvente:

```javascript
export function hilbertEnvelope(buf) {
  const input = ArrayBuffer.isView(buf) ? buf : new Float64Array(buf);
  const nInput = input.length;
  if (!nInput) return new Float64Array(0);
  const n = nextPow2(nInput);
  const re = new Float64Array(n);
  const im = new Float64Array(n);
  re.set(input);

  fftRadix2(re, im, false);
  if (n > 1) {
    const half = n >> 1;
    for (let k = 1; k < half; k++) {
      re[k] *= 2;
      im[k] *= 2;
    }
    for (let k = half + 1; k < n; k++) {
      re[k] = 0;
      im[k] = 0;
    }
  }
  fftRadix2(re, im, true);

  const out = new Float64Array(nInput);
  for (let i = 0; i < nInput; i++) out[i] = Math.hypot(re[i], im[i]);
  return out;
}
```

El algoritmo es el método estándar de transformada de Hilbert vía FFT: se calcula la FFT de la señal real, se duplican las componentes de frecuencia positiva y se anulan las negativas (construyendo así la señal analítica en el dominio de la frecuencia), se aplica la FFT inversa, y la envolvente es simplemente el módulo (`Math.hypot`, es decir `sqrt(re²+im²)`) de la señal analítica compleja resultante. Es exactamente el tipo de herramienta de post-procesamiento que tiene sentido para un sistema MASW: la envolvente de la señal sísmica es útil para visualizar la llegada de un pulso (el golpe del martillo, por ejemplo) sin que el detalle oscilatorio de alta frecuencia oculte el sobre de energía.

La integración en `plot.js` es deliberadamente no invasiva:

```javascript
const raw = (this._showEnvelope && this._raw) ? hilbertEnvelope(this._raw) : this._raw;
const filt = (this._showEnvelope && this._filt) ? hilbertEnvelope(this._filt) : this._filt;
```

El cálculo de la envolvente ocurre únicamente en la capa de dibujo (`draw()`), sobre una copia local de los datos ya cargados en el `ChannelPlot` — nunca sobre los buffers `_raw`/`_filt` originales que se usan para exportar o almacenar. El propio `TEST_LOG` lo aclara: "It does not alter stored/exported raw or filtered data." Es una decisión de diseño acertada: una herramienta de visualización exploratoria (útil para "ver mejor" en el momento) no debería tener la posibilidad de contaminar silenciosamente los datos que después se usan para el análisis MASW real.

**10. El giro narrativo del día: BP/LP no era un bug, era un pin sin soldar.**

La sección más reveladora de todo el día está en la actualización del `TEST_LOG_2026-06-26.md` que cierra este segundo commit:

> "User update: BP/LP calibration issue was due to unsoldered pins; calibration works after soldering. Do not spend the next handoff re-tuning the PI as the first hypothesis unless a new fresh log proves a firmware regression."

Después de toda la sesión de ajuste de constantes PI, límites de integral, timeouts, y el descubrimiento correcto en el commit anterior de que BP "no mostraba autoridad útil" (el VDAC se escribía pero la medición no respondía), la causa raíz termina siendo enteramente física: un pin sin soldar en la placa GEO. Es un recordatorio contundente de la disciplina de diagnóstico que atraviesa buena parte de esta bitácora: cuando un sistema mixto de software y hardware analógico no se comporta como se espera, la telemetría y el código correcto son necesarios pero no suficientes — a veces la respuesta está literalmente en la soldadura, y ninguna cantidad de ajuste de Kp/Ki iba a arreglar un cable que no estaba conectado. La nota explícita para "el próximo handoff" —no volver a sintonizar el PI como primera hipótesis sin un log fresco que pruebe regresión de firmware— es una forma madura de capturar esta lección para uno mismo en el futuro, evitando repetir una ronda entera de ajuste de constantes sobre un problema que ya se sabe que es de otro tipo.

#### Impacto en el proyecto

Este commit cierra el ciclo de bring-up inicial de GEO con tres logros simultáneos, de naturaleza bastante distinta entre sí: (1) una purga radical de deuda técnica que deja `calibration.c` mucho más liviano y legible al eliminar ~800 líneas de bisección legacy que ya no aportaban valor activo; (2) un refinamiento algorítmico genuino del PI (el paso de "refine" de un LSB) que mejora la precisión final sin la complejidad de la bisección completa; y (3) la primera pieza de post-procesamiento de señal en el frontend web (envolvente de Hilbert), que empieza a acercar la herramienta de visualización a las necesidades reales del análisis MASW. Y, quizás lo más valioso desde el punto de vista metodológico, deja documentado por escrito que el problema de convergencia de BP/LP no era software sino un defecto de soldadura — evitando que ese error de diagnóstico se repita en sesiones futuras.
