## Contexto del día

El 8 de junio fue un día de euforia: la Web UI embebida funcionaba, el WebSocket relay retransmitía datos del PSoC hacia el navegador, y el sistema podía operar sin computadora conectada. Pero en el camino hacia ese hito, se había acumulado una deuda técnica específica: un filtro de notch armónico implementado en JavaScript del lado del cliente, que supuestamente debía eliminar los 50 Hz de la red eléctrica argentina y sus armónicos antes de mostrar las señales.

El problema con ese notch JS era doble. Por un lado, la implementación usaba mínimos cuadrados (least-squares harmonic notch), un algoritmo relativamente costoso para ejecutar en el navegador sobre buffers de datos en tiempo real. Por otro lado, la presencia de ruido de red en los datos levantaba una pregunta más profunda: ¿era correcto lidiar con el 50 Hz en el cliente, o habría que resolverlo en el hardware o en el firmware? El DFB notch del PSoC 5LP que se había explorado meses antes era el candidato natural para eso.

El martes 9 de junio, con la Web UI ya funcionando, Elías toma una decisión de diseño importante: **sacar el notch del cliente web** y al mismo tiempo **empezar a investigar en MATLAB** si el algoritmo de Mewett et al. (estimación espectral adaptativa) es lo suficientemente bueno como para usarlo offline en el análisis post-campo. La sesión de trabajo arranca en la tarde y el commit cae a las 20:10 hs.

Este día es también notable por otra decisión: el ADC_SAMPLE_RATE_HZ sube de 1020 Hz a 3000 Hz. A 1020 Hz, la ventana de Nyquist llega solo hasta 510 Hz, lo cual es marginal para una primera visión de las ondas Rayleigh de superficie (que suelen estar entre 5 y 100 Hz, pero sus armónicos y el contenido de la onda martillo pueden llegar más arriba). A 3000 Hz, hay holgura suficiente. Que este cambio conviva en el mismo commit que la remoción del notch no es casual: a 3000 Hz la FFT tiene más resolución para ver el 50 Hz, lo que hace que la alternativa MATLAB sea más relevante.

##### 2. Subida de tasa de muestreo: 1020 → 3000 Hz

En el firmware del master (`main.cpp`), una sola línea cambia el default de `ADC_SAMPLE_RATE_HZ`:

```diff
 #ifndef ADC_SAMPLE_RATE_HZ
-  #define ADC_SAMPLE_RATE_HZ 1020
+  #define ADC_SAMPLE_RATE_HZ 3000
 #endif
```

Este valor es el fallback que usa el master ESP32 para calcular la duración de una captura cuando todavía no ha recibido el `HELLO` packet de los esclavos. La tasa real la reporta el PSoC a través del frame `FS_REPORT` y llega al master en el `HELLO` response de cada esclavo. A 3000 Hz, una captura de 512 batches × 30 muestras/batch × 2 canales = 30.720 muestras dura ~10.24 segundos. A 1020 Hz hubiera sido ~30 segundos. La subida de tasa es consistente con el ADC_DelSig configurado en el PSoC (el Delta-Sigma puede alcanzar fácilmente 3 kHz en modo de conversión continua).

Para que esta tasa se propague correctamente a los cálculos de timeout, se agrega la función `effectiveSampleRateHz()`:

```cpp
static uint16_t effectiveSampleRateHz()
{
    for (uint8_t i = 1; i <= NUM_SLAVES; i++) {
        if (g_cachedHello[i].valid && g_cachedHello[i].sample_rate > 0)
            return g_cachedHello[i].sample_rate;
    }
    return ADC_SAMPLE_RATE_HZ;
}

{
    uint32_t samples = (uint32_t)nBatches * (uint32_t)(SAMPLES_PER_PART * 2);
-    return (samples * 1000UL + (ADC_SAMPLE_RATE_HZ - 1)) / ADC_SAMPLE_RATE_HZ;
+    uint16_t fs = effectiveSampleRateHz();
+    return (samples * 1000UL + (fs - 1)) / fs;

```

El cambio es sutil pero importante: `captureDurationMs()` ahora usa la tasa reportada por el primer esclavo válido que encuentre en `g_cachedHello`, y solo cae al default si no hay ninguno disponible. Esto significa que si el PSoC está configurado a, digamos, 2048 Hz (un valor razonable para el ADC_DelSig en modo diferencial de 16 bits), el master calculará el timeout de captura con esa tasa y no con el valor hardcoded.

##### 5. MATLAB: `prueba_removeLineNoise.m` y `removeLineNoise_SpectrumEstimation.m`

Los dos scripts nuevos son la parte más reveladora del día. `removeLineNoise_SpectrumEstimation.m` es una implementación de MATLAB del algoritmo de Mewett, Nazeran y Reynolds (EMBS 2001, DOI: 10.1109/IEMBS.2001.1017205), que remueve ruido de red por estimación espectral adaptativa: estima el contenido de frecuencia de línea en una ventana deslizante y lo sustrae de la señal, incluyendo manejo de transitorios en bordes mediante padding con un modelo de Fourier de un término.

```matlab

```

El script de prueba (`prueba_removeLineNoise.m`) carga datos reales del directorio `C:\Github\Tesis\Crudos\muestra_20260608_174501` —una captura del día anterior— y genera una figura comparativa de 3 paneles por nodo: señal en tiempo, espectro completo en escala logarítmica, y zoom en 0–300 Hz. Además imprime en consola la reducción en dB para cada armónico:

```matlab

El análisis en MATLAB es offline, ideal para el flujo de trabajo de tesis: se captura en campo, se trae al PC, se procesa en MATLAB, se genera figura para el informe.
