## Contexto del día

Cinco días de pausa entre el commit del 5 de mayo (madrugada) y este domingo 10 de mayo. El trabajo del 4 de mayo había producido 38 sesiones de captura y el framework de análisis espectral. La pregunta pendiente era: ¿cómo se calibra el sistema para que la señal esté centrada alrededor de cero?

El amplificador diferencial del PSoC (OpaBuffer → PGA → OpaDiff → TIA) tiene un problema de offset DC. Las imperfecciones de los componentes internos hacen que la salida del ADC no sea cero cuando el geófono está en reposo — hay un valor constante de offset que domina la señal. Sin calibración, el offset satura el rango del ADC y no se puede medir nada.

"Calibración a manita" = ajustar manualmente los voltajes de referencia de los TIAs (amplificadores de transimpedancia) usando los VDAC del PSoC hasta que la señal quede centrada. El domingo 10 de mayo, esa calibración finalmente funcionó.

#### Nuevo proyecto PSoC: `ComparacionDiferencialMFBIntegradorDigital.cydsn`

Este nombre revela un salto conceptual: el proyecto ya no es "ComparacionDiferencial" genérico. Ahora tiene:
- **MFB**: topología del filtro analógico activo (Multiple Feedback, ya elegida)
- **IntegradorDigital**: un integrador digital en el PSoC

Un integrador digital en el dominio discreto tiene función de transferencia:
```
H(z) = 1 / (1 - z⁻¹)
```

En el dominio continuo equivale a `H(s) = 1/s`. ¿Por qué integrar la señal del geófono?

El SM-24 es un sensor de **velocidad** (la bobina genera voltaje proporcional a la velocidad de movimiento). Para MASW se trabaja típicamente con velocidad de partícula, así que no es necesario integrar para uso normal. Sin embargo:

1. **Comparación con acelerómetros**: Algunos sensores alternativos miden aceleración. Para comparar con el SM-24, se puede integrar la aceleración a velocidad.
2. **Compensación de geófono**: La respuesta de compensación del SM-24 involucra operaciones que en la práctica funcionan como una integración parcial en baja frecuencia.
3. **Desplazamiento**: Para algunos análisis geotécnicos, el desplazamiento (integral de velocidad) es más útil que la velocidad.

El CLEAN.log y REBUILD.log del nuevo proyecto indican que se hizo una limpieza y recompilación completa — primer build del nuevo proyecto desde cero.

#### `DiferencialConRrefs.cydsn`: Variante con resistencias de referencia

El segundo commit también menciona `DiferencialConRrefs.cydsn` — "Diferencial Con Rrefs" (Differential with R references). En contraste con el VDAC digital, esta variante usa resistencias físicas como fuente de referencia para los TIAs.

Las dos aproximaciones:
- **VDAC**: Voltaje de referencia programable digitalmente → calibración por software, pero VDAC tiene ruido de cuantización (8-bit, 16mV/step para rango 0-4V)
- **R refs**: Divisor resistivo → mayor precisión estática, pero no programable

La existencia de dos proyectos paralelos (VDAC y Rrefs) sugiere que se está experimentando para determinar cuál aproximación da menor ruido de offset.

### Commit 2: `b9a956a7` — "Calibración a manita ya funciona" — 23:47 hs

Este es el commit central del día. El mensaje "ya funciona" tiene el peso de resolver un problema que llevaba días pendiente.

#### El problema del offset DC diferencial

El amplificador diferencial del PSoC está compuesto por:
```
V+ (geófono +) → OpaBuffer_p → PGA_p → 
                                          } → OpaDiff → TIA_p → ADC_DelSig → DFB
V- (geófono -) → OpaBuffer_n → PGA_n →
```

En teoría, cuando el geófono está en reposo (V+ = V-), la salida del ADC debería ser 0. En la práctica:
- Cada op-amp tiene un offset de entrada (tipicamente 1-10 mV)
- El PGA tiene su propio offset
- El OpaDiff tiene mismatch entre las dos entradas
- Los TIAs tienen offset de corriente de entrada

El resultado: con el geófono en reposo, la salida del ADC puede estar en +100,000 counts o -200,000 counts — cualquier valor que no es cero. A 18 bits (rango ±262,144 counts), un offset de ±100,000 counts significa que 38% del rango dinámico está perdido en offset.

#### La solución: VDAC references

Los TIAs del PSoC tienen una entrada de referencia. Al cambiar esa referencia, se puede inyectar un offset controlado que cancela el offset del sistema:

```
TIA_p: V_out = -Rf × (I_in) + V_ref_p
TIA_n: V_out = -Rf × (I_in) + V_ref_n
```

El VDAC del PSoC es un DAC de 8 bits que genera el `V_ref`. Al ajustar el código del VDAC (0-255), se mueve la referencia del TIA y por tanto el offset de la señal diferencial.

"Calibración a manita" = ajustar el código VDAC_p y VDAC_n manualmente, mirando el scope en MATLAB, hasta que la señal quede centrada alrededor de cero.

#### Las nuevas estructuras del scope

```matlab
% VRef — referencias de los TIAs
S.vdac_p_ref   = 148;    % pVRef: referencia VDAC_p (0x94 default)
S.vdac_n_ref   = 148;    % nVRef: referencia VDAC_n
S.servo_vdac_p = 148;    % posición actual leída del PSoC
S.servo_vdac_n = 148;
S.dc_mean_mv   = 0;      % promedio DC actual de la señal (mV)
S.servo_configs = struct('pga_code',{},'p_vref',{},'n_vref',{});
```

El código 148 (0x94) es el valor default. En un VDAC de 8 bits:
- 0 → voltaje mínimo (tipicamente 0V o Vss)
- 255 → voltaje máximo (tipicamente Vdd o 4.096V)
- 148 ≈ 148/255 × 4V ≈ 2.32V → punto medio aproximado

Al tener VDp=142 y VDn=142 en el log final del día:
```
VRef actual: VDp=142 VDn=142
VRef cfg: pVRef=142 (0x8E) nVRef=142 (0x8E)
```

El código 142 (0x8E) es ligeramente menor que el default 148. La calibración encontró que reducir ambas referencias por 6 counts cancela el offset residual del sistema.

#### El protocolo de configuración (0xA5)

El log del primer intento (20:12 hs) muestra:
```
[20:12:34.083] Solicitando config al PSoC (0xA5)...
[20:12:36.182] WARN: config incompleta (tipo2=0 tipo3=0) — usando defaults
[20:12:36.196] WARN: sin respuesta de servo (0x04/0x05)
[20:12:36.203] WARN: sin límites VDAC (0x06)
```

La conexión de las 20:12 fue una falla — el firmware no respondía al 0xA5. Pero a las 23:30 hs:
```
[23:30:41.087] Solicitando config al PSoC (0xA5)...
[23:30:41.145] Config OK: ADC=18 bits  fs=1500 SPS  PGA code=0(1x (0x00))
                Vref=±6144mV  pktsz=5  scale=7.324e-04 mV/cnt
[23:30:41.146] VRef actual: VDp=142 VDn=142
```

Entre las 20:12 y las 23:30, se actualizó el firmware del PSoC para responder al comando de configuración. El firmware ahora responde con:
- Bits del ADC (18)
- Frecuencia de muestreo (1500 SPS — ¡volvió a 1500 desde 1020!)
- Código de ganancia PGA (0 = 1×)
- Voltaje de referencia del ADC (±6144 mV)
- Tamaño de paquete (5 bytes)
- Escala en mV/count

La escala `7.324e-04 mV/count`:
```
7.324e-4 = 6144 × 2 / 2^24 = 12288 / 16777216 ≈ 7.324e-4
```

Esto confirma que el PSoC está enviando los datos como enteros de 24 bits (left-shifted desde los 18 bits del ADC_DelSig, multiplicados por 2^6 = 64 por el DFB alignment). La escala está calculada correctamente para este modo.

La línea `*.log` es el cambio crítico. Hasta ahora, todos los archivos de log del scope habían sido rastreados en git — cientos de archivos, millones de líneas. Esto inflaba el repositorio y hacía los commits lentos.

**El domingo de calibración.** El 10 de mayo fue un domingo dedicado a resolver el problema de offset. Desde las 19:33 hs hasta las 23:49 hs — cuatro horas y veinte minutos de trabajo intenso.

**El ciclo de debug del firmware.** La secuencia es clara: firmware sin 0xA5 → primer intento fallido → dos horas de programación → firmware con protocolo de configuración → éxito. No hay atajos en el debug de firmware embebido. Tenés que escribir el código, compilar, flashear, conectar, probar.

**"Calibración a manita"** — el término manual tiene carga técnica. En un sistema ideal, la calibración sería automática: el PSoC mediría su propio offset y lo compensaría. Pero en la fase de desarrollo, el "servo manual" (el usuario ajusta los diales en MATLAB hasta que la señal se centra) es una solución que funciona. Después se puede automatizar.

**VDp=142, VDn=142.** Dos números que representan horas de trabajo. El código 142 cancela el offset de este circuito específico con estos componentes específicos. Si se cambia el PGA gain, el offset cambia (por eso `S.servo_configs` guarda la configuración por código PGA). La calibración no es un valor universal — es una propiedad del circuito físico.

**El .gitignore dos minutos después.** Esta secuencia es elegante: primero commitear el trabajo técnico ("Calibración a manita ya funciona"), luego inmediatamente hacer la limpieza del repositorio ("Ignorando archivos de log"). Primero celebrás el logro, luego hacés el mantenimiento.

**La nueva dirección: IntegradorDigital.** El nuevo proyecto `ComparacionDiferencialMFBIntegradorDigital` abre una pregunta que los siguientes commits van a responder: ¿qué hace el integrador digital? ¿Integra la señal de velocidad para obtener desplazamiento? ¿Es parte de la compensación del geófono? ### Cualitativo

"Ya funciona" — esas dos palabras valen todo el día. El problema de offset del amplificador diferencial era el obstáculo más grande para obtener señal limpia del SM-24 a través del circuito analógico diferencial del PSoC. El sistema de referencias VDAC programables con calibración manual por software resuelve ese problema.

La fs volvió a 1500 SPS (desde 1020 del firmware del notch). Esto sugiere que el filtro notch de 464 taps fue removido o modificado, y el firmware volvió a la tasa de muestreo original del ADC_DelSig. El trade-off es que a 1500 SPS el 50Hz cae en el 6.7% de la banda de Nyquist — más fácil de filtrar con el DFB.

El protocolo de configuración (0xA5 request → respuesta con parámetros) es una mejora arquitectónica importante: MATLAB ya no asume valores hardcodeados del PSoC, sino que los pregunta. Esto hace el sistema más robusto ante cambios de firmware.
