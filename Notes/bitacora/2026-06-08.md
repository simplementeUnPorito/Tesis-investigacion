## Contexto del día

El 7 de junio el sistema había terminado de cerrar su pila de comunicación vertical: PSoC con filtro notch por mínimos cuadrados, protocolo FS_REPORT para que el master sepa la frecuencia real de cada esclavo, símbolos KiCad para el esquemático del nodo. El stack estaba robusto. Pero para operar en campo todavía había una dependencia incómoda: **una laptop conectada por USB al master**. El ESP32 master enviaba sus paquetes binarios por `Serial` hacia MATLAB o geophone_scope.py. Sin la PC, no había forma de armar esclavos, iniciar capturas, ni ver datos en tiempo real.

Esa dependencia tenía un nombre: `MatlabTransport`. Era el canal de salida único del master. Todo —heartbeats, datos, ACKs, estado— pasaba por `Serial`. La arquitectura era correcta para el laboratorio, pero frágil para el campo: llevar una laptop a un terreno sísmico es un obstáculo real.

La solución que se construyó esta noche es elegante porque **no rompe nada**: en lugar de reemplazar `MatlabTransport`, se le agrega un callback de relay. Cada paquete de 6 bytes que ya iba a `Serial` ahora también va a `web_relay.h`, que lo espeja por WebSocket a cualquier navegador conectado al AP `GeoNetwork`. Los comandos del navegador —JSON plano— se decodifican de vuelta a `MatlabTransport::RxCmd` y se encolan para que `loop()` los drene exactamente igual que los comandos de MATLAB. **Un solo despachador. Sin lógica duplicada.**

El resultado: conectás el celular a `GeoNetwork`, abrís `192.168.4.1`, y tenés una interfaz completa de campo. La PC puede quedarse en el auto.

#### Capa 1 — Firmware: `web_server.h`

```cpp
#pragma once
/*
 * web_server.h — Servidor HTTP/WS embebido del maestro (interfaz web de campo).
 *
 * Sirve la SPA desde LittleFS (partición de datos "spiffs" ya presente en la
 * tabla de particiones default.csv, ~1.4 MB) y expone un WebSocket en /ws
 * para telemetría y comandos (ver web_relay.h).
 */
#include <LittleFS.h>
#include <ESPAsyncWebServer.h>
#include "master_log.h"
#include "web_relay.h"

static AsyncWebServer webServer(80);

inline bool webServerBegin()
{
    bool fsOk = LittleFS.begin(true);   /* true = formatear si el mount falla */
    ...
    DefaultHeaders::Instance().addHeader("Cache-Control", "no-store, no-cache, must-revalidate");
    ...
    webServer.on("/health", HTTP_GET, [fsOk](AsyncWebServerRequest *request) { ... });
    webServer.on("/", HTTP_GET, [fsOk](AsyncWebServerRequest *request) {
        if (fsOk && LittleFS.exists("/index.html")) {
            request->send(LittleFS, "/index.html", "text/html");
            return;
        }
        // Fallback HTML con instrucciones de pio run -t uploadfs
        request->send(200, "text/html", "<!doctype html>...<pre>pio run -t uploadfs</pre>...");
    });
    if (fsOk) webServer.serveStatic("/", LittleFS, "/");
    webRelayBegin(webServer);
    webServer.begin();
    return true;

```

`web_server.h` hace tres cosas:
1. Monta LittleFS (con auto-formato si falla el mount — útil en flash virgen)
2. Agrega headers `Cache-Control: no-store` en todas las respuestas (evita que el browser muestre una versión vieja del JS después de un `uploadfs`)
3. Expone `/health` como endpoint de diagnóstico: devuelve texto plano con IP, estado de LittleFS, bytes usados/totales. Útil para saber si el FS está montado sin necesitar UART.
4. Si LittleFS no tiene `/index.html`, sirve un fallback HTML con las instrucciones exactas para subsanar el problema (`pio run -t uploadfs`)

El patrón `AsyncWebServer` con `serveStatic` es el correcto para una SPA: todos los archivos en el directorio raíz del FS son accesibles directamente. El módulo JS usa imports relativos (`./config.js`, `./protocol.js`) que funcionan naturalmente en este esquema.

#### Capa 1 — Firmware: `web_relay.h` — el corazón del sistema

Este header es el logro técnico central de toda la sesión. Son 294 líneas que resuelven un problema de integración sin perturbar el código existente.

```cpp
#pragma once
/*
 * web_relay.h — Puente WebSocket <-> protocolo binario existente.
 *
 * Telemetría (maestro -> navegador): cada paquete de 6 bytes que
 * MatlabTransport ya emite hacia MATLAB (vía _emit, ver matlab_transport.h)
 * se espeja por WS como mensaje binario, con el mismo framing
 * [0x56][node][type][b2][b1][b0] — protocol.js lo decodifica igual que
 * protocol.py, así que no hace falta un protocolo nuevo.
 *
 * Comandos (navegador -> maestro): mensajes de texto JSON planos
 *   {"cmd":"A1","param":1}                      comando estándar   (4 bytes)
 *   {"cmd":"A3","value":100}                    set-N de 16 bits   (5 bytes)
 *   {"cmd":"BD","node":1,"sub":"A6","param":3}  dirigido a esclavo (6 bytes)
 */
```

**Batching de telemetría**

El relay no envía un frame WebSocket por cada paquete de 6 bytes. En su lugar mantiene un buffer de 96 paquetes (576 bytes):

```cpp
static constexpr size_t WEB_RELAY_PKT_LEN = 6;
static constexpr size_t WEB_RELAY_TX_PKTS = 96;
static constexpr size_t WEB_RELAY_TX_BYTES = WEB_RELAY_TX_PKTS * WEB_RELAY_PKT_LEN;
static uint8_t  g_webRelayTxBuf[WEB_RELAY_TX_BYTES];
static size_t   g_webRelayTxLen = 0;
static uint32_t g_webRelayTxFirstMs = 0;
```

Los paquetes se acumulan. El flush ocurre en tres situaciones: el buffer llega a 576 bytes, han pasado más de 2 ms desde el primer paquete encolado (`webRelayService()` lo detecta en `loop()`), o llega un paquete que no es tipo `DATA` (`pkt[2] != 0x00`). Este último caso fuerza flush inmediato para que los heartbeats y ACKs lleguen al navegador sin demora. El cliente JS sabe manejar frames que contienen múltiples paquetes concatenados: itera de 6 en 6 bytes.

**Sección crítica con portMUX**

La función `webRelayPacket()` —el callback que `MatlabTransport` llama por cada paquete emitido— puede ejecutarse desde el contexto del loop principal o desde callbacks de ESP-NOW. Para proteger el buffer compartido:

```cpp
#if defined(ESP32)
static portMUX_TYPE g_webRelayTxMux = portMUX_INITIALIZER_UNLOCKED;
#endif

static inline void webRelayTxLock()  { portENTER_CRITICAL(&g_webRelayTxMux); }
static inline void webRelayTxUnlock(){ portEXIT_CRITICAL(&g_webRelayTxMux); }
```

`portENTER_CRITICAL` en FreeRTOS (ESP32) deshabilita las interrupciones en el core actual. Es la protección más ligera disponible para regiones críticas muy cortas. El diseño es correcto: las secciones protegidas son solo los accesos al buffer y a los índices, no el flush completo.

**Decodificación JSON mínima sin ArduinoJson**

El relay trae su propio parser JSON de propósito específico. La razón es clara: los mensajes del navegador tienen un esquema fijo de dos o tres campos. Sumar ArduinoJson como dependencia —que sí introduce un heap significativo— sería overkill:

```cpp
static bool webRelayJsonStr(const String &s, const char *key, String &out) {
    String pat = String("\"") + key + "\"";
    int k = s.indexOf(pat);
    if (k < 0) return false;
    int colon = s.indexOf(':', k + pat.length());
    if (colon < 0) return false;
    int q1 = s.indexOf('"', colon + 1);
    if (q1 < 0) return false;
    int q2 = s.indexOf('"', q1 + 1);
    if (q2 < 0) return false;
    out = s.substring(q1 + 1, q2);
    return true;
}
```

La función `webRelayDecodeCmd` toma el JSON y lo convierte en `MatlabTransport::RxCmd` — la misma estructura que `matlab.lastCmd()` devuelve cuando llega un comando por Serial:

```cpp
static void webRelayDecodeCmd(const String &json) {
    String cmdStr;
    if (!webRelayJsonStr(json, "cmd", cmdStr)) return;
    uint8_t cmd = webRelayHexByte(cmdStr);

    MatlabTransport::RxCmd r = {0, 0, 0, 0, 0, true};
    r.cmd = cmd;

    if (cmd == MATLAB_CMD_DIRECTED) {            /* 0xBD */
        long node = 0, param = 0;
        String subStr;
        webRelayJsonInt(json, "node", node);
        webRelayJsonStr(json, "sub", subStr);
        webRelayJsonInt(json, "param", param);
        r.node_id = (uint8_t)node;
        r.sub_cmd = webRelayHexByte(subStr);
        r.param   = (uint8_t)param;
    } else if (cmd == 0xA3 || cmd == 0xAE) {    /* set-N de 16 bits */
        long value = 0;
        webRelayJsonInt(json, "value", value);
        r.value = (uint16_t)value;
        r.param = (uint8_t)(value & 0xFF);
    } else {
        long param = 0;
        webRelayJsonInt(json, "param", param);
        r.param = (uint8_t)param;
    }

    webRelayPushCmd(r);
}
```

El resultado llega a una cola circular de 8 posiciones que `loop()` drena en cada iteración. Esta es exactamente la misma ruta que los comandos de MATLAB. No hay un segundo `switch-case` para el web; `handleMatlabCmd()` no sabe si el comando vino de `Serial` o del WebSocket.

**Límite de 1 cliente WS**

El evento `WS_EVT_CONNECT` rechaza conexiones entrantes si ya hay otro cliente activo:

```cpp
case WS_EVT_CONNECT: {
    bool busy = false;
    for (auto &c : server->getClients()) {
        if (c.id() != client->id() && c.status() == WS_CONNECTED) { busy = true; break; }
    }
    if (busy) {
        MASTER_LOG_PRINTF("[WEB] WS cliente #%u rechazado (ya hay otro conectado, %s)\n", ...);
        client->close();
        break;
    }
    MASTER_LOG_PRINTF("[WEB] WS cliente #%u conectado (%s)\n", ...);
    if (g_webRelayClientHook) g_webRelayClientHook();
    break;
}
```

El comentario explica el razonamiento para rechazar al *recién llegado* en lugar de desalojar al *que ya está*: si se expulsa al cliente existente, `ws_client.js` reconecta en 1 segundo, expulsa al nuevo que acaba de entrar, que reconecta... loop infinito de "conectado/desconectado todo el rato" que ya fue observado en campo durante el desarrollo. Rechazar al nuevo rompe el ciclo: el segundo celular simplemente no entra.

**El beacon_interval y el zombie-connection problem:**

Este es probablemente el cambio más interesante desde el punto de vista de los trade-offs de diseño. El comentario del commit lo explica con honestidad:

```cpp

{
    /* Mantener el AP visible durante capturas web.
     *
     * La pausa RF anterior usaba beacon_interval=60000 TU (~61 s). Eso protege
     * más contra beacons, pero Windows/telefonos abandonan GeoNetwork en
     * capturas reales y la pagina queda inaccesible durante minutos. Como el
     * muestreo real queda guardado en RAM en los esclavos y el dump ocurre
     * despues, preferimos mantener la asociacion WiFi viva con beacons lentos.
     */
    wifi_config_t ap_cfg = {};
    esp_wifi_get_config(WIFI_IF_AP, &ap_cfg);
-   ap_cfg.ap.beacon_interval = 60000u;   /* max documentado TU ≈ 61 s */
```

El beacon original de 61 segundos fue puesto en la sesión de mayo para reducir la interferencia WiFi sobre el ADC del PSoC durante la captura. Con la interfaz web, ese beacon largo provocaba que Windows y Android consideraran que la red había desaparecido y abandonaran la asociación. Cuando el dump terminaba y el AP volvía a beaconear normal, el navegador estaba desconectado y perdía todos los datos del dump que habían llegado mientras tanto. La solución es un beacon de ~1 segundo: lento para reducir tráfico RF, pero suficientemente frecuente para mantener la asociación WiFi viva.

**La delivery guard para batches duplicados:**

Las pruebas del relay descubrieron que en condiciones de alta carga WebSocket, el callback `onBatchReady` a veces procesaba el mismo batch dos veces. La causa: el flag `g_dumpBatchRx` tardaba un ciclo de `loop()` en activarse, y si `onBatchReady` era llamado dos veces muy seguidas (por fragmentación de paquetes ESP-NOW o por timing), el segundo llamado pasaba la guardia inicial:

```cpp
static volatile bool     g_dumpBatchDelivered = false;
static volatile uint8_t  g_dumpDeliveredNode  = 0;
static volatile uint16_t g_dumpDeliveredSeq   = 0;

static inline void dumpDeliveryGuardReset() {
    g_dumpBatchDelivered = false;
    g_dumpDeliveredNode = 0;
    g_dumpDeliveredSeq = 0;
}

static inline void dumpDeliveryGuardMark(uint8_t nodeId, uint16_t seq) {
    g_dumpDeliveredNode = nodeId;
    g_dumpDeliveredSeq = seq;
    g_dumpBatchDelivered = true;
}
```

En `onBatchReady`:

```cpp
if (g_dumpBatchDelivered &&
    g_dumpDeliveredNode == batch.node_id &&
    g_dumpDeliveredSeq == batch.seq) {
    MASTER_LOG_PRINTF("[MASTER] DUMP duplicate delivered ignored node=%d seq=%d\n", ...);
    return;
}
if (g_dumpBatchRx) {
    MASTER_LOG_PRINTF("[MASTER] DUMP duplicate ignored node=%d seq=%d\n", ...);
    return;
}
dumpDeliveryGuardMark(batch.node_id, batch.seq);
```

La guardia marca el par `(node, seq)` tan pronto como se entrega el batch, antes de que `g_dumpBatchRx` se active. El `dumpDeliveryGuardReset()` se llama cada vez que se avanza al siguiente batch o se reinicia el dump. Es una solución idiomática: una variable extra de "ya entregué esto" más granular que el flag booleano existente.

#### Capa 2 — Frontend: `protocol.js`

```js
// protocol.js — decode 6-byte master->browser packets, encode browser->master
// commands as the flat JSON the firmware's web_relay.h expects.
// Mirrors python/geophone_scope/protocol.py — keep field names/semantics in sync.

export class Packet {
  constructor(nodeId, ptype, b2, b1, b0) {
    this.nodeId = nodeId;
    this.ptype = ptype;
    const raw = (b2 << 16) | (b1 << 8) | b0;
    this.value24u = raw >>> 0;
    this.value24 = (raw & 0x80_0000) ? raw - 0x100_0000 : raw;
  }
  get isData() { return this.ptype === cfg.PTYPE_DATA; }
  get helloFsHz() { return this.b0 * 100; }
  ...
}

export function decodePacket(buf) {
  if (buf.length < cfg.PKT_LEN) return null;
  if (buf[0] !== cfg.PKT_HEADER) return null;
  return new Packet(buf[1], buf[2], buf[3], buf[4], buf[5]);
}

const hex2 = (b) => b.toString(16).toUpperCase().padStart(2, '0');
export function encodeStd(cmd, param) {
  return JSON.stringify({ cmd: hex2(cmd & 0xFF), param: param & 0xFF });
}
export function encodeDirected(nodeId, subCmd, param) {
  return JSON.stringify({
    cmd: hex2(cfg.CMD_DIRECTED),
    node: nodeId & 0xFF,
    sub: hex2(subCmd & 0xFF),
    param: param & 0xFF,
  });
}
```

`protocol.js` es el espejo exacto de `protocol.py`. La signatura de los campos, los nombres de los tipos de paquete (`PTYPE_DATA`, `PTYPE_HEARTBEAT`, `PTYPE_READY`), la extensión de signo del valor de 24 bits: todo es idéntico. El comentario lo dice explícitamente: "keep field names/semantics in sync". Esto es diseño de API consciente: si mañana se agrega un nuevo tipo de paquete en el firmware, hay que agregarlo en `protocol.py` Y en `protocol.js`. El costo es manual, pero el beneficio es que la lógica de decodificación no está duplicada en el firmware.

#### Capa 2 — Frontend: `signal_proc.js`

El módulo más técnico del frontend. Implementa en JavaScript puro los mismos filtros FIR que `signal_proc.py` implementa con scipy:

```js
function firwin(numTaps, bands, scaleFrequency) {
  const alpha = 0.5 * (numTaps - 1);
  const b = new Float64Array(numTaps);
  for (let n = 0; n < numTaps; n++) {
    const m = n - alpha;
    let h = 0;
    for (const [left, right] of bands) {
      h += right * sinc(right * m) - left * sinc(left * m);
    }
    b[n] = h * hamming(n, numTaps);
  }
  // Normalización de ganancia
  let response = 0;
  for (let n = 0; n < numTaps; n++) {
    const m = n - alpha;
    response += b[n] * Math.cos(Math.PI * m * scaleFrequency);
  }
  if (Math.abs(response) < 1e-15) throw new Error('cannot scale FIR coefficients');
  for (let n = 0; n < numTaps; n++) b[n] /= response;
  return b;
}
```

La implementación de `firwin` en JS replica el algoritmo de scipy: suma de funciones sinc ventaneadas con Hamming, normalización para ganancia unitaria en la frecuencia de referencia. Las validaciones son explícitas: cutoff dentro del rango `(0, fs/2)`, tap count impar (requerido para high-pass y band-stop con scipy), verificación de respuesta no nula antes de normalizar.

La interfaz de usuario del filtrado es una línea de texto: `"lp 20"`, `"bp 5 40"`, `"notch 50"`. La función `compileFirCmd` parsea ese string y llama al diseñador FIR correcto. El DSP es puramente client-side: los datos ya llegaron al browser, se filtran ahí, nunca vuelven al ESP32.

#### Capa 2 — Frontend: `app.js` — orquestación de la SPA

`app.js` (824 líneas) es el cerebro del frontend. Instancia todos los módulos y conecta los eventos:

El patrón de `WsClient` con reconexión automática es fundamental para el campo: si el AP pierde momentáneamente la conexión, el cliente reconecta solo (backoff desde 1 segundo). La lógica de captura traduce segundos a batches con la frecuencia real del sistema:

```js
function batchesForSeconds(seconds) {
  const fs = currentFsHz();
  const batches = Math.ceil((seconds * fs) / cfg.SAMPLES_PER_BATCH);
  return clamp(Math.max(1, batches), 1, cfg.PSOC_CAPTURE_MAX_BATCHES);
}
```

`cfg.PSOC_CAPTURE_MAX_BATCHES = 512` es el límite del firmware (PSoC RAM). Si el usuario pide 10 segundos a 1000 Hz con 100 samples/batch, son 100 batches — dentro del límite. A 2000 Hz serían 200 batches. La función `updateCapturePreview()` muestra en tiempo real cuántos batches serán y cuántos segundos reales durará la captura (que puede diferir del pedido por el ceiling de los batches).

### Cualitativo

Este es el commit más grande del proyecto en términos de líneas. También es probablemente el más importante en términos de impacto operacional: elimina la dependencia de la laptop en el flujo de trabajo de campo.

La decisión de diseño más importante es la que no se ve en el código nuevo: **no tocar `handleMatlabCmd()`**. Todo el trabajo nuevo está en la capa de transporte —`web_relay.h`— que convierte JSON a `RxCmd` y lo encola. El despachador de comandos es el mismo de siempre. Esta decisión evita introducir bugs en código que ya funciona y garantiza que el comportamiento del sistema desde el web sea exactamente idéntico al comportamiento desde MATLAB. Si ARM funciona por MATLAB, funciona por web. Si START falla por web, también fallaría por MATLAB con los mismos parámetros.

El bug de los batches duplicados (`g_dumpBatchDelivered`) es interesante porque sugiere que el relay introduce carga nueva en el loop principal del ESP32 —el WS flush agrega trabajo— que puede cambiar sutilmente el timing de `onBatchReady`. La solución con una delivery guard por `(node, seq)` es correcta y robusta: incluso si `onBatchReady` es llamado tres veces para el mismo batch (hipótesis extrema), solo el primero pasa.

El cambio de `beacon_interval` de 60000 TU a 1000 TU durante la captura es un trade-off explícitamente documentado. La interferencia del beacon sobre el ADC del PSoC fue uno de los problemas centrales de las sesiones de mayo: 10 Hz de spikes visibles en la señal del geófono. Bajar el beacon a 1 segundo significa más beacons durante la captura que con el beacon de 61 segundos, pero la captura en el PSoC es independiente del WiFi (queda en RAM), y el dump post-captura necesita que el AP esté disponible. El comentario lo dice: "el muestreo real queda guardado en RAM en los esclavos y el dump ocurre después, preferimos mantener la asociación WiFi viva". Es una decisión razonable para el caso de uso real.
