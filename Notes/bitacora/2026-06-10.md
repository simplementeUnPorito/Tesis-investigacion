##### La secuencia PRESTART → ARMED → SYNC_IN → SAMPLING

El primer hallazgo es una confirmación positiva: la arquitectura de handshake funciona como se pensaba. Cuando el master manda `CMD_PRESTART` (`0x61`) a cada esclavo por ESP-NOW unicast, el esclavo ejecuta `enterHotWait()`, que a través de dos mensajes UART prepara al PSoC:

```c
psoc.setN(n_batches)  // manda 0xA3 → PSoC configura cuántos batches capturar
psoc.preStart()       // manda 0xB1 → PSoC entra a psoc_arm()
```

Y el PSoC al recibir `0xB1` ejecuta:

```c
static void psoc_arm(void)
{
    ADC_StopConvert();
    capture_reset_locked();
    g_state = PSOC_ARMED;   // <- queda armado, NO muestrea todavía
}
```

Es decir: el PSoC detiene cualquier conversión en curso, resetea los buffers, y queda en espera activa. No va a muestrear hasta recibir un flanco en el pin `SYNC_IN` (P12[6]). El disparo real es por GPIO, no por UART. La ISR correspondiente es `isr_SyncIn`:

```c
CY_ISR(isr_SyncIn)
{
    if (SYNC_IN_Read())
    {
        if (g_state == PSOC_ARMED)
            psoc_enter_sampling(g_debug_psoc);  // recién aquí -> PSOC_SAMPLING
    }
    ...
}
```

Y `psoc_enter_sampling` hace:

```c
static void psoc_enter_sampling(uint8 debugMode)
{
    ADC_StopConvert();
    timer_stop_quiet();
    saved = CyEnterCriticalSection();
    capture_reset_locked();
    g_state = PSOC_SAMPLING;
    CyExitCriticalSection(saved);
    ADC_StartConvert();
}
```

Este diseño es conceptualmente sólido. Todo el trabajo pesado —configurar N batches, armar buffers, hacer arm del ADC— está fuera del camino crítico de tiempo real. Cuando llega el `CMD_START`, el único trabajo que queda del lado PSoC es atender la ISR de GPIO y poner `ADC_StartConvert()`. Es lo que debería hacerse.

##### No hay DMA: el muestreo es muestra a muestra por interrupción

Un detalle que no era obvio desde el código de alto nivel: no hay DMA en este firmware. La ISR `isr_DelSigReady` (`main.c:237-260`) lee `ADC_GetResult32()` muestra a muestra, cada vez que el ADC Delta-Sigma señaliza conversión completa. Esto tiene implicancias para el análisis de determinismo: la ruta de código en `psoc_enter_sampling` no tiene ramas por datos, pero el `ADC_StopConvert()` + `ADC_StartConvert()` reinicia el pipeline del modulador sigma-delta. Hay un retardo de grupo fijo —varios períodos del modulador— antes de que la primera muestra sea válida. Este retardo es igual en todos los nodos (porque todos usan el mismo firmware y la misma configuración de ADC), por lo tanto no agrega jitter *relativo entre nodos*, pero desplaza el "t=0 real" respecto al flanco SYNC. Nunca fue medido ni calibrado.

##### El hallazgo crítico: ACK antes de GPIO

El punto más importante del documento, y probablemente el que mantuvo a Elías despierto hasta la madrugada, está en la sección P2 del análisis. En `slave/src/main.cpp`, el handler de `CMD_START` en `onDataRecv` hace esto:

```c
sendStartAck(1, startToken, nowUs);   // <-- esp_now_send() ANTES del GPIO
digitalWrite(SYNC_TO_PSOC_PIN, HIGH); // <-- esto dispara al PSoC
g_t_start_us    = msg->t_start_us;
...
g_state = SAMPLING;
```

El problema: `sendStartAck()` llama internamente a `esp_now_send()`, que involucra el driver WiFi del ESP32: adquisición de locks, copia de buffer al stack WiFi, encolado en la cola de transmisión. Esta es una operación de latencia **variable y no acotada**, y está ocurriendo *antes* del flanco que activa al PSoC.

Peor aún: `onDataRecv` no es una ISR de hardware. Es el callback de recepción de ESP-NOW, que corre en la **tarea WiFi de FreeRTOS**. Esta tarea compite con otras: el web server, la lectura UART desde el PSoC (`psoc.poll()`), y cualquier otra tarea del sistema. El scheduler de FreeRTOS puede preemptear o no esta tarea en cualquier momento entre que llega el paquete ESP-NOW y que se ejecuta `digitalWrite(SYNC_TO_PSOC_PIN, HIGH)`.

Para comparar: existe en el firmware un handler `onSyncEdge` marcado `IRAM_ATTR` —una ISR de GPIO real, que corre con latencia mínima y determinista— pero **no está conectado al flujo actual**. El pin `SYNC_OUT_PIN` del master existe solo como marcador de osciloscopio según el `README.md` y el `platformio.ini`. La cadena de sincronización real es 100% software ESP-NOW.

El jitter medido `<400µs` reportado por el usuario corresponde a la llegada del paquete a nivel radio. El camino completo que no está medido es:

```
radio RX → tarea WiFi despacha onDataRecv → checks de estado →
sendStartAck() [esp_now_send, variable] → digitalWrite(SYNC_TO_PSOC) →
isr_SyncIn (PSoC) → ADC_Stop/Start (pipeline delay fijo) → primera muestra
```

Y lo crítico para MASW no es el jitter absoluto de cada nodo, sino el jitter **relativo entre nodos**: si el ESP de un nodo tarda 200µs más que otro en ejecutar `sendStartAck()` antes del `digitalWrite`, eso se traduce directamente en un error de fase entre los canales.

##### El presupuesto de jitter para MASW

El documento cierra el análisis técnico con el cálculo que define si el sistema es válido para su propósito:

Para error de fase `Δφ < 5°` (≈ 0.087 rad) entre nodos:

```
Δφ = 2π · f · Δt
Δt < Δφ / (2π · f)

A f = 50 Hz  → Δt < 280 µs
A f = 100 Hz → Δt < 140 µs
```

Con un jitter inter-nodo desconocido (y potencialmente dominado por el tramo `sendStartAck()` → `digitalWrite`), el sistema no puede garantizar esta especificación para la banda 5-50 Hz objetivo de MASW sin medir primero. El fix es simple (invertir dos líneas de código), pero no medir su efecto real sería un error.

Lo que ocurrió fue una auditoría técnica del camino crítico de sincronización, hecha a las madrugada con la meticulosidad de alguien que sabe que está por cerrar el proyecto y que un error de fase entre geófonos puede invalidar todos los datos de campo.

El hallazgo del orden incorrecto `sendStartAck()` → `digitalWrite()` es de esos que dan ganas de golpearse la frente: el código funciona, el sistema sincroniza "suficientemente bien" para muchos casos, pero si se quisiera hacer MASW serio —y ese es el objetivo de la tesis— hay que garantizar que el jitter inter-nodo está por debajo de los 280µs a 50 Hz. Y ese número no se puede garantizar sin (a) invertir el orden de las dos líneas, (b) medir con osciloscopio de tres canales el jitter resultante, y (c) calibrar el pipeline delay del Delta-Sigma del PSoC.

La otra observación importante es sobre el Delta-Sigma: no hay DMA. El muestreo es muestra a muestra por interrupción. Esto es más simple pero tiene implicancias: cuando llega el flanco SYNC, `ADC_StopConvert()` + `ADC_StartConvert()` reinicia el pipeline del modulador, y hay un retardo de grupo —constante pero no calibrado— antes de la primera muestra válida. Este offset fijo es igual en todos los nodos, por lo tanto no contribuye al jitter *relativo*, pero sí desplaza el t=0 real respecto al flanco de disparo. Si alguna vez se necesita sincronizar con una fuente externa (golpe de mazo, GNSS PPS) este retardo importa.

El documento de handoff termina con cuatro próximos pasos claros. El primero y más urgente es invertir el orden de `sendStartAck()` y `digitalWrite()` en `onDataRecv`. El segundo es instrumentar un pin de debug para medir el jitter con osciloscopio. El tercero es caracterizar el pipeline delay del Delta-Sigma. El cuarto es recalcular el techo de frecuencia con los números reales.

La madrugada del 10 de junio es el momento en que el proyecto pasa de "funciona" a "sabemos exactamente por qué funciona, qué no está garantizado, y qué hay que medir". Eso es madurez de ingeniería.
