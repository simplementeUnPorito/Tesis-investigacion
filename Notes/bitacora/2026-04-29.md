### Desarrollo detallado: `notch.m` — Diseño del filtro FIR notch 50Hz

El 50Hz de la red eléctrica es el enemigo público número uno de cualquier sensor de baja frecuencia. El SM-24 opera principalmente en el rango 4-100Hz. La interferencia de la red eléctrica cae exactamente en el medio de la banda de interés. Sin un notch a 50Hz, cualquier medición en campo va a estar contaminada.

#### El script `notch.m`

```matlab
Fs = 1020;              % Nueva frecuencia de muestreo (baja de 1500 SPS)
f_notch = 50;           % Hz — red eléctrica argentina
bw = 10;                % Hz — ancho de banda de rechazo
atenuacion_db = 30;     % dB — atenuación en la muesca
```

Hay un cambio importante aquí: la frecuencia de muestreo bajó de **1500 SPS a 1020 SPS**. Esto no es arbitrario — 1020 es múltiplo de 60 (1020 = 17×60) y también permite muestreo estable a 50Hz con Nyquist en 510Hz. El filtro notch a 50Hz sobre 1020Hz cae exactamente en el 9.8% del ancho de banda de Nyquist.

#### Las bandas del filtro

```matlab
f1 = f_notch - bw;   % = 40 Hz — inicio de la banda de transición izquierda
f2 = f_notch - 0.5;  % = 49.5 Hz — inicio de la muesca
f3 = f_notch + 0.5;  % = 50.5 Hz — fin de la muesca
f4 = f_notch + bw;   % = 60 Hz — fin de la banda de transición derecha
```

La muesca es de solo **1Hz de ancho** (de 49.5 a 50.5 Hz), con transiciones de 9.5Hz en cada lado. Esta especificación agresiva explica por qué el filtro necesita tantos coeficientes.

#### Por qué `firls` en lugar de `firpm`

```matlab
try
    b = firls(n, fo, ao, w);
catch
    disp('firpm falló, usando diseño por ventana (fir1)...');
    b = fir1(n, [f2 f3]/(Fs/2), 'stop');
end
```

`firpm` (Parks-McClellan equiripple) es el estándar para FIR de mínimo orden, pero falla con bandas de transición muy estrechas — el algoritmo de intercambio de Remez diverge cuando la banda es demasiado angosta. `firls` (Least Squares) es más robusto para esos casos aunque no garantiza equiripple.

La corrección crítica documentada en el código:

```matlab
if n < 40
    n = 40;  % mínimo para que la muesca exista
end
if mod(n,2) ~= 0
    n = n + 1;  % Tipo I: simétrico, orden par → fase lineal garantizada
end
```

Un filtro FIR tipo I (longitud impar, N+1 coeficientes con N par) tiene fase lineal exacta. Para geofísica esto es crítico: la información sísmica está tanto en la amplitud como en la fase de las ondas. Un filtro IIR o FIR tipo II distorsionaría la forma de onda.

#### Resultado: 465 coeficientes double

```
/* Generado para Atenuacion: 40 dB - Orden: 464 */
#define FILTER_ORDER 464
double filter_coeffs[465] = {
    0.000047647201723,
    0.000061991840294,
    ...
};
```

El filtro final tiene **464 taps** (orden N=464, N+1=465 coeficientes). Para cada muestra, el PSoC necesita realizar 465 multiplicaciones y sumas. Con `double` (flotante de 64 bits), esto es computacionalmente intensivo para un Cortex-M3.

El costo computacional: a 1020 SPS con 465 multiplicaciones float64 por muestra:
- 1020 × 465 ≈ 474,300 operaciones float/segundo

El Cortex-M3 del PSoC 5LP no tiene FPU (Floating Point Unit). Cada operación `double` se ejecuta en software, lo que puede tomar 10-100 ciclos. Esto es un tema que va a aparecer en commits posteriores.

### Desarrollo detallado: `main.c` — Firmware completamente reescrito

El `main.c` de `ComparacionDiferencial.cydsn` fue reescrito desde cero. El flujo de señal es:

```
SM-24 geophone
  ↓ (señal diferencial analógica)
OpaBuffer_p / OpaBuffer_n  (buffers de alta impedancia para los terminales del geófono)
  ↓
PGA_p / PGA_n  (amplificadores de ganancia programable, canal + y -)
  ↓
OpaDiff_p / OpaDiff_n  (etapas op-amp diferenciales)
  ↓
TIA_p / TIA_n  (transimpedancia, convierte diferencial a single-ended)
  ↓
ADC_DelSig  (Delta-Sigma 18-bit, ~1020 SPS, ±6.144V)
  ↓ (por DMA / ISR)
Filter DFB  (Digital Filter Block hardware — pasa-banda en hardware)
  ↓ (por ISR)
FIR notch software  (464 taps, 50Hz, en main loop)
  ↓
UART_PC @ 460800 baud  (protocolo 5 bytes → MATLAB)
```

#### Los dos ISR

El firmware es completamente orientado a interrupciones:

```c
/* ISR: ADC_DelSig fin de conversión */
CY_ISR(isr_DelSigReady)
{
    ds = ADC_DelSig_GetResult32();
    Filter_Write24(Filter_CHANNEL_A, (uint32)ds);
}

/* ISR: Filter DFB canal A listo */
CY_ISR(isr_FilterReady)
{
    uint32 raw = Filter_Read24(Filter_CHANNEL_A);
    g_flt = (raw & 0x800000u) ? (int32)(raw | 0xFF000000u) : (int32)raw;
    g_filter_ready = 1u;
}
```

`isr_DelSigReady` — se ejecuta al final de cada conversión del ADC. Lee el resultado de 32 bits con signo y lo escribe directamente al Filter DFB mediante `Filter_Write24`. El Filter no tiene DMA en esta configuración — se alimenta directo desde la ISR.

`isr_FilterReady` — se ejecuta cuando el Filter DFB termina de procesar (señal `Filter_DFB_DMA_Req_A`). Lee el resultado de 24 bits y hace sign-extension manual:

```c
g_flt = (raw & 0x800000u) ? (int32)(raw | 0xFF000000u) : (int32)raw;
```

Este es el truco clásico de sign-extension de 24 a 32 bits: si el bit 23 está seteado (negativo en int24), se completan los bits 24-31 con `0xFF` para obtener el int32 correcto.

#### El filtro FIR notch en C

```c
static int32 fir_notch(int32 input)
{
    static int32  fir_buf[FILTER_ORDER + 1];  /* 465 samples de historia */
    static uint16 fir_head = 0u;
    double acc = 0.0;
    uint16 i, idx;

    fir_buf[fir_head] = input;
    idx = fir_head;
    for (i = 0u; i <= (uint16)FILTER_ORDER; i++)
    {
        acc += (double)fir_buf[idx] * filter_coeffs[i];
        idx = (idx == 0u) ? (uint16)FILTER_ORDER : (idx - 1u);
    }
    fir_head = (fir_head >= (uint16)FILTER_ORDER) ? 0u : (fir_head + 1u);
    return (int32)acc;
}
```

Buffer circular de 465 entradas, acumulador `double` para precisión máxima. La indexación reversa `idx - 1` recorre el buffer de más reciente a más antiguo, que coincide con el orden de los coeficientes FIR (coeficientes 0, 1, ..., N en orden del más antiguo al más reciente en las muestras).

#### El nuevo protocolo: 5 bytes con header `0x56`

```c
pkt[0] = 0x56u;   /* Header 'V' (ASCII 86) */
pkt[1] = 0x00u;   /* Tipo: 0=dato, 1=heartbeat */
pkt[2] = (uint8)((notch_out >> 16) & 0xFFu);
pkt[3] = (uint8)((notch_out >>  8) & 0xFFu);
pkt[4] = (uint8)( notch_out        & 0xFFu);
UART_PC_PutArray(pkt, 5u);
```

El header `0x56` es 'V' en ASCII — posiblemente "Versión" o simplemente un marcador diferente al `0xAA` del firmware original para evitar confusiones de parser.

**Heartbeat automático cada ~2 segundos:**

```c
if (++hb_count >= 3000u)
{
    hb_count = 0u;
    pkt[0] = 0x56u; pkt[1] = 0x01u;
    pkt[2] = 0u;    pkt[3] = 0u;   pkt[4] = 0u;
    UART_PC_PutArray(pkt, 5u);
}
```

3000 muestras @ 1500 SPS = 2 segundos. El heartbeat permite a MATLAB detectar si el PSoC se colgó (si no hay heartbeat en más de 2s, algo está mal).

#### Modo DEBUG con rampa

```c
#define DEBUG 0

#if DEBUG
    {
        static uint32 dbg_cnt = 0u;
        pkt[2] = (uint8)((dbg_cnt >> 16) & 0xFFu);
        pkt[3] = (uint8)((dbg_cnt >>  8) & 0xFFu);
        pkt[4] = (uint8)( dbg_cnt        & 0xFFu);
        dbg_cnt = (dbg_cnt + 1u) & 0x00FFFFFFu;
    }
```

`DEBUG=1` envía una rampa 0, 1, 2, 3, ... en lugar de datos reales. Si MATLAB muestra una rampa suave, el byte order y la alineación de paquetes son correctos. Si la rampa está scrambled (saltando, con huecos), hay un problema de sincronización. Es un test de protocolo clásico.

#### Inicialización del circuito diferencial

```c
/* Front-end analógico diferencial */
OpaBuffer_p_Start();
OpaBuffer_n_Start();
PGA_p_Start();
PGA_n_Start();
OpaDiff_p_Start();
OpaDiff_n_Start();
TIA_p_Start();
TIA_n_Start();

/* Filter DFB */
Filter_Start();
Filter_SetCoherency(Filter_CHANNEL_A, Filter_KEY_LOW);
Filter_SetDalign(Filter_STAGEA_DALIGN, 0u);
Filter_SetDalign(Filter_HOLDA_DALIGN,  0u);
```

La cadena analógica completa del PSoC 5LP tiene 8 componentes analógicos activos, todos en el chip. Esta es la ventaja de trabajar con un PSoC — no necesitás chips externos para la cadena de amplificación diferencial. Todo está en silicio.

`Filter_SetCoherency(Filter_CHANNEL_A, Filter_KEY_LOW)` — controla el modo de lectura coherente del DFB: `KEY_LOW` significa que la captura del resultado se dispara por el byte menos significativo, garantizando que los 24 bits se lean de forma atómica.

### El DFB (`dfb.v2`): El filter hardware del PSoC

La configuración del Digital Filter Block (DFB) del PSoC está en ensamblador de muy bajo nivel:

```asm
initialize:
    acu(setmod,setmod) dmux(sa,sa) alu(set0) mac(clra)
    acu(clear,clear) dmux(sa,sa) alu(hold) mac(hold) jmp(eob, WaitForNew)

WaitForNew:
    acu(hold,hold) dmux(sa,sa) alu(hold) mac(hold) jmpl(in1, PROGRAM)

ChA_init:
    acu(loadm, loadm) addr(ChA_MAX) dmux(sa,sa) alu(hold) mac(hold)
    ...

ChA_fir:
    acu(incr,incr) dmux(sra,srm) alu(setb) mac(macc) jmpl(eob,acubeq, ChA_firFinish)
```

El DFB del PSoC 5LP es un procesador DSP dedicado con su propio set de instrucciones. `mac(macc)` es la instrucción de Multiply-Accumulate — el corazón de cualquier filtro FIR. El ensamblador DFB es generado por PSoC Creator basado en los coeficientes que se le pasan al componente Filter, pero el texto `.v2` muestra la implementación real.

El DFB puede ejecutar FIR hardware mucho más rápido que el Cortex-M3 software. Pero el filtro DFB tiene limitaciones de coeficientes (tipicamente 32-128 taps). El FIR notch de 464 taps se hace en software en el Cortex-M3 porque excede la capacidad del DFB.

### `geophone_scope_simple.m` — Actualizado para el protocolo de 5 bytes

La actualización principal del scope es el cambio de parser. La versión anterior buscaba header `0xAA` con paquetes de 8 bytes. La nueva versión busca `0x56` con paquetes de 5 bytes:

```matlab
S.pktBuf = zeros(5, 1, 'uint8');   % antes era 8
S.fs     = 1020;                   % antes era 1500
S.scale  = 1000.0 / (21333.0 * 2.0 * 64.0);  % escala correcta
```

La escala `1000/(21333×2×64)` tiene tres factores:
- `21333`: counts por voltio del ADC_DelSig en CFG1 ±6.144V
- `×2`: ganancia del buffer analógico (OpaBuffer)
- `×64`: shift de 6 bits en el DFB (el DFB escala la salida left-shifted 6 bits para aprovechar el rango de 24 bits)

El factor 64 viene del `Filter_SetDalign(Filter_STAGEA_DALIGN, 0u)` — el alineamiento "izquierdo" del DFB multiplica efectivamente la salida por 2^(24-18) = 2^6 = 64.

El scope también gana nueva funcionalidad: **configuración persistida en `scope_config.mat`** y un campo para ingresar expresiones MATLAB del filtro (`fir1(40, 0.1)`, `firpm(...)`, etc.) directamente desde la UI.

### Narrativa del proceso

**Nueve horas de firmware y pruebas.** El 28 de abril fue una jornada intensa. Dieciocho sesiones de captura no son prueba de que todo salió bien — son prueba de que el proceso de depuración fue riguroso. Conectar, capturar, revisar, desconectar, cambiar algo, volver a conectar.

**La interferencia de 50Hz era el problema real.** Cuando empezaste a ver señal en el scope, probablemente lo primero que notaste fue la enorme oscilación sinusoidal de 50Hz superpuesta a cualquier señal genuina. La red eléctrica contamina todos los sensores analógicos de baja frecuencia. El diseño del filtro notch de 464 taps no es entusiasmo excesivo — es necesidad técnica real.

**El PSoC hace todo en un chip.** La lista de componentes en `main.c` (OpaBuffer, PGA, OpaDiff, TIA, ADC_DelSig, Filter_DFB) son todos bloques analógicos y digitales configurables del PSoC 5LP. En un diseño de hardware tradicional, necesitarías 4-6 chips externos para lograr lo mismo. La arquitectura PSoC compacta el BOM (Bill of Materials) drásticamente.

**Problema de flotante en Cortex-M3.** El filtro FIR de 464 taps con coeficientes `double` era computacionalmente costoso. El Cortex-M3 sin FPU tiene que emular aritmética de punto flotante de 64 bits en software. A 1020 SPS, esto iba a ser un cuello de botella que se resolvería en iteraciones futuras (conversión a Q1.15 o simplificación del filtro).

**El commit del 29 a las 13:10 hs** fue el cierre del trabajo del 28 — commitear todo lo probado la noche anterior antes de seguir avanzando.
