#### El PSoC firmware: arquitectura de testbench

La estructura del firmware `Comparation.cydsn` es la más sofisticada del proyecto hasta este punto. El archivo `main.c` documenta la secuencia de boot en sus comentarios:

```c
/* Boot sequence:
 *   1. Interrupts on
 *   2. Debug system (UART) — first so boot events can be logged

 *   4. Register USB transport (UART transport disabled by default)
 *   5. Init all hardware modules; send debug event per module
 *   6. Start ADC pipeline

 */
```

Esta secuencia revela una arquitectura de firmware cuidadosamente diseñada. El sistema de debug se inicializa **primero** — incluso antes de los periféricos — para que todos los eventos de boot puedan ser loggeados. Es una técnica de firmware profesional: el sistema de diagnóstico debe estar disponible desde el primer instante para poder depurar problemas de inicialización.

**Las cuatro configuraciones ADC del `AnalogToDigital.h`:**

```c
/* ADC_DelSig configurations:
 *   CFG1: ±0.512 V, 18-bit, 1464 SPS  — buffer gain MUST stay 1
 *   CFG2: ±1.024 V, 17-bit, 1488 SPS  — buffer gain default 8, adjustable
 *   CFG3: ±2.048 V, 17-bit, 1488 SPS  — buffer gain default 8, adjustable
 *   CFG4: ±6.144 V, 17-bit, 1488 SPS  — buffer gain default 8, adjustable
 */
```

El ADC_DelSig del PSoC 5LP es un Sigma-Delta de alta resolución — el mismo principio que los `FIR x4 Decimation` del Simulink del 18 de abril, pero ahora implementado en hardware. Las cuatro configuraciones cubren el rango completo del geófono SM-24:

- **CFG1 (±0.512 V, 18-bit):** Para señales sísmicas muy fuertes (explosiones cercanas). El buffer gain se mantiene en 1 para no saturar.
- **CFG2 (±1.024 V, 17-bit):** Rango normal de trabajo con buffer gain 8 (4× ampliación del SM-24).
- **CFG3 (±2.048 V, 17-bit):** Para señales con ruido alto o fuente energética.
- **CFG4 (±6.144 V, 17-bit):** Rango máximo para calibración o señales muy grandes.

La tasa de muestreo 1464-1488 SPS (~1.5 kHz) es consistente con la cadena de decimación 64× del Simulink: si la señal sísmica máxima es 300 Hz, Nyquist requiere ≥600 Hz, y 1500 Hz da un margen 2.5× para las transiciones del filtro.

**El protocolo de comunicación PSoC ↔ Python:**

El `protocol.py` define el protocolo binario de comunicación:

```python
# Packet types (PSoC -> PC):
#   0xAA — 12-byte data sample
#   0xCC — 10-byte state/config confirmation  
#   0xDD — N-byte debug event

# Command frame (PC -> PSoC): [0xBB][CMD][LEN][payload...]
PKT_DATA_HEADER  = 0xAA
PKT_STATE_HEADER = 0xCC
CMD_MARKER       = 0xBB

# Command codes
CMD_START       = 0x01
CMD_STOP        = 0x02
CMD_SET_MUX_IN  = 0x03
CMD_SET_MUX_OUT = 0x04
CMD_SET_ADC     = 0x05  # payload: [cfg][buf_gain]
CMD_VDAC_LOAD   = 0x06  # con longitud de 2 bytes
CMD_SET_DEBUG   = 0x07
CMD_SET_DBG_CH  = 0x08
CMD_GET_STATE   = 0x09
```

El paquete de datos `0xAA` (12 bytes) es el corazón del protocolo: contiene un valor ADC de 24 bits (3 bytes, Q1.15 extendido a 24 bits), flags de configuración, número de secuencia, y CRC XOR. La sincronización se hace por header byte `0xAA` — el parser en Python busca este marcador y lee 11 bytes adicionales.

El diseño de `Communication.c` implementa un **registro de transportes**:

```c
static Transport_t *s_transports[COMM_MAX_TRANSPORTS];

void Communication_RegisterTransport(Transport_t *t)
{
    if (s_transport_count < COMM_MAX_TRANSPORTS)
        s_transports[s_transport_count++] = t;
}
```

Esto permite que el sistema soporte múltiples canales de comunicación (USB CDC, UART, y en el futuro SPI hacia el ESP32) con el mismo código de alto nivel. En el boot, solo USB CDC se registra; UART está deshabilitado pero el código está listo.

#### Commit 8 — `60e3d535` — *"Auto-mejora: Fase 8 — FIR de 256 coeficientes cargable desde Python"* — 04:40 hs

La Fase 8 es el hito técnico más importante de la sesión nocturna.

**¿Qué es el FIR Q1.15?**

Q1.15 es un formato de punto fijo de 16 bits donde:
- 1 bit de signo
- 15 bits fraccionarios

El rango es [-1, +0.999969...] con resolución de 1/32768. Los coeficientes de un filtro FIR normalizado (donde todos los coeficientes están entre -1 y +1) caben perfectamente en Q1.15.

Para convertir un coeficiente float a Q1.15:
```python
q15 = int(round(coeff * 32768))
q15 = max(-32768, min(32767, q15))  # saturación
```

**La implementación en PSoC:**

El CHANGELOG describe la implementación exacta del MAC (Multiply-Accumulate) en el PSoC:

```c
/* Algoritmo exacto del PSoC (delay circular int32, MAC int64, >>15, saturación) */
for (i = 0; i < fir_taps; i++) {
    acc += (int64_t)fir_coeffs[i] * delay_line[(delay_idx + i) % fir_taps];
}
out = (int32_t)(acc >> 15);
if (out >  0x7FFFFF) out =  0x7FFFFF;  // saturación 24-bit
if (out < -0x800000) out = -0x800000;
```

El acumulador `int64` evita el overflow durante la suma de 256 productos. La saturación a 24 bits mantiene la compatibilidad con el formato de salida del ADC_DelSig.

**Comandos de carga de filtro:**

```python
CMD_LOAD_FIR  = 0x0A  # payload: [len_hi][len_lo][coeff_q15_0_hi][coeff_q15_0_lo]...
CMD_FIR_CLEAR = 0x0B  # payload: vacío
```

La carga usa el mismo protocolo de longitud de 2 bytes que `CMD_VDAC_LOAD` — esto sugiere que los dos comandos fueron diseñados juntos como parte del mismo protocolo de carga de datos largos.

**El helper Python:**

```python
# digital_filters.py
def design_fir_lowpass(fc_hz, fs_hz=1500, n_taps=63):
    """Diseña un FIR pasa-bajos (ventana Hamming) y devuelve coefs float."""
    return scipy.signal.firwin(n_taps, fc_hz / (fs_hz / 2))

def coeffs_to_q15(coeffs):
    """Convierte float array a lista Q1.15 int16."""
    return [max(-32768, min(32767, int(round(c * 32768)))) for c in coeffs]
```

Con estos dos helpers, el workflow completo desde Python es:

```python
coeffs = design_fir_lowpass(20.0, n_taps=63)  # pasa-bajos a 20 Hz
q15 = coeffs_to_q15(coeffs)
protocol.cmd_load_fir(serial_port, q15)        # enviar al PSoC
```

El PSoC aplica el FIR a cada muestra ADC antes de enviarla por USB. Esta es la arquitectura de procesamiento en tiempo real: el PC define el filtro, el PSoC lo ejecuta en el hardware.

**El flujo de datos completo:**

```
SM-24 geófono
    ↓ (señal analógica)
PSoC 5LP (Comparation.cydsn)
    ↓ (ADC_DelSig + DFB + FIR Q1.15)
[SPI Slave transport] ← transport_spi.h/c

    ↓ (psoc_spi.cpp — SPI Master)
NTP sync → timestamp en microsegundos
    ↓ (wifi_transport.h)

    ↓ (WiFi 802.11)
Base station (base/receiver.py)
    ↓

```

El ESP32 conecta la cadena de adquisición embedded con el servidor de procesamiento. Los componentes clave:

**`psoc_spi.h/cpp` — SPI Master en el ESP32:**

```cpp
class PsocSPI {
    // SPI Mode 0 @ 4 MHz
    // Frame start: 0xAB
    // 30 muestras/batch = 306 bytes/transacción
    // DATA_READY GPIO: pulldown activo cuando hay datos
};
```

Los 30 samples/batch corresponden a ~20 ms de datos a 1500 SPS — un tamaño de paquete que equilibra la latencia con la overhead de SPI.

**`time_sync.h` — sincronización NTP:**

```cpp
// NTPClient wrapper con nowUs() (epoch + micros sub-segundo)
uint64_t nowUs();  // microsegundos desde epoch UNIX
```

La sincronización temporal es crítica para MASW: para calcular la curva de dispersión de Rayleigh, se necesita la diferencia de tiempo de llegada de las ondas entre los distintos geófonos. Sin una referencia temporal común, no se puede calcular la velocidad de fase.

**`receiver.py` — base station multi-nodo:**

```python
# UDP server, puerto 4321
# Acumulador por nodo: {node_id: [samples...]}
# Save a .npz cuando se recibe señal STOP o timeout
```

La base station puede recibir de múltiples nodos ESP32 simultáneamente — esto es el soporte para un arreglo MASW multi-canal. En teoría, cada geófono del arreglo tiene su propio ESP32, todos sincronizados por NTP, transmitiendo a la base station en tiempo real.

#### Commit 13 — `40eb1565` — *"Auto-mejora: test builders (19/19 PASS) + fix bug Q1.15 (32767→32768)"* — 13:10 hs

**Un bug importante corregido:**

El bug `32767→32768` en Q1.15 es sutil. En Q1.15, el valor máximo representable es 32767 (= 0x7FFF). Pero la conversión de float a Q1.15 usaba `int(round(1.0 * 32768))` = 32768, que no cabe en un int16 (overflow a -32768).

```python
# Antes (buggy):
q15 = int(round(c * 32768))  # para c=1.0: da 32768, no cabe en int16

# Después (fixed):  
q15 = max(-32768, min(32767, int(round(c * 32768))))  # saturación explícita
```

Este bug haría que un filtro FIR con coeficiente exactamente 1.0 (como un delta function para bypass) se representara incorrectamente en el PSoC.

#### Commit 14 — `5b02b8b5` — *"Fix ChannelFilter block state bug + expand test coverage to 119 tests"* — 13:25 hs

El bug del `ChannelFilter.process()` es el más impactante de los corregidos. El CHANGELOG lo describe:

> **Bug**: `zi * x[0]` se aplicaba en *cada* llamada a `process()`, no solo en la primera. Al procesar bloque a bloque, el estado `zi` era multiplicado por el primer sample del nuevo bloque, destruyendo la continuidad del filtrado. **Diferencia bloque vs. pasada única: 1.91 (RMS)**.

Un error RMS de 1.91 entre el filtrado bloque-a-bloque y la pasada única es un error enorme — significa que el filtrado en tiempo real (que procesa bloques de muestras) daba resultados completamente diferentes del filtrado offline. En MASW, donde la coherencia de la señal es crítica, esto habría invalidado todas las mediciones.

**Fix**: un flag `_fresh` asegura que el truco de inicialización de estado se aplica solo en la primera llamada.

Con el fix, la diferencia RMS baja a < 1e-10 — perfectamente consistente.

### La suite de tests como red de seguridad

El hecho de que el loop nocturno terminara con 125/125 tests passing no es accidental — es la condición de terminación del loop. Sin una suite de tests que confirme que el código funciona, el loop podría introducir bugs en cada mejora.

Los tests cubren varios niveles:
- **Unit tests**: protocol parsers, command builders, filtros FIR
- **Integration tests**: simulador + protocolo + canal
- **Stress tests**: rate, rollover de secuencia, reload de filtros
- **Adversarial tests**: garbage bytes, paquetes truncados, plen oversized

Esta cobertura multi-nivel garantiza que las "mejoras" del loop no rompan nada existente.

Ya hizo esto con el INVESTIGADOR (loop de 118 commits en 34 horas) y ahora lo repite con el loop de auto-mejora del PSoC/ESP32. La diferencia es la escala: el INVESTIGADOR procesó 52 papers académicos; el auto-mejora construyó una pila de firmware completa en una noche.

La decisión de correr el loop en deiStation es elegante: la máquina remota tiene recursos dedicados, conexión estable, y puede trabajar toda la noche sin interrumpir el entorno de desarrollo principal. El Tailscale VPN configurado el 15 de abril ahora tiene un uso claro.
