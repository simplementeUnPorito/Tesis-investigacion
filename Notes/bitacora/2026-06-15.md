## Contexto del día

El 14 de junio se había completado la auto-calibración V1 usando búsqueda binaria y telemetría `0xC4`. Pero el sistema tenía un problema estructural: el algoritmo de promediado era acumulativo (cada ventana de `avg_n` muestras se sumaba a un total creciente), lo que hacía que con el tiempo el promedio fuera dominado por mediciones viejas y el sistema "sobre-ajustara al ruido", saturando `GEO_LP`. Además, la arquitectura de parámetros era monolítica: `calibration_tables.h` concentraba todos los valores de las cuatro etapas GEO en un solo bloque de `#define`, sin posibilidad de afinar etapas individualmente.

El 15 de junio se encaró una reescritura profunda:

1. **Separar los parámetros por etapa** en headers independientes (`calibration_tables_geo_pga.h`, `_bp.h`, `_adder.h`, `_lp.h`)
2. **Reemplazar el promediado acumulativo** por una ventana deslizante de peso constante
3. **Agregar detección de saturación** por etapa, con manejo especial cuando todos los candidatos saturan
4. **Agregar la fase `realcheck`**: una fase final donde el PSoC mide con la entrada real del geófono (no con la referencia) y puede aplicar nudges finos de ±1 LSB VDAC
5. **Agregar un botón físico de calibración** conectado a `Clock_1` e `isr_Button`
6. **Escalar correctamente las cuentas ADC** aplicando el ratio de spans ADC/VDAC

A esto se sumó el problema clásico del hardware embebido: durante la tarde, la auto-calibración no disparaba porque el PSoC nunca fue flasheado con el firmware nuevo, y el ESP ya tenía `PSOC_AUTO_CAL_ON_READY=0`. Commit "ayuda Dios" — exacto.

**1. Eliminación de `AMux_IN` de la ruta de calibración**

Uno de los cambios más importantes y menos obvios: se elimina `AMux_IN` de toda la ruta de calibración durante bisección y verify.     cal_diag(PSOC_EVT_CAL_AMUX_IN, channel);
```c
}

void psoc_calibration_amux_active(void) {
    cal_amux_in_select(CAL_INPUT_NORMAL_CHANNEL);
}
void psoc_calibration_amux_idle(void) {
    cal_amux_in_select(CAL_INPUT_REF_CHANNEL);
}

// En calibration.h también se eliminan:
void psoc_calibration_amux_active(void);
void psoc_calibration_amux_idle(void);
```

Y en `main.c`, todas las llamadas a `psoc_set_amux_active()` y `psoc_set_amux_idle()` desaparecen. Las funciones `psoc_arm()` y `psoc_start_now()` ya no llaman a `psoc_set_amux_active()` antes de comenzar a muestrear. Esto refleja una decisión de arquitectura: la fase `realcheck` maneja el switch de AMux explícitamente cuando toca medir con la entrada real, pero durante bisección/verify el ADC mide directamente contra lo que tenga la entrada analógica (señal de geófono o lo que esté conectado). **2. Reemplazo del promediado acumulativo por ventana deslizante**

El cambio más técnico del commit. El viejo `async_measure_service()` usaba:

```c
// ANTES:
g_cal_async.cum_sum += g_cal_async.acc;
g_cal_async.cum_count += (int32)stage->avg_n;
// ...
cum_avg = g_cal_async.cum_sum / g_cal_async.cum_count;
g_cal_async.measured = cum_avg;
// criterio de estabilidad: |cum_avg - prev_avg| <= CAL_SETTLE_TOL_COUNTS
// o timeout CAL_SETTLE_MAX_WINDOWS
```

El problema con este enfoque: el denominador crece indefinidamente. Una medición de la primera ventana tiene el mismo peso que una de la última. Si hubo un transitorio de 100 ms al arrancar, ese ruido nunca desaparece del promedio. Esto causaba que el algoritmo "viera" puntos mejores de lo que eran realmente y terminara sobre-ajustando.

El nuevo código usa un buffer circular de sumas:

```c
// AHORA: buffer circular de sumas-de-ventana
if (g_cal_async.window_filled_count < window_count) {
    g_cal_async.window_buf[g_cal_async.window_pos] = g_cal_async.acc;
    g_cal_async.window_sum += g_cal_async.acc;
    g_cal_async.window_filled_count++;
} else {
    g_cal_async.window_sum -= g_cal_async.window_buf[g_cal_async.window_pos];
    g_cal_async.window_buf[g_cal_async.window_pos] = g_cal_async.acc;
    g_cal_async.window_sum += g_cal_async.acc;
}
g_cal_async.window_pos++;
if (g_cal_async.window_pos >= window_count) {
    g_cal_async.window_pos = 0u;
}

sliding_avg = g_cal_async.window_sum /
    ((int32)avg_n * (int32)g_cal_async.window_filled_count);
g_cal_async.measured = sliding_avg;
```

El promedio ahora tiene peso constante: siempre son las últimas `avg_n * window_count` muestras. Cuando se agrega una ventana nueva, se descarta la más antigua. El criterio de estabilidad también cambia: requiere `stable_streak` comparaciones consecutivas con diferencia ≤ `settle_tol_counts`, y el buffer debe estar lleno (no se evalúa estabilidad hasta que `window_filled_count >= window_count`). Si no estabiliza, el techo `max_samples` fuerza la salida.

La nueva estructura `PsocCalAvgCfg` encapsula todos estos parámetros:

```c
typedef struct {
    uint8  avg_n;             // Muestras por ventana parcial
    uint8  window_count;      // Cantidad de ventanas en el promedio
    uint16 max_samples;       // Techo duro de muestras
    int32  settle_tol_counts; // Diferencia máxima entre promedios para estable
    uint8  stable_streak;     // Ventanas estables consecutivas requeridas
} PsocCalAvgCfg;
```

Y `PsocCalStage` ahora tiene dos instancias de `PsocCalAvgCfg`: una para bisección (`avg`) y otra para verify (`verify_avg`), cada una con parámetros independientes. Los defaults GEO quedan en los headers por VDAC: bisección con `64×16=1024` muestras mínimas/`2048` techo, verify con `64×16=1024` mínimas/`4096` techo.

**3. Detección de saturación por etapa**

Nueva invariante: el sistema distingue candidatos "saturados" de candidatos "no saturados". Un candidato satura si `|medido| >= sat_counts` donde `sat_counts` es un umbral por etapa (inicialmente `120000` counts, que corresponde a ~+-0.9V para el GEO_LP a 18 bits signed).

```c
static uint8 cal_stage_saturated(const PsocCalStage *stage, int32 measured)
{
    if (stage->sat_counts <= 0L) {
        return 0u;
    }
    return (abs_counts(cal_scale_counts(measured)) >= 
            cal_scale_counts(stage->sat_counts)) ? 1u : 0u;
}
```

La lógica de selección del mejor candidato se vuelve bi-nivel: preferir siempre un candidato no saturado sobre uno saturado. Solo si todos los candidatos saturan, se elige el de menor error entre los saturados (como fallback de diagnóstico). Si ocurre esto, se emite `PSOC_EVT_CAL_STAGE_SAT_ALL` y la etapa se marca como fallida:

```c
static void cal_async_update_best(uint8 dac, int32 measured, int32 abs_error,
                                  uint8 saturated)
{
    if (saturated) {
        g_cal_async.saturated_seen = 1u;
        if (!g_cal_async.non_saturated_seen &&
            (g_cal_async.best_saturated ||
             abs_error < g_cal_async.best_abs_error)) {
            // Solo actualiza si todavía no hay candidato no-saturado
            g_cal_async.best_abs_error = abs_error;
            // ...
        }
        return;
    }
    // Si llega un candidato no saturado: descarta toda la historia saturada
    g_cal_async.non_saturated_seen = 1u;
    if (g_cal_async.best_saturated || abs_error < g_cal_async.best_abs_error) {
        // ...
    }
}
```

Además, el cálculo de `abs_error` cambia: ya no compara directamente en cuentas ADC de 18 bits, sino que escala ambos valores a unidades comparables con el VDAC8 (8 bits):

```c
// ANTES:
int32 abs_error = abs_counts(stage->target_counts - g_cal_async.measured);

// AHORA:
int32 abs_error = abs_counts(cal_scale_counts(stage->target_counts) -
                              cal_scale_counts(g_cal_async.measured));
```

La función `cal_scale_counts()` en este commit hace simplemente `value >> 10`, reduciendo 18 bits a ~8 bits de escala. (En el siguiente commit se refina para incorporar el ratio de spans ADC/VDAC.)

#### Diff y análisis técnico

**1. Corrección del escalado ADC → VDAC**

El commit anterior introducía `cal_scale_counts()` como un simple shift derecho de 10 bits. Pero hay un error de escala: el ADC del PSoC (ADC_DelSig en modo diferencial) tiene un span efectivo de 5V (±2.5V alrededor de VDDA/2), mientras el VDAC8 en modo 1× tiene un span de 0 a 4.080V. Un shift de 10 bits sobre 18 bits da escala "ADC units / 1024", que no coincide directamente con "VDAC units". La corrección:

```c
#define CAL_DAC_SCALE_SHIFT 10
#define CAL_ADC_SPAN_MV    5000L
#define CAL_VDAC_SPAN_MV   4080L

static int32 cal_div_round_signed(int32 value, int32 divisor)
{
    if (value >= 0L) {
        return (value + (divisor / 2L)) / divisor;
    }
    return -(((-value) + (divisor / 2L)) / divisor);
}

static int32 cal_scale_counts(int32 value)
{
    int32 shifted = value >> CAL_DAC_SCALE_SHIFT;
    return cal_div_round_signed(shifted * CAL_ADC_SPAN_MV, CAL_VDAC_SPAN_MV);
}
```

El factor `5000/4080 ≈ 1.225` hace que 1 count escalado equivalga aproximadamente a 1 código VDAC. La función `cal_div_round_signed()` maneja correctamente el redondeo para valores negativos (evita el sesgo del truncamiento hacia −∞ del operador `/` de C para enteros negativos).

**2. Cambio del criterio de éxito final: `psoc_calibration_async_result_ok()`**

El criterio de éxito reportado al ESP cambia de "la última etapa GEO_LP está dentro de `tolerance_counts`" a "la última etapa GEO_LP está dentro del rango operativo":

```c
// ANTES:

// AHORA:
return cal_measured_out_of_range(
    g_psoc_cal_results[PSOC_CAL_STAGE_COUNT - 1u].final_measured

```

`CAL_OPERATING_RANGE_COUNTS = 26214L` (≈±0.5V a 18 bits con 52429 counts/V). Si la etapa final queda dentro de ±0.5V, se considera éxito aunque no haya cumplido la tolerancia fina de verify/realcheck. Esto evita que la UI web marque una calibración como fallida solo porque el adjust fino no llegó al último LSB, cuando la señal ya está prácticamente centrada.

El explanation doc define claramente la dicotomía TOL/DEADBAND que confundía al sistema anterior:

> `TOL` es el criterio formal de éxito: si `abs(medido - target) <= TOL`, ese punto puede considerarse OK.
>
> `DEADBAND` es un criterio de control: si el error actual ya entró en esa zona, deja de mover el DAC para no perseguir ruido o alternar entre códigos vecinos.
>
> Si `DEADBAND == TOL`, el corte de movimiento coincide con el criterio formal. Si `DEADBAND > TOL`, la bisección puede terminar antes, pero luego `verify` podría marcar esa etapa como no OK.

Se agregan modelos Simulink para las etapas GEO: `simulacionStepVref/Adder.slx`, `simulacionStepVref/BP.slx`, `simulacionStepVref/LP.slx`, `simulacionStepVref/pga.slx`. Estos son los modelos de respuesta escalón de `Vref` para estimar las constantes de tiempo de asentamiento de cada etapa, que luego se usan para parametrizar `CAL_SETTLE_SAMPLES_FROM_MS()`.
