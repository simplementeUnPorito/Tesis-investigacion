### Cambio 1 — Eliminación del filtro pasivo analógico

`FiltroPasivoEntrada.m` y sus `.mat` asociados son eliminados. No como error — como decisión consciente. El usuario lo exploró, lo ejecutó (los `.mat` de resultados lo confirman), y lo descartó.

Las razones probables del descarte del filtro RC pasivo:

1. **Tolerancia de componentes**: un filtro RC con fc = 6.50 Hz requiere valores muy precisos de R y C. Con tolerancias estándar de componentes (5-10%), la frecuencia de corte real puede variar ±0.5 Hz. Para un geófono de 4.5 Hz, esto es crítico: si fc cae a 6.0 Hz, se amplifica excesivamente la resonancia del geófono.

2. **Dependencia de temperatura**: la capacitancia de condensadores electrolíticos varía con la temperatura. En trabajo de campo con temperaturas entre 0°C y 40°C, fc puede variar otro ±5%.

3. **El PSoC ya tiene DSP**: el PSoC 5LP tiene un ARM Cortex-M3 a 80 MHz, precisamente suficiente para implementar un filtro FIR en tiempo real. ¿Para qué agregar un RC externo cuando el procesador puede hacer el filtrado digitalmente sin componentes físicos?

4. **La simulación reveló el problema**: los `.mat` de simulación (`AnalogOut.mat`, `NoisyIn.mat`) sugieren que el usuario corrió el Simulink del circuito analógico con señal ruidosa y vio el resultado. Posiblemente el resultado no satisfizo las especificaciones.

### Cambio 2 — El nuevo paradigma: FIR con decimación en cadena

El nuevo Simulink se llama `FiltroDigital.slx` (101 KB). El nombre revela la decisión: "Digital" vs. el descartado "PasivoEntrada".

La variable registry del modelo revela la arquitectura:

```xml
<blockPath>FiltroDigital/FIR x4(a) Decimation</blockPath>
<blockPath>FiltroDigital/FIR x4(b) Decimation</blockPath>
<blockPath>FiltroDigital/FIR x4(c) Decimation</blockPath>
<expression>fir1(31, 0.15)</expression>
```

**Tres filtros FIR idénticos, cada uno seguido de decimación 4×.**

El filtro `fir1(31, 0.15)`:
- `31` — orden del filtro (32 coeficientes)
- `0.15` — frecuencia de corte normalizada = 0.15 × frecuencia Nyquist
- `fir1` — ventana de Hamming por defecto (la ventana óptima para minimizar el lóbulo lateral)

La cadena completa es un **filtro de decimación en tres etapas**:

```
Señal entrada (fs_alta)
    ↓
[FIR x4(a)] → decimación 4× → fs_alta / 4
    ↓
[FIR x4(b)] → decimación 4× → fs_alta / 16
    ↓
[FIR x4(c)] → decimación 4× → fs_alta / 64
    ↓
Señal salida (fs_alta / 64)
```

La decimación total es 4 × 4 × 4 = **64×**.

**¿Por qué decimación en cadena en lugar de un solo filtro con factor 64?**

La razón es eficiencia computacional. Un filtro FIR de orden 32 que decima 64× tiene que procesar 32 multiplicaciones por muestra a la frecuencia de entrada alta. Pero una cadena de tres filtros 4× procesa:
- Etapa a: 32 mults a fs_alta
- Etapa b: 32 mults a fs_alta/4 (4× menos trabajo)
- Etapa c: 32 mults a fs_alta/16 (16× menos trabajo)

Total ≈ 32(1 + 1/4 + 1/16) × fs_alta = 32 × 1.31 × fs_alta ≈ 42 mults/muestra equivalente

Vs. un filtro equivalente sin etapas que necesitaría ~2000 coeficientes para la misma performance → mucho más costoso.

**La relación con el PSoC y el Delta-Sigma ADC:**

El PSoC 5LP tiene un Delta-Sigma ADC de 20 bits que opera internamente con oversampling 256× y decimación. La arquitectura de `FiltroDigital` refleja exactamente este principio: muestrear a alta frecuencia (oversampling) y luego filtrar + decimar para obtener mayor resolución efectiva.

Esta es la conexión teórica clave: el Teorema de Nyquist dice que 1 bit de resolución extra se gana por cada 4× de oversampling con filtro ideal. Con 64× oversampling total, se ganan 3 bits de resolución efectiva (log₂(64) / 2 = 3). Si el PSoC tiene un SAR ADC de 12 bits base, con 64× oversampling se pueden obtener efectivamente 15 bits de resolución.

### Por qué el filtro digital es superior para este caso

La cadena FIR 32-tap × 3 etapas × 4× decimación tiene varias ventajas sobre el RC pasivo:

**1. Linealidad de fase**

Un filtro FIR de orden par (32 coeficientes) tiene **fase lineal exacta**: todos los componentes de frecuencia se retardan exactamente el mismo tiempo. En contraste, el RC pasivo del 17 de abril tiene fase no lineal — introduce distorsión de tiempo diferente según la frecuencia.

Para MASW, la **fase es crítica**: el método se basa en medir la velocidad de fase de las ondas Rayleigh a diferentes frecuencias. Un error de fase de un filtro no-lineal contamina directamente la curva de dispersión. El FIR lineal es el único tipo de filtro que no introduce este error.

**2. Estabilidad absoluta**

El filtro FIR es inherentemente estable — no tiene polos, solo ceros. El RC pasivo acoplado al geófono (como muestra la función de transferencia de tercer orden del 17 de abril) tiene polos complejos y puede aproximarse a la inestabilidad con ciertos valores de R y C.

**3. Coeficientes exactos**

`fir1(31, 0.15)` produce 32 coeficientes exactos en punto flotante doble. No hay incertidumbre de componentes, no hay variación con temperatura, no hay envejecimiento.

**4. Adaptabilidad**

Con el filtro en software, cambiar la frecuencia de corte es una línea de código (`fir1(31, fc_nueva)`). Con el RC pasivo, hay que cambiar el componente físico.

### La implicación para el PSoC

Con el pivote al filtro digital, el rol del PSoC cambia sutilmente:
- **Antes (April 17)**: PSoC como ADC puro — toma la señal ya filtrada por el RC pasivo y la digitaliza
- **Después (April 18)**: PSoC como ADC + DSP — toma la señal raw del INA y aplica el filtro FIR en tiempo real antes de transmitir

El ARM Cortex-M3 del PSoC 5LP a 80 MHz puede procesar el filtro FIR de decimación en tiempo real para una señal de geófono a las frecuencias de trabajo de MASW. El cálculo de viabilidad:

- Señal sísmica: hasta 300 Hz de componentes relevantes (Rayleigh waves fundamentales)
- Nyquist: muestrear a ≥600 Hz mínimo, más práctico 1000-2000 Hz
- Con oversampling 64×: muestrear a 64 kHz, decimar a 1 kHz en salida
- A 64 kHz, el ARM a 80 MHz tiene 80,000,000 / 64,000 = 1,250 ciclos por muestra
- Un FIR de 32 taps en punto fijo necesita ~32 MAC = ~64 ciclos en ARM con DSP instructions
- Con la cadena de 3 etapas (64x + 16x + 4x rate): el trabajo total es proporcional
- Conclusión: **perfectamente viable** en tiempo real para el PSoC 5LP
