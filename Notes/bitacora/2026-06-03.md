#### PSOC_UART_BAUD: 460800 → 115200

El primer cambio que salta en `psoc_uart.h` es el baud rate:

```cpp
#ifndef PSOC_UART_BAUD
-#define PSOC_UART_BAUD 460800UL
+#define PSOC_UART_BAUD 115200UL   /* 460800/921600 no son divisibles exactos a 24 MHz */
```

El comentario lo explica todo. El PSoC 5LP opera con un oscilador maestro de 24 MHz (IMO, Internal Main Oscillator). Para generar el baud rate del UART, el hardware hace una división entera de ese clock. Veamos si 460800 es divisible exacto de 24 MHz:

```
24 000 000 / 460 800 = 52.08... → no es entero
```

El UART del PSoC aproxima al divisor entero más cercano (52), que da:

```
24 000 000 / 52 = 461 538 bps (error: +0.16%)
```

Para UART asíncrono ese error es aceptable en teoría (el estándar permite hasta ±2.5%), pero en la práctica — con cables largos, capacitancia del cableado, y 95 bytes en tren continuo — ese error acumulado puede desincronizar los bytes al final del frame.

115200 sí es divisible exacto de 24 MHz:
```
24 000 000 / 208 = 115 384 bps (error: +0.16%)
```

Espera — tampoco es exacto. Pero el comentario dice "no son divisibles exactos" para 460800 y 921600, implicando que 115200 es *más* preciso o al menos que el error a 115200 se manifiesta menos porque el tren de bits es más lento y tiene más margen de timing. A 115200, 95 bytes tardan ≈8.3ms (vs 2.06ms a 460800). Con un batch cada 29.4ms, hay margen más que suficiente.

El cambio tiene un costo: la capacidad de diagnóstico es menor (si se quiere ver el stream UART con un analizador lógico, 115200 es más fácil de muestrear). Y la latencia aumenta. Pero la confiabilidad es lo primero.

#### MAC del maestro configurable vía platformio.ini

Antes, la MAC del ESP maestro era una constante hardcodeada en `slave/src/main.cpp`:

```cpp
/* MAC del ESP maestro — cambiar según el hardware */
static const uint8_t MASTER_MAC[6] = {0xFF, 0xFF, 0xFF, 0xFF, 0xFF, 0xFF};
```

Ahora:

```cpp
#ifndef MASTER_MAC0
  #define MASTER_MAC0 0xFF
#endif
/* ... MASTER_MAC1 a MASTER_MAC5 ... */
static const uint8_t MASTER_MAC[6] = {
    MASTER_MAC0, MASTER_MAC1, MASTER_MAC2,
    MASTER_MAC3, MASTER_MAC4, MASTER_MAC5
};
```

Con esta estructura, el `platformio.ini` de cada nodo esclavo puede tener:

```ini
build_flags =
    -DNODE_ID=1
    -DMASTER_MAC0=0xAA
    -DMASTER_MAC1=0xBB
    ...
```

Esto resuelve un problema práctico: antes, si cambias el ESP32 del maestro (funde, se pierde, se reemplaza), tenías que editar el código de cada esclavo y recompilar. Ahora solo cambias el `platformio.ini`. El código es idéntico para todos los esclavos — la configuración está en la capa de build, no en la lógica.

Los pines de SYNC también se volvieron configurables via `#ifndef`:

```cpp
#ifndef SYNC_IN_PIN
  #if defined(ESP8266)
    #define SYNC_IN_PIN 5
  #else
    #define SYNC_IN_PIN 26
  #endif
#endif
```

Misma filosofía: el hardware target (ESP8266 vs ESP32, pines específicos del PCB) se configura en platformio.ini, no hardcodeado en el código.

#### PING/PONG y CFG_ACK — nuevo sub-protocolo PSoC

El protocolo UART PSoC↔ESP se extendió con tres nuevos frame types (en `psoc_uart.h`):

```cpp
/* Control PSoC -> ESP */
#define PSOC_CTRL_PING     0xC0
#define PSOC_CTRL_PONG     0xC1
#define PSOC_CTRL_CFG_ACK  0xC2
```

Los frames tienen el mismo marcador `0xAB`, pero el segundo byte distingue el tipo en lugar de usar `SPI_BATCH_SAMPLES=30`. En el parser del ESP:

```cpp
if (_buf[1] == PSOC_CTRL_PING) {
    _pingsRx++;
    _lastOK = true;
    const uint8_t pong[4] = {
        PSOC_FRAME_MARKER, PSOC_CTRL_PONG, 0x00u,
        (uint8_t)(PSOC_CTRL_PONG ^ 0x00u)
    };
    _ser->write(pong, sizeof(pong));
```

El PSoC envía `[0xAB][0xC0]` (PING). El ESP responde inmediatamente con `[0xAB][0xC1][0x00][XOR]` (PONG). Esto permite que el PSoC sepa que el ESP está recibiendo — un handshake de bajo nivel para diagnóstico.

El `PSOC_CTRL_CFG_ACK = 0xC2` es más importante para el sistema:

```cpp
void PsocUART::_parseConfigAck()
{
    const uint8_t cmd = _buf[2];
    const uint8_t val = _buf[3];
    const uint8_t crc = _buf[4];
    if (crc != (uint8_t)(PSOC_CTRL_CFG_ACK ^ cmd ^ val)) {
        _lastOK = false;
        _badLen++;
        return;
    }
    _lastOK = true;
    _configAcksRx++;
    _cfgAckCmd = cmd;
    _cfgAckVal = val;
    _cfgAckPending = true;
```

Cuando el ESP envía un comando de configuración al PSoC (ej: `setPga(4)` → código 0xA6), el PSoC ahora responde con `[0xAB][0xC2][0xA6][valor_aplicado][CRC]` confirmando el valor que aplicó. El ESP puede verificar que el PSoC aplicó exactamente lo que se pidió.

Esto es el loop de confirmación que faltaba desde el principio: antes, el ESP enviaba comandos "al vacío" y asumía que el PSoC los procesó. Ahora hay un handshake real.

#### `probe()` — detección no invasiva del PSoC

```cpp
bool probe(uint32_t timeoutMs = 150);
```

El método `probe()` intenta detectar si el PSoC está conectado sin iniciar una captura. La docstring del código dice: "Primero usa el probe no invasivo 0xA5 y solo como fallback activa la rampa para firmware PSoC viejo."

El comando `0xA5 = PSOC_CMD_STATUS` es el probe primario: el PSoC responde con su estado actual. Si el PSoC tiene firmware viejo que no soporta `0xA5`, el probe hace un fallback a activar la rampa de debug (`0xB3`) y esperar que lleguen frames. En ambos casos, si algo llega por UART, el PSoC está vivo.

En `setup()` del esclavo:

```cpp
SLAVE_LOG_PRINTF("[SLAVE %d] Buscando PSoC...\n", NODE_ID);
g_psocConnected = psoc.probe(300);
```

Y el HELLO que el esclavo manda al maestro ahora incluye `psoc_ok`:

```cpp
matlab.sendHelloNotif(msg.node_id, msg.psoc_ok, senderMac);
```

#### Canal WiFi explícito en el esclavo

```cpp
/* Fijar canal 1 explícitamente para coincidir con el AP del maestro.
 * Sin esto, STA desconectado puede quedar en canal 0 (indefinido) y los
 * paquetes ESP-NOW se pierden porque maestro y esclavo están en canales distintos. */
esp_wifi_set_channel(1, WIFI_SECOND_CHAN_NONE);
```

Este es uno de esos bugs que son imposibles de depurar sin conocer bien el WiFi del ESP32: un ESP32 en modo STA (station) sin asociación a un AP puede quedar en "canal 0" o en el último canal escaneado. Si el maestro está en canal 1 (el AP siempre fija el canal al crear el access point), y el esclavo está en canal 6 (por ejemplo, el último AP que vio durante boot), los paquetes ESP-NOW se pierden silenciosamente porque ESP-NOW es capa de link WiFi — necesitan estar en el mismo canal.

La solución es forzar al esclavo al canal 1 explícitamente en `setup()`, después de `WiFi.mode(WIFI_STA)` y antes de `esp_now_init()`.

#### Instrumentación del diagnóstico UART PSoC

```cpp
static void logPsocUartDiag(const char *tag)
{
    const unsigned long bytes = (unsigned long)psoc.bytesRx();
    const unsigned long mark  = (unsigned long)psoc.markersRx();
    const unsigned long drop  = (unsigned long)psoc.syncDrops();
    ...
    SLAVE_LOG_PRINTF(
        "[SLAVE %d] %s bOK=%lu bBad=%lu uartBytes=%lu mark=%lu drop=%lu badLen=%lu ping=%lu rx=%d last=0x%02X age=%lu txOK=%lu txFail=%lu state=%d view=%u\n",
        ...);
}
```

Esta función de diagnóstico agrega en una sola línea todo lo necesario para depurar el link UART PSoC↔ESP: bytes recibidos, markers 0xAB vistos, drops (bytes descartados al buscar marker), frames de longitud incorrecta, pings recibidos, nivel lógico del pin RX, último byte visto y su edad en ms. Es la función que permite entender en Serial si el PSoC está enviando algo, qué está enviando, y si está corrompido.

El campo `last=0x%02X age=%lu` es particularmente útil: si el PSoC manda 0xAB pero nada más, el `age` revela cuántos milisegundos lleva el ESP esperando el resto del frame — lo que indicaría una transmisión truncada.

#### VER store-and-forward: sin RF durante muestreo

```cpp
/* "Ver": disparo único — guarda en RAM primero (sin TX ESP-NOW durante el
 * muestreo), luego el maestro hace dump vía REQ_BATCH. Así se elimina la
 * interferencia de RF que ensuciaba la señal en el modo live anterior. */
```

El VER anterior (del refactor del 31 de mayo) tenía un problema: durante el muestreo, el esclavo enviaba batches por ESP-NOW mientras el ADC estaba activo. La RF de ESP-NOW genera spikes de corriente que se acoplan al geófono o al ADC y crean artefactos en la señal.

El nuevo flujo:

```
MATLAB "Ver" →
  Maestro: entra en RUNNING, g_viewDump=true, g_viewNode=X
  Maestro: envía CMD_VIEW al nodo X
  Esclavo: recibe CMD_VIEW, llama enterHotWait(n) → store buffer allocado
  Esclavo: responde CFG_ACK(CMD_VIEW, ok=1) → "listo para HOT_WAIT"
  Maestro: recibe CFG_ACK(ok=1) → inicia HOTWAIT_QUERY como en START normal
  PSoC: flanco SYNC_IN → inicia captura → guarda en g_store_buf (sin RF)
  PSoC: captura completa → esclavo envía CFG_ACK(CMD_VIEW, ok=2) → "captura lista"
  Maestro: recibe CFG_ACK(ok=2) → beginDump() → REQ_BATCH
  Esclavo: responde con batches desde g_store_buf
```

El esclavo ya no hace TX ESP-NOW durante el muestreo. El maestro hace el dump DESPUÉS de que el PSoC terminó. Esto requiere que el esclavo tenga RAM suficiente para guardar todos los batches (n × 30 muestras × sizeof(SampleBytes)).

#### `beacon_pause()` / `beacon_resume()` — el descubrimiento más importante del día

```cpp
static void beacon_pause(void)
{
    wifi_config_t ap_cfg = {};
    esp_wifi_get_config(WIFI_IF_AP, &ap_cfg);
    ap_cfg.ap.beacon_interval = 60000u;   /* max documentado TU ≈ 61 s */
    esp_wifi_set_config(WIFI_IF_AP, &ap_cfg);
    MASTER_LOG_PRINTLN("[MASTER] beacon pausado (captura activa)");
}

static void beacon_resume(void)
{
    wifi_config_t ap_cfg = {};
    esp_wifi_get_config(WIFI_IF_AP, &ap_cfg);
    ap_cfg.ap.beacon_interval = 1000u;    /* 1000 TU ≈ 1.024 s */
    esp_wifi_set_config(WIFI_IF_AP, &ap_cfg);
    MASTER_LOG_PRINTLN("[MASTER] beacon resumido (inicio dump)");
}
```

Y en `setup()`:

```cpp
/* Aumentar intervalo de beacon para reducir interferencia RF en el ADC.
 * El default (100 TU ≈ 100 ms) genera spikes de 10 Hz visibles en el geófono.
 * ESP-NOW no depende de beacons, así que 1000 ms no afecta la comunicación. */
{
    wifi_config_t ap_cfg = {};
    esp_wifi_get_config(WIFI_IF_AP, &ap_cfg);
    ap_cfg.ap.beacon_interval = 1000;   /* 1000 TU ≈ 1024 ms */
    esp_wifi_set_config(WIFI_IF_AP, &ap_cfg);
}
```

Este es el hallazgo clave del día. Un TU (Time Unit) en WiFi es 1024 microsegundos. El beacon interval por defecto es 100 TU = 102.4ms. Eso significa que el AP del maestro transmite un beacon de management frame **cada 102.4ms** — una frecuencia de aproximadamente **9.77 Hz**.

El ADC Delta-Sigma del PSoC muestrea a ~1020 SPS. La frecuencia de Nyquist es 510 Hz. 10 Hz está bien dentro de la banda pasante del sistema MASW (0.5–50 Hz para ondas Rayleigh). Cada vez que el beacon se transmite, genera un spike de corriente en la fuente de alimentación del ESP32, que se irradia como RF, que se acopla capacitivamente o inductivamente al cableado del geófono o al PCB del PSoC, y aparece como un spike de amplitud en la señal del ADC.

El resultado: la señal del geófono tenía spikes a 10 Hz que arruinaban cualquier análisis espectral en la banda de interés (especialmente porque las ondas Rayleigh en suelos típicos tienen frecuencias entre 5–40 Hz — exactamente donde estaría el artefacto de 10 Hz).

La solución:
1. En reposo: `beacon_interval = 1000 TU = 1.024 s` (frecuencia de beacon: ~0.98 Hz — fuera de la banda MASW)
2. Durante captura: `beacon_interval = 60000 TU ≈ 61 s` (prácticamente silencio RF)
3. Después del dump: `beacon_resume()` vuelve a 1000 TU

El beacon a 1000 TU no afecta ESP-NOW porque ESP-NOW es un protocolo de envío espontáneo — los nodos no dependen de los beacons para sincronizarse. Solo un cliente WiFi estándar asociado al AP dependería de los beacons, y en este sistema no hay ninguno.

La consecuencia de este arreglo para la tesis es profunda: si el sistema se hubiera presentado sin este fix, cualquier espectrograma tendría un pico ficticio a 10 Hz. Una curva de dispersión tendría ruido en la banda donde más importa la precisión. Nadie hubiera sabido por qué — hasta que alguien revisara el beacon interval.

Y cuando termina el PRESTART:

#### `clampRecordBatches()` / `PSOC_CAPTURE_MAX_BATCHES = 512`

```cpp
#ifndef PSOC_CAPTURE_MAX_BATCHES
  #define PSOC_CAPTURE_MAX_BATCHES 512
#endif

static uint16_t clampRecordBatches(uint16_t n)
{
    if (n > PSOC_CAPTURE_MAX_BATCHES) {
        MASTER_LOG_PRINTF("[MASTER] n_batches %u > PSoC max %u, clamp\n",
                          n, (unsigned)PSOC_CAPTURE_MAX_BATCHES);
        return (uint16_t)PSOC_CAPTURE_MAX_BATCHES;
    }
    return n;
}
```

512 batches × 30 muestras / 1020 SPS = 15 segundos de grabación máxima por VER. El límite es el RAM del ESP32 disponible para `g_store_buf`:

```
512 batches × 30 SampleBytes × sizeof(SampleBytes)
```

`SampleBytes` tiene 10 bytes (raw_lo, raw_hi, alog0/1/2, digi0/1/2, gain, flags). Entonces:
```
512 × 30 × 10 = 153 600 bytes ≈ 150 KB
```

El ESP32 tiene 520 KB de RAM total, pero el sistema usa parte para el stack, heap de Arduino, buffers ESP-NOW, etc. 150 KB para el store buffer es razonable. El N de 16 bits (hasta 65535 batches ≈ 32 min) sigue disponible para el modo START normal (que usa streaming, no store).

#### Rename `DiferencialToSingleEnded__ESP.cydsn` → `AcondicionamientoAnalogico.cydsn`

El commit incluye el rename (74% de similitud según git) del proyecto PSoC del geófono:

```
R074 src/psoc/DiferencialToSingleEnded__ESP.cydsn/... → src/psoc/AcondicionamientoAnalogico.cydsn/...
```

El nombre anterior era funcional pero descriptivo de la topología del circuito: "diferencial a single-ended para el ESP". El nuevo nombre es descriptivo del propósito del sistema: "acondicionamiento analógico". Es el proyecto PSoC que contiene toda la cadena del geófono: PGAp/PGAn → LPF → OPAbp → OPAadder → OPAlp → ADC_DelSig.

Junto con el rename, aparecen archivos de `ADC_1` en los outputs del build:

```
src/psoc/AcondicionamientoAnalogico.cydsn/CortexM3/ARM_GCC_541/Debug/ADC_1.lst
src/psoc/AcondicionamientoAnalogico.cydsn/CortexM3/ARM_GCC_541/Debug/ADC_1_AMux.lst
```

`ADC_1` y `ADC_1_AMux` son un segundo ADC con un multiplexor analógico. El PSoC 5LP tiene un solo ADC_DelSig y varios SAR ADCs. El ADC_1 aquí es posiblemente un SAR para un canal adicional — el martillo de impacto, que necesita muestreo de alta velocidad (SAR >> DelSig en velocidad).

El hecho de que el acondicionamiento del geófono y del martillo estén en el **mismo PSoC** (`AcondicionamientoAnalogico`) es un cambio de arquitectura significativo: en lugar de dos PSoCs separados (geófono en `DiferencialToSingleEnded` y martillo en `AcondicionamientoMartillo`), ahora el objetivo es un solo PSoC que maneja ambos sensores.

El 3 de junio es el día donde el sistema deja de ser "prometedor" y se convierte en "funciona de verdad". Los seis horas entre commits no son de trabajo trivial: son de debug intenso con el hardware real.

El hallazgo del beacon a 10Hz es el tipo de bug que te puede costar semanas si no sabes buscarlo. La pista probable fue ver picos periódicos en la señal del geófono que no correspondían con nada mecánico ni eléctrico obvio. Calculando 1/periodo_pico ≈ 10 Hz y sabiendo que el AP WiFi emite beacons a ~10 Hz por defecto... el diagnóstico se vuelve obvio. Pero llegar a ese punto requiere conocer en detalle cómo funciona el WiFi del ESP32.

El rename del PSoC (`DiferencialToSingleEnded` → `AcondicionamientoAnalogico`) junto con la aparición de `ADC_1` señala que la arquitectura del hardware está madurando: en vez de dos PSoCs independientes (geófono y martillo), se apunta a un diseño integrado. Eso simplifica el PCB, reduce la cantidad de chips, y unifica el firmware.

"Locura parece que ya funciona super bien" — el "locura" reconoce la cantidad de cambios que fueron necesarios para que algo aparentemente simple (capturar una señal y transmitirla) funcione sin artefactos. Cada uno de esos cambios (baud rate, canal WiFi, beacon interval, probe PSoC, VER flow) era un fragmento de un rompecabezas que solo se ve completo cuando el sistema finalmente funciona como se espera.
