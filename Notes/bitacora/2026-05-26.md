El sistema puede comunicarse, pero la sincronización START→muestreo todavía tiene problemas: cuando MATLAB envía el comando ARM→START, a veces el esclavo no estaba en estado listo (la transición IDLE→ARMED→SAMPLING tiene una ventana de tiempo que puede fallar si los mensajes llegan fuera de orden).

## Commits

### `8aeee8dc` — "Add PRESTART and scope multi-start support" — 20:24 hs

#### PRESTART — resolver la ventana de tiempo ARM→START

El flujo anterior:
```
MATLAB → ARM → [espera N ms] → MATLAB → START
```

El problema: el "espera N ms" era fijo y a veces insuficiente. Si el ESP-NOW tardaba más que lo esperado en entregar CMD_ARM, el esclavo recibía CMD_START antes de estar en estado ARMED.

El flujo PRESTART:
```
MATLAB → ARM
    Maestro:
    1. Broadcast CMD_ARM a esclavos
    2. Espera todos los ARM_ACK (timeout configurable)
    3. Entra en estado PRESTART (verifica HOT_WAIT)
    4. HOT_WAIT: query cada esclavo para verificar estado listo
    5. Timeout o todos confirmados → emite CMD_START real

MATLAB → START (triggering el PRESTART)
```

Los parámetros de HOT_WAIT son configurables en `platformio.ini`:

```cpp
#ifndef HOTWAIT_QUERY_DELAY_MS
  #define HOTWAIT_QUERY_DELAY_MS 20     // esperar 20ms entre queries
#endif
#ifndef HOTWAIT_QUERY_TIMEOUT_MS
  #define HOTWAIT_QUERY_TIMEOUT_MS 120  // timeout total 120ms
#endif
#ifndef HOTWAIT_QUERY_RETRIES
  #define HOTWAIT_QUERY_RETRIES 3       // 3 intentos antes de abandonar
#endif
```

Este diseño es no-bloqueante: el maestro no hace `delay()` (que bloquearía el loop de Arduino y causaría pérdida de paquetes ESP-NOW), sino que usa un mecanismo de timer para transicionar entre estados:

```
MasterState: IDLE → ARMING → ARMED → PRESTART → RUNNING → STOPPING
                                         ↑
                                    nuevo estado
```

#### `CMD_SCOPE_START` — pulso de referencia para el osciloscopio

```cpp
#define CMD_SCOPE_START 0x60  /* master → esclavos: pulso START falso, sin muestreo ni ACK */
```

Una herramienta de diagnóstico: el maestro envía CMD_SCOPE_START que hace que todos los nodos generen un pulso GPIO de 1ms (definido en `scope_measurement.h` como `SCOPE_START_PULSE_US = 1000`) sin iniciar muestreo real. Con el osciloscopio conectado a los pines de sync de los tres nodos, se puede medir la diferencia de tiempo entre los pulsos — que es exactamente la latencia de sincronización del sistema.

```cpp
/* scope_measurement.h — centraliza el ancho del pulso de START */
#ifndef SCOPE_START_PULSE_US
  #define SCOPE_START_PULSE_US 1000
#endif
```

Los pines de sync para el osciloscopio son:
- **Master**: GPIO 15
- **Slave 1**: GPIO 32
- **Slave 2**: GPIO 22

Con un osciloscopio de 4 canales (CH1=maestro, CH2=esclavo1, CH3=esclavo2, CH4=tierra), se puede ver la latencia de inicio entre nodos. El objetivo: todos los pulsos dentro de < 1ms entre sí.

#### `SCOPE_MULTI` — repetición automática para el osciloscopio

El estado `SCOPE_MULTI` envía CMD_SCOPE_START `SCOPE_MULTI_START_COUNT` veces (default 128) con un gap de `SCOPE_MULTI_START_GAP_MS` (default 1000ms = 1 segundo) entre cada uno. Esto permite usar el trigger del osciloscopio para capturar múltiples eventos y medir estadísticas de jitter de sincronización automáticamente — sin tener que presionar el botón 128 veces.

#### Non-blocking pulse handling — el detalle crítico

Antes de esta versión, los pulsos GPIO de debug se implementaban con `delayMicroseconds()` — que bloquea el CPU durante el pulso. Para pulsos de 1ms, eso significa que el maestro pierde ~1ms de recepción ESP-NOW, lo que puede hacer que pierda el ARM_ACK de un esclavo.

La nueva implementación usa timestamps para el pulso:

```cpp
/* No-blocking: registra t_pulse_start, pone el pin HIGH.
 * En el loop, verifica si pasó SCOPE_START_PULSE_US y baja el pin. */
static uint32_t g_pulse_start_us = 0;
static bool     g_pulse_active   = false;

if (g_pulse_active && (micros() - g_pulse_start_us) >= SCOPE_START_PULSE_US) {
    digitalWrite(DEBUG_HW_START_PIN, DEBUG_HW_START_IDLE);
    g_pulse_active = false;
}
```

Es un cambio pequeño en código pero grande en comportamiento: el CPU nunca se bloquea, los callbacks de ESP-NOW nunca se pierden, y los pulsos son igualmente precisos.

#### Qué fue lo que finalmente funcionó

El "parece que funciona" (no "funciona" sino "parece que funciona" — la prudencia de ingeniería después de horas de debugging) indica que en este momento:

1. **El PSoC** estaba tomando muestras del ADC y enviándolas via SPI al ESP32 esclavo
2. **El ESP32 esclavo** estaba empaquetando las muestras en MsgData y enviándolas via ESP-NOW al maestro
3. **El ESP32 maestro** estaba reensamblando los batches y enviándolos via TCP al PC
4. **MATLAB** estaba recibiendo los datos y graficándolos en la GUI `InterfaceESP`

Probablemente era una señal ruidosa, no necesariamente una señal de geófono calibrada. Pero se veían datos en la pantalla de MATLAB que venían del hardware real — y no eran todos ceros, no eran una rampa de debug, eran datos con variación que parecían ruido/señal real.

La exclamación final "putaaaa" (con cuatro 'a') es proporcional a las horas de trabajo: desde el 24 de mayo a las 4 PM hasta el 26 de mayo a las 10 PM son aproximadamente 54 horas en las que el sistema pasó de código en papel a un sistema inalámbrico multi-nodo que transmite datos. Incluyendo una noche entera (commit a las 4:48 AM del 25).

#### Los fixes del commit

El commit modifica `espnow_transport.h` del esclavo — probablemente el último bug antes de que funcionara. La API de ESP-NOW en ESP8266 vs ESP32 tiene diferencias en los callbacks:

```cpp
/* ESP8266 */
static void onSendCb(uint8_t *mac, uint8_t status);

/* ESP32 */
static void onSendCb(const uint8_t *mac, esp_now_send_status_t status);
```

Si el código del esclavo intentaba registrar el callback con la firma equivocada, el envío ESP-NOW fallaba silenciosamente (no se enviaba, pero tampoco había error visible). El `espnow_compat.h` del commit del 24 intentó resolver esto, pero posiblemente el bug persistió hasta este commit.

Los cambios en `sync_protocol.h` del último commit (en ambos master y slave) seguramente incluyen sincronización de la versión del protocolo — es mencionado en la descripción del commit del 25 que el protocolo debe mantenerse "sincronizado manualmente" entre maestro y esclavo. Un desajuste de versión hace que los structs tengan tamaños diferentes y la deserialización falla.

#### El estado de la máquina al momento del "funciona"

Del `main.cpp` del maestro en este commit:

```
                 SCOPE_MULTI
                     ↑
IDLE → ARMING → ARMED → PRESTART → RUNNING → STOPPING
         ↑                ↑
    ARM_ACK(N)        HOT_WAIT OK
```

El flujo que funcionó:
1. MATLAB abre socket TCP al maestro en 192.168.X.X:5005
2. MATLAB envía `[0xAB][0xA2][0x00][0xA2]` (ARM)
3. Maestro: IDLE→ARMING, broadcast CMD_ARM
4. Esclavos: WAIT_ARM→ARMED, envían CMD_ARM_ACK
5. Maestro: ARMING→ARMED (recibió N ACKs)
6. MATLAB envía `[0xAB][0xA3][0x00][0xA3]` (START)
7. Maestro: ARMED→PRESTART, verifica HOT_WAIT
8. Maestro: PRESTART→RUNNING, emite GPIO HIGH + CMD_START
9. Esclavos: ISR GPIO → ARMED→SAMPLING, PSoC inicia muestreo
10. Esclavos → PSoC: SPI read → ESP-NOW → Maestro → TCP → MATLAB
11. MATLAB: muestra datos en tiempo real
