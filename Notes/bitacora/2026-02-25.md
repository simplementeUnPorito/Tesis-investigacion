## Contexto del día

Todo proyecto de tesis tiene un momento en que deja de ser una idea en la cabeza y se convierte en algo concreto, registrado, público. Para este trabajo sobre sistemas de adquisición sísmica de bajo costo basados en ondas superficiales, ese momento fue el 25 de febrero de 2026 a las 4 y media de la tarde.

El proyecto que empieza acá apunta a desarrollar un **sistema completo de medición sísmica** para caracterización geotécnica del suelo usando el método MASW (*Multichannel Analysis of Surface Waves*), pero sin el equipamiento industrial que normalmente cuesta decenas de miles de dólares. La idea base es usar geófonos del tipo SM-24 (sensores económicos de velocidad de partícula del suelo), electrónica embebida accesible (PSoC de Cypress, ESP32), y procesamiento en MATLAB, para replicar la funcionalidad de equipos comerciales a una fracción del costo.

En este momento, todo eso todavía es solo una idea y un marco teórico por construir. El primer paso es técnico y burocrático: crear el repositorio.

Una sola línea. El título del repositorio. El `README.md` es el archivo de bienvenida estándar de cualquier repositorio de GitHub — aparece en la página principal del repo. En este caso tiene exactamente una línea: `# Tesis`. Sin descripción, sin instrucciones, sin nada más. Es el mínimo absoluto necesario para que GitHub inicialice el repositorio con algo de contenido (un repositorio completamente vacío sin ningún commit no permite clonar ni crear branches).

El diff muestra:
- `new file mode 100644` — archivo nuevo con permisos de lectura/escritura estándar (no ejecutable)
- `index 00000000..c2843346` — el objeto Git pasa de no existir (`00000000`) al hash `c2843346` del contenido
- El contenido es simplemente `# Tesis` seguido de un newline implícito

**¿Por qué no commitear más cosas desde el inicio?** La estrategia más común al arrancar un proyecto de tesis con Git es justamente esta: crear el repo con un commit mínimo, y luego ir agregando contenido a medida que se define la estructura. Un commit de setup masivo desde el día 1 rara vez funciona bien porque la estructura del proyecto todavía no está clara. Esto cambia radicalmente en los próximos 7 días.

#### Qué se pensó en este momento

La creación del repositorio implica que la tesis ya tiene un nombre oficial: simplemente **"Tesis"**. El nombre del repo es `simplementeUnPorito/Tesis` — el usuario `simplementeUnPorito` corresponde a Elías David Álvarez. El nombre de usuario es una broma interna que hace referencia a ser "simplemente un porito" (alguien sin pretensiones).

En este punto de la tesis, el marco teórico estaba en construcción. El libro de referencia principal — *"Soil Characterization by Surface Waves"* de Sebastiano Foti (CRC Press, 2014) — ya había sido identificado como la fuente principal. El método MASW estaba siendo estudiado a partir de los papers fundacionales de Park et al. (1998, 1999) y Xia et al. (1999).

La decisión de usar Git para registrar el progreso de la tesis es en sí misma una decisión de ingeniería: permite rastrear exactamente qué se hizo, cuándo, y en qué orden. Muchos tesistas guardan todo en Google Drive o carpetas locales; usar Git con GitHub da versionado, backup automático en la nube, y la posibilidad de colaborar con el asesor o recuperar versiones anteriores.

#### Impacto en el proyecto

Este commit no tiene impacto técnico por sí mismo. Su importancia es simbólica y organizacional: a partir de este punto, todo el trabajo quedará registrado con fecha y hora exacta. Cada cambio tendrá un autor, un mensaje de commit, y un diff que muestra exactamente qué cambió. Esto es lo que hace posible escribir esta bitácora más de 3 meses después y reconstruir con precisión qué se hizo cada día.
