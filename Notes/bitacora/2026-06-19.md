**Etapa:** Activación del promediado por hardware (canal DMA "Lote") en la calibración analógica
**Hito:** El canal DMA Lote que quedó cableado pero inerte el 18 de junio se conecta por primera vez: la calibración deja de promediar muestra por muestra en el ISR y pasa a consumir ventanas ya sumadas por hardware, con una verificación de coherencia en tiempo de compilación entre el tamaño del lote DMA y el tamaño de ventana de calibración

## Contexto del día

El 18 de junio ("prelocura") había dejado la infraestructura de un canal DMA "Lote" — un canal capaz de transferir `DMA_LOTE_SAMPLES` muestras consecutivas del ADC en una sola ráfaga hacia un buffer, en vez de disparar una interrupción por cada muestra individual como hace el canal "Single". Los comentarios de ese día eran explícitos: "hoy ningún consumidor lo usa todavía; el canal ya está armado e inicializado para cuando se conecte el promediado". El 19 de junio es exactamente ese momento de conexión.

La calibración analógica del sistema GEO funciona, desde hace semanas, con una ventana deslizante de promediado: para cada valor candidato de un DAC, el sistema toma varias muestras consecutivas del ADC, las promedia, y compara ese promedio contra un valor objetivo para decidir si subir, bajar o aceptar el DAC actual (la búsqueda binaria de calibración). Hasta el 18 de junio, ese promediado se hacía muestra por muestra: el ISR de la ruta "Single" entregaba una muestra a la vez, y el código de calibración las iba acumulando en un acumulador de software (`g_cal_async.acc`) hasta juntar las `avg_n` muestras necesarias para una ventana.

El cambio del 19 de junio es reemplazar ese acumulado software por el acumulado que ya hace el hardware: en vez de que la CPU sume N muestras una por una en el ISR, el DMA en modo Lote transfiere las N muestras de una sola vez a un buffer, y es el propio ISR (que ahora se ejecuta una sola vez cada N muestras, no N veces) el que hace la suma en un loop corto, y se la entrega ya lista al código de calibración como una "ventana" (`sum`, `n`) en vez de una muestra individual. La motivación es doble: reduce drásticamente la cantidad de interrupciones (de N por ventana a 1 por ventana) y garantiza que el tamaño de la ventana de calibración coincide exactamente con el tamaño del lote de hardware — algo que antes dependía de contar bien en software y ahora se fuerza a nivel de tipo.

#### Diff y análisis técnico

**La limpieza previa del `TopDesign`.** Antes de entrar al código, vale notar algo que el `git show --stat` revela: el commit elimina una cantidad considerable de componentes DMA e ISR con nombres que parecen restos de iteración — `isr_DMA_Filtler_RAM_RAM` (con un typo, "Filtler" en vez de "Filter"), `isr_DMA_Filer_RAM` (typo, "Filer" sin la "t"), `DMA_RAM_RAM`, `DMA_DelSig_RAM_RAM`, `DMA_Filter_RAM_RAM`, `isr_DelSig_1`. Esto sugiere que, entre el 18 y el 19 de junio, Elías experimentó varias veces con distintos nombres y topologías para los componentes DMA dentro del editor esquemático de PSoC Creator antes de asentarse en la nomenclatura final (`DMA_DelSig_RAM_Single/Lote`, `DMA_Filter_RAM_Single/Lote`), y el commit "Precambios" es, en parte, el resultado de limpiar ese rastro de experimentación una vez que la topología definitiva quedó clara. También aparece un componente `Reg_Selec` (con un typo, sin la "t" final de "Select") que se elimina, reemplazado por el `Reg_Select` correcto usado en el código.

**La verificación de coherencia en tiempo de compilación.** El cambio más importante conceptualmente está al principio de `calibration.c`:

```c
/* El lote HW (PSOC_ADC_LOTE_SAMPLES, psoc_adc.h) debe equivaler exactamente
 * a una ventana de calibración GEO (CAL_AVG_N_GEO_*) — async_measure_service
 * trata cada lote como una ventana ya sumada. Si alguna vez divergen, mejor
 * fallar el build que misconvergir en silencio. */
#if (PSOC_ADC_LOTE_SAMPLES != CAL_AVG_N_GEO_PGA) || \
    (PSOC_ADC_LOTE_SAMPLES != CAL_AVG_N_GEO_BP) || \
    (PSOC_ADC_LOTE_SAMPLES != CAL_AVG_N_GEO_ADDER) || \
    (PSOC_ADC_LOTE_SAMPLES != CAL_AVG_N_GEO_LP)
#error "PSOC_ADC_LOTE_SAMPLES (psoc_adc.h) debe ser igual a CAL_AVG_N_GEO_* (calibration_tables_geo_*.h)"
#endif
```

Esta es una decisión de diseño que vale la pena detenerse a analizar. `PSOC_ADC_LOTE_SAMPLES` (definido en `psoc_adc.h`, valor 64) es cuántas muestras trae cada ráfaga del DMA en modo Lote. `CAL_AVG_N_GEO_PGA`, `CAL_AVG_N_GEO_BP`, etc. son, para cada una de las cuatro etapas de calibración GEO, cuántas muestras se promedian por ventana en el algoritmo de calibración. El código nuevo *asume* que cada lote de DMA es exactamente una ventana de calibración completa — no llama a `psoc_adc_take_isr_window()` varias veces para juntar una ventana, la trata como si un solo lote ya *fuera* la ventana entera. Esa suposición es correcta solo si las constantes coinciden numéricamente, y el commit prefiere reventar la compilación con un `#error` legible antes que dejar que el firmware arranque con una inconsistencia silenciosa que produciría una calibración sutilmente incorrecta (por ejemplo, promediando de a 64 muestras cuando el algoritmo de calibración esperaba ventanas de 48). El comentario lo resume con una frase que funciona casi como principio de ingeniería: "mejor fallar el build que misconvergir en silencio".

**El cambio de unidad en el descarte inicial.** Otro efecto colateral de que el hardware ahora entregue ventanas completas en vez de muestras individuales es que el conteo de "muestras a descartar" al principio de una medición (para dejar que la señal se asiente tras cambiar el DAC) tiene que expresarse en lotes, no en muestras sueltas:

```c
/* discard_samples llega en unidades de muestra cruda; el lote HW entrega
 * ventanas completas, así que se convierte a lotes-a-descartar con
 * redondeo hacia arriba (nunca menos descarte que antes). */
g_cal_async.discard_count = (uint16)((discard_samples + (PSOC_ADC_LOTE_SAMPLES - 1u)) /
                                      PSOC_ADC_LOTE_SAMPLES);
```

La fórmula `(discard_samples + (PSOC_ADC_LOTE_SAMPLES - 1u)) / PSOC_ADC_LOTE_SAMPLES` es la técnica estándar de "división entera redondeada hacia arriba" (ceiling division) sin usar aritmética de punto flotante: sumar el divisor menos uno antes de dividir garantiza que cualquier resto no nulo empuje el resultado al entero siguiente. El comentario explica la elección de redondear hacia arriba en vez de hacia abajo: "nunca menos descarte que antes" — es preferible descartar una ventana de más (perder algo de tiempo de convergencia) que descartar de menos y contaminar la medición con muestras tomadas antes de que la señal analógica se asentara tras el cambio de DAC. Es una decisión conservadora consistente con el principio general de la calibración: preferir lentitud a imprecisión.

**El nuevo par de funciones "window" en `psoc_adc`.** Para separar la ruta de muestra individual (que sigue existiendo, para la ruta Single de captura normal) de la ruta de ventana agregada (nueva, para calibración), se agrega un segundo par de funciones en `psoc_adc.c`, con la misma disciplina de sección crítica que ya tenía `psoc_adc_take_isr_sample`:

```c
static volatile int32 g_psoc_adc_isr_window_sum   = 0;
static volatile uint8 g_psoc_adc_isr_window_n      = 0u;
static volatile uint8 g_psoc_adc_isr_window_ready  = 0u;

void psoc_adc_note_isr_window(int32 sum, uint8 n)
{
    g_psoc_adc_isr_window_sum = sum;
    g_psoc_adc_isr_window_n = n;
    g_psoc_adc_isr_window_ready = 1u;
}

uint8 psoc_adc_take_isr_window(int32 *out_sum, uint8 *out_n)
{
    uint8 ok = 0u;
    uint8 saved;
    if (out_sum == (int32 *)0 || out_n == (uint8 *)0) {
        return 0u;
    }
    saved = CyEnterCriticalSection();
    if (g_psoc_adc_isr_window_ready) {
        *out_sum = g_psoc_adc_isr_window_sum;
        *out_n = g_psoc_adc_isr_window_n;
        g_psoc_adc_isr_window_ready = 0u;
        ok = 1u;
    }
    CyExitCriticalSection(saved);
    return ok;
}
```

Es exactamente el mismo patrón productor/consumidor de siempre (bandera `ready`, sección crítica al leer y limpiar), aplicado ahora a un par `(suma, cantidad)` en vez de a una muestra suelta.

**La ISR compartida y el orden correcto de las operaciones de reescalado.** El cambio más delicado desde el punto de vista numérico ocurre en `isr_DMA_DelSig_RAM_Handler`, que ahora tiene que manejar dos casos posibles según qué sub-ruta esté activa (Single o Lote), y lo hace consultando un espejo de software del estado del multiplexor de hardware:

```c
/* ISR compartida del canal raw (Single + Lote, OR'd por HW en un solo vector).
 * g_raw_route_lote (espejo de dma_route_select) dice cuál llegó. */
CY_ISR(isr_DMA_DelSig_RAM_Handler)
{
    if (g_raw_route_lote)
    {
        /* Lote: DMA_DelSig_RAM_Lote depositó DMA_LOTE_SAMPLES muestras en
         * g_dma_raw_lote_buf. psoc_adc_counts_right_aligned debe aplicarse a
         * CADA muestra antes de sumar (no una vez al final) — sumar primero
         * y dividir después trunca distinto, rompiendo la equivalencia con
         * el camino Single muestra-a-muestra que esto reemplaza. */
        int32 sum = 0;
        uint8 k;
        for (k = 0u; k < DMA_LOTE_SAMPLES; k++)
        {
            const volatile uint8 *s = &g_dma_raw_lote_buf[(uint16)k * 3u];
            sum += psoc_adc_counts_right_aligned(dma_buf_to_i24(s));
        }
        psoc_adc_note_isr_window(sum, (uint8)DMA_LOTE_SAMPLES);
        return;
    }

    g_adc_raw = psoc_adc_counts_right_aligned(dma_buf_to_i24(g_dma_raw_buf));
    ...
```

El comentario dentro del `for` documenta explícitamente una trampa numérica sutil: `psoc_adc_counts_right_aligned()` es probablemente una operación de reescalado o desplazamiento de bits que ajusta el valor crudo del ADC a un rango de trabajo (right-aligned = alineado a la derecha, típico de ADCs Delta-Sigma que entregan el dato en los bits altos de una palabra más ancha). Si esa función involucra truncamiento (por ejemplo, un shift a la derecha que descarta bits bajos), aplicarla a la *suma* de 64 muestras crudas daría un resultado numéricamente distinto que aplicarla a *cada* muestra individual y sumar después — el error de truncamiento se acumula distinto en cada caso. El comentario dice exactamente esto: "sumar primero y dividir después trunca distinto, rompiendo la equivalencia con el camino Single". Es el tipo de bug que sería casi imposible de detectar por inspección superficial del código (ambas versiones "parecen" hacer lo mismo, matemáticamente equivalente en aritmética real, pero no en aritmética de enteros con truncamiento) y que solo se evita entendiendo profundamente la semántica de `psoc_adc_counts_right_aligned()`.

**La invariante ignorada a propósito.** Dentro de `async_measure_service()`, en `calibration.c`, aparece un patrón de "ignorar deliberadamente" que vale la pena resaltar como estilo de programación defensiva madura:

```c
/* win_n solo puede valer PSOC_ADC_LOTE_SAMPLES (garantizado por el
 * #error de arriba == avg_n en todo build posible); se ignora a
 * propósito en vez de ramificar de forma defensiva sin salida sensata. */
(void)win_n;
g_cal_async.acc = win_sum;
g_cal_async.avg_count = avg_n;
```

En vez de escribir un `if (win_n != avg_n) { manejar_error(); }` — código defensivo que en la práctica nunca se ejecutaría (porque el `#error` de compilación ya garantiza la igualdad) pero que ensuciaría el flujo de lectura con una rama muerta — Elías opta por silenciar explícitamente el warning de "variable no usada" con `(void)win_n` y dejar un comentario que explica por qué esa rama defensiva sería ruido, no seguridad. Es una decisión de estilo consistente con la filosofía de "no validar lo que las garantías del sistema ya excluyen", en vez de sembrar el código con chequeos redundantes.

**El re-armado manual del canal DMA Lote.** Un detalle de bajo nivel que revela cuánto entendió Elías el comportamiento interno del controlador de DMA del PSoC 5LP:

```c
/* Re-arma DMA_DelSig_RAM_Lote a un estado limpio (offset cero) antes de
 * seleccionarlo. Si el canal se deselecciona a mitad de lote, su cuenta
 * interna (burstcount_remain) queda parcialmente avanzada — CyDmaChSetInitialTd
 * no resetea esa cuenta viva por sí solo (ver CyDmac.c), así que hace falta
 * un ciclo disable → re-set TD inicial → re-enable. Solo debe llamarse
 * mientras el canal está deseleccionado (sin DRQs entrantes). */
static void dma_raw_lote_rearm(void)
{
    CyDmaChDisable(DMA_DelSig_RAM_Lote_DmaHandle);
    CyDmaChSetInitialTd(DMA_DelSig_RAM_Lote_DmaHandle, g_td_raw_lote);
    CyDmaChEnable(DMA_DelSig_RAM_Lote_DmaHandle, 1u);
}
```

Esto documenta un comportamiento no obvio del controlador DMA de Cypress: si el canal Lote se desconecta del flujo de datos (porque el registro `Reg_Select` cambió a otra ruta) mientras estaba a mitad de acumular una ráfaga de 64 muestras, su contador interno de "cuántas muestras faltan para completar el burst" (`burstcount_remain`, mencionado explícitamente citando el archivo fuente `CyDmac.c` del framework de Cypress) queda en un valor intermedio. Simplemente volver a apuntar el canal a su Transaction Descriptor inicial con `CyDmaChSetInitialTd()` no resetea ese contador — hay que deshabilitar el canal por completo, re-setear el TD, y volver a habilitarlo, para garantizar que la próxima vez que se seleccione empiece a contar desde cero. Sin este re-armado, el primer "lote" leído tras cambiar de ruta podría contener una mezcla de muestras viejas y nuevas, corrompiendo la primera ventana de cada calibración. Este es exactamente el tipo de detalle que solo se descubre leyendo el código fuente del framework de Cypress o depurando un comportamiento errático — y el hecho de que el comentario cite el archivo (`CyDmac.c`) sugiere que Elías efectivamente fue a leer ese código para entender por qué el DMA se comportaba de forma inesperada.

Este re-armado se invoca justo antes de activar la ruta Lote para calibración:

```c
psoc_calibration_servo_abort();
dma_raw_lote_rearm();          /* arranque limpio del canal Lote antes de seleccionarlo */
dma_route_select(0u, 1u, 0u);  /* fuerza raw+Lote: calibración consume ventanas HW */
```

Nótese el cambio respecto al 18 de junio: donde antes `dma_route_select(0u, 0u, 0u)` forzaba raw+Single para calibrar, ahora `dma_route_select(0u, 1u, 0u)` fuerza raw+**Lote** — la calibración es, a partir de este commit, el primer y único consumidor real del canal de promediado por hardware que se había dejado preparado el día anterior.

**Diagnóstico de recorte de captura.** Un agregado menor pero práctico: cuando el número de lotes pedido por el operador (`g_n_batches`) excede la capacidad física del buffer de captura (`PSOC_CAPTURE_MAX_BATCHES`), en vez de truncar silenciosamente el pedido, el firmware ahora avisa explícitamente por UART:

```c
if (g_n_batches > PSOC_CAPTURE_MAX_BATCHES) {
    /* Pedido real recortado al cap físico — avisar en vez de truncar en
     * silencio (la web puede así informar la duración real soportada). */
    uart_send_diag(PSOC_EVT_CAPTURE_CLAMPED, diag_u16_sat(g_n_batches));
}
```

con un nuevo código de evento `PSOC_EVT_CAPTURE_CLAMPED = 0x3Eu` documentado en `psoc_hw.h`. Es una mejora de usabilidad pequeña pero significativa para un sistema que va a operar en campo: si un operador pide, sin darse cuenta, una duración de captura mayor a la que el hardware soporta, antes el sistema simplemente capturaba menos de lo pedido sin decir nada; ahora la interfaz web puede mostrar una advertencia explícita.

**Renombrado del flag de modo debug.** Se corrige un nombre engañoso: `PSOC_DEBUG_LED` (que en realidad controlaba una rampa de datos sintéticos para depuración, no un LED) pasa a llamarse `PSOC_RAMP_DEBUG_ENABLE`, y todo el código que dependía de ese flag se envuelve en `#if PSOC_RAMP_DEBUG_ENABLE` en vez de comprobarlo en runtime — de forma que, si se desactiva, el código de la rampa de depuración ni siquiera se compila, ahorrando espacio de flash en builds de producción.

#### Impacto en el proyecto

Este commit es la primera vez que el sistema de calibración analógica consume datos que no vienen de una interrupción por muestra, sino de una ráfaga de hardware ya agregada. El impacto práctico esperado es una reducción sustancial de la carga de interrupciones durante la calibración (de una interrupción por muestra a una interrupción cada 64 muestras) y, potencialmente, una calibración más rápida al no competir tanto por ciclos de CPU con el resto del sistema. Pero el valor más importante del commit, para la tesis, es metodológico: documenta con comentarios extensos y precisos tres trampas no triviales de trabajar con DMA y aritmética de enteros en un microcontrolador (el orden de truncamiento en el reescalado, el re-armado manual del contador de ráfaga del DMA, y la verificación de coherencia en tiempo de compilación entre dos subsistemas que comparten una suposición implícita). Son exactamente el tipo de detalle que separa un sistema embebido que "funciona la mayor parte del tiempo" de uno que funciona de forma predecible y verificable.
